### Q1. Difference Between Window Functions vs Aggregate Functions

---

📌 **Opening line:**
> *"Both compute values over a set of rows — but the key difference is what happens to the rows after the computation."*

---

🗣️ **Script to say:**

*"So aggregate functions like `SUM`, `COUNT`, `AVG` — they **collapse rows**. You GROUP BY something, and your result has fewer rows than the input. The individual row identity is gone.*

*Window functions, on the other hand, **keep all the rows**. They compute a value for each row based on a window — a defined set of related rows — but the row itself stays in the output. So you can have both the original row data AND the computed metric sitting side by side.*

*The syntax difference is the `OVER()` clause — that's what makes it a window function.*

*For example — if I want total sales per region, `SUM(sales) GROUP BY region` gives me one row per region. But if I want each sales rep's row PLUS their region total in the same output, I use `SUM(sales) OVER (PARTITION BY region)` — every rep row stays, and each one now has the region total attached.*

*Window functions also unlock things aggregate functions can't do at all — like `RANK()`, `ROW_NUMBER()`, `LAG()`, `LEAD()`, `FIRST_VALUE()`. These require row-level context that gets destroyed by GROUP BY.*

*They're also evaluated AFTER WHERE and GROUP BY in the SQL execution order — which is important to understand when you're writing complex queries."*

---

✅ **Closing hook:**
> *"At AbbVie, I used window functions extensively — `ROW_NUMBER()` for deduplication in the Silver layer, `LAG()` to calculate time between patient visits, and `RANK()` to identify top-performing territories per region. These would have been impossible or very slow with GROUP BY alone."*

---

**Quick Reference Table to draw if asked:**

| Feature | Aggregate Function | Window Function |
|---|---|---|
| Syntax | `SUM(col) GROUP BY x` | `SUM(col) OVER (PARTITION BY x)` |
| Rows in output | Fewer (collapsed) | Same (all rows kept) |
| Row identity | ❌ Lost | ✅ Preserved |
| Can use RANK/LAG/LEAD | ❌ No | ✅ Yes |
| Execution order | After WHERE | After WHERE + GROUP BY |
| Use case | Summary reports | Per-row analytics |

---

**Code to back it up:**
```sql
-- AGGREGATE: collapses to one row per region
SELECT
    region,
    SUM(visit_count)   AS total_visits,   -- one row per region
    COUNT(patient_id)  AS patient_count
FROM patient_events
GROUP BY region;
-- Output: 4 rows (US, EU, APAC, LATAM)

-- WINDOW: keeps all rows, adds computed columns
SELECT
    patient_id,
    territory_id,
    region,
    visit_count,
    SUM(visit_count)  OVER (PARTITION BY region)                    AS region_total,
    AVG(visit_count)  OVER (PARTITION BY region)                    AS region_avg,
    RANK()            OVER (PARTITION BY region ORDER BY visit_count DESC) AS rank_in_region,
    ROW_NUMBER()      OVER (PARTITION BY patient_id ORDER BY event_date DESC) AS recency_rank,
    LAG(visit_count)  OVER (PARTITION BY patient_id ORDER BY event_date)    AS prev_visit_count,
    visit_count - LAG(visit_count) OVER (
        PARTITION BY patient_id ORDER BY event_date
    )                                                               AS visit_delta
FROM patient_events;
-- Output: ALL rows, each with additional computed columns
```

### Q2. Explain Indexes and When NOT to Use Them

---

📌 **Opening line:**
> *"An index is a separate data structure that lets the database find rows without scanning the entire table — like a book's index pointing you to the right page instead of reading every page."*

---

🗣️ **Script to say:**

*"Conceptually, a database index works like a sorted lookup structure — typically a B-tree. When you query `WHERE patient_id = 'P001'`, without an index the DB does a full table scan — reads every row. With an index on `patient_id`, it traverses the B-tree, finds 'P001' in O(log n) time, and jumps directly to those rows.*

*There are a few common types — B-tree index is the default and works for equality and range queries. Bitmap index works well for low-cardinality columns like status or region. Composite index covers multiple columns together — but column order matters, the leftmost column must be in the WHERE clause.*

*Now — when NOT to use indexes, which is what interviewers really want to hear:*

*First — **small tables**. If a table has 500 rows, a full scan is faster than the index overhead. The optimizer often ignores the index anyway.*

*Second — **high write frequency**. Every INSERT, UPDATE, DELETE must also update every index on that table. If you have 5 indexes on a heavily written table, writes become 5x more expensive. For our Iceberg ingestion tables at AbbVie that had millions of appends daily, adding indexes on them would have killed write throughput.*

*Third — **low-cardinality columns**. An index on `status` with values 'ACTIVE' or 'INACTIVE' — half the table is ACTIVE. Scanning half a table through an index is actually SLOWER than a full table scan because of the random I/O pattern.*

*Fourth — **columns not used in WHERE, JOIN, or ORDER BY**. An index nobody queries through is just write overhead with zero read benefit.*

*Fifth — **in Spark/Iceberg/Delta context** — traditional row-level indexes don't exist. Instead you have Z-Ordering, partitioning, and min/max file statistics. Understanding this distinction matters in modern data engineering."*

---

✅ **Closing hook:**
> *"In my AbbVie pipelines, I worked with Oracle as the source OLTP system — heavily indexed for point lookups. But in our Iceberg analytical layer, we used partitioning and sort ordering instead of indexes. The analogy is similar but the mechanism is file-level skipping rather than row-level B-tree traversal."*

---

**When to use vs avoid:**
```
✅ USE index when:
   - High-cardinality column (patient_id, order_id)
   - Frequently in WHERE / JOIN / ORDER BY
   - Read-heavy table (OLAP dimension tables)
   - Column with selective queries (< 15% of rows returned)

❌ AVOID index when:
   - Small table (< 10,000 rows)
   - High write frequency (appends every minute)
   - Low-cardinality column (status: Active/Inactive)
   - Column never used in WHERE / JOIN
   - Spark / Iceberg / Delta (use partitioning + Z-order instead)
```

### Q3. What is the Execution Plan?

---

📌 **Opening line:**
> *"An execution plan is the database engine's step-by-step recipe for how it will physically retrieve and process your data — it shows WHAT operations will happen, in WHAT order, and at WHAT cost."*

---

🗣️ **Script to say:**

*"When you run a SQL query, the database doesn't just run it as written. The query optimizer takes your SQL, considers multiple ways to execute it, estimates the cost of each, and picks the cheapest plan. The execution plan is that chosen plan — made visible to you.*

*To see it — in most databases you prefix your query with `EXPLAIN` or `EXPLAIN ANALYZE`. In Spark you call `.explain()` or `.explain('formatted')` on a DataFrame.*

*What you're looking for in a plan:*

*First — **join strategy**. Is it a hash join, nested loop, or sort-merge join? A nested loop join on a large table is usually a red flag — O(n²) complexity. You want hash join for large-large joins, or broadcast hash join when one side is small.*

*Second — **table scans**. Is it doing a full table scan (`Seq Scan`) or using an index (`Index Scan`)? For large tables, full scans are expensive. But sometimes the optimizer CHOOSES a full scan even if an index exists — because statistically it's cheaper for that query.*

*Third — **estimated row counts vs actual**. If the optimizer estimated 100 rows but got 10 million — that's a statistics problem. Stale statistics lead to bad plan choices. Running `ANALYZE TABLE` refreshes those statistics.*

*Fourth — **sort operations**. Unexpected sorts mean the optimizer needs to ORDER data you didn't ask for — often because of a missing index or a join strategy requiring sorted input.*

*Fifth — **cost numbers**. Higher cost = more resource-intensive. The root node's cost is the total query cost. Drill into the most expensive child node — that's where to optimize.*

*In Spark specifically, `explain()` shows 4 plan levels — Parsed, Analyzed, Optimized, Physical. The Physical plan is what actually executes — look for BroadcastHashJoin, SortMergeJoin, and FileScan with partition pruning.*"

---

✅ **Closing hook:**
> *"At AbbVie, when Spark jobs were slow, my first step was always `.explain('formatted')`. I once found a SortMergeJoin where a BroadcastHashJoin should have been — the optimizer didn't know the lookup table was small after a filter. Wrapping it in `broadcast()` explicitly cut that stage from 8 minutes to 40 seconds."*

---

**Reading an execution plan:**
```sql
-- PostgreSQL / Hive
EXPLAIN ANALYZE
SELECT p.patient_id, t.region, COUNT(*) as visit_count
FROM patient_events e
JOIN patients p ON e.patient_id = p.patient_id
JOIN territories t ON p.territory_id = t.territory_id
WHERE e.event_date >= '2024-01-01'
GROUP BY p.patient_id, t.region;

/* Example output to read:
HashAggregate (cost=12500.00 rows=50000)         ← Step 5: aggregate
  → Hash Join (cost=5000.00 rows=100000)          ← Step 4: join patients
      → Hash Join (cost=2000.00 rows=500000)      ← Step 3: join territories
          → Seq Scan on patient_events            ← Step 1: full scan ← PROBLEM
              Filter: (event_date >= '2024-01-01')
          → Hash on territories (300 rows)        ← Step 2: small table cached
      → Hash on patients

Fix: Add index on event_date, or partition by event_date
*/
```

```python
# Spark execution plan
df.join(broadcast(territory_df), "territory_id") \
  .filter(col("event_date") >= "2024-01-01") \
  .groupBy("region").count() \

# Look for:
# ✅ BroadcastHashJoin       ← small table broadcasted (good)
# ✅ PartitionFilter         ← partition pruning working (good)
# ❌ SortMergeJoin           ← large shuffle happening (optimize)
# ❌ FileScan (no pruning)   ← reading all files (add filter on partition col)
```

### Q4. Explain Inner vs Left vs Anti Joins

---

📌 **Opening line:**
> *"Joins define which rows survive when combining two tables — and the choice between inner, left, and anti completely changes your result set, so picking the wrong one is a logic bug that's often hard to spot."*

---

🗣️ **Script to say:**

*"Let me walk through each with a concrete example. Say I have a `patients` table and a `appointments` table. Not every patient has an appointment.*

*`INNER JOIN` — **only rows that match in BOTH tables**. A patient with no appointment? Gone. An appointment with no matching patient? Gone. You get the intersection. Use this when you only care about records that exist on both sides — like 'show me patients who have had at least one appointment.'*

*`LEFT JOIN` (or LEFT OUTER JOIN) — **all rows from the left table, plus matching rows from the right**. A patient with no appointment? Still in the result, with NULL for the appointment columns. Use this when the left table is your primary entity and the right table is optional context — like 'show me all patients and their most recent appointment if they have one.'*

*`RIGHT JOIN` — mirror of LEFT. In practice, most engineers just rewrite it as a LEFT JOIN by swapping table order — easier to reason about.*

*`FULL OUTER JOIN` — everything from both tables, NULLs where there's no match on either side. Use for reconciliation — 'show me all patients AND all appointments, highlight which ones don't match.'*

*`ANTI JOIN` — this is the one people forget about. It returns rows from the left table that have **NO match** in the right table. In SQL you write it as `WHERE NOT EXISTS` or `WHERE ... IS NULL` after a LEFT JOIN, or using `EXCEPT`. Classic use case: 'find patients who have NEVER had an appointment' or 'find records in source that are MISSING from target' — which is exactly what I used for data reconciliation.*

*Performance note: INNER JOIN is typically fastest because it produces the smallest result set. LEFT JOIN can be expensive if the left table is huge. ANTI JOIN with `NOT IN` has a known performance trap — if the right side contains NULLs, the entire query returns nothing. Use `NOT EXISTS` instead — it's safer and often faster."*

---

✅ **Closing hook:**
> *"I used ANTI JOIN pattern constantly at AbbVie — for CDC reconciliation, finding records in Bronze that hadn't made it to Silver, and identifying API records that weren't in the Oracle source. The `NOT EXISTS` pattern was my go-to over `NOT IN` because NULL safety matters with patient data."*

---

**Visual and code:**
```
patients table:          appointments table:
P001 | Alice             P001 | 2024-01-10
P002 | Bob               P001 | 2024-03-15
P003 | Carol             P004 | 2024-02-20  ← no matching patient!

INNER JOIN result:       LEFT JOIN result:
P001 | Alice | 2024-01-10    P001 | Alice | 2024-01-10
P001 | Alice | 2024-03-15    P001 | Alice | 2024-03-15
                             P002 | Bob   | NULL         ← no appointment, kept
                             P003 | Carol | NULL         ← no appointment, kept

ANTI JOIN result (patients with NO appointments):
P002 | Bob
P003 | Carol
```

```sql
-- INNER JOIN: patients who have appointments
SELECT p.patient_id, p.name, a.appointment_date
FROM patients p
INNER JOIN appointments a ON p.patient_id = a.patient_id;

-- LEFT JOIN: all patients, appointment if exists
SELECT p.patient_id, p.name, a.appointment_date
FROM patients p
LEFT JOIN appointments a ON p.patient_id = a.patient_id;

-- ANTI JOIN pattern 1: NOT EXISTS (recommended — NULL safe)
SELECT p.patient_id, p.name
FROM patients p
WHERE NOT EXISTS (
    SELECT 1 FROM appointments a
    WHERE a.patient_id = p.patient_id
);

-- ANTI JOIN pattern 2: LEFT JOIN + IS NULL
SELECT p.patient_id, p.name
FROM patients p
LEFT JOIN appointments a ON p.patient_id = a.patient_id
WHERE a.patient_id IS NULL;

-- ANTI JOIN pattern 3: EXCEPT (set-based)
SELECT patient_id FROM patients
EXCEPT
SELECT patient_id FROM appointments;

-- ⚠️ ANTI JOIN trap — NOT IN with NULLs returns nothing!
SELECT patient_id FROM patients
WHERE patient_id NOT IN (
    SELECT patient_id FROM appointments  -- if ANY row is NULL here → query returns 0 rows!
);
-- FIX: add WHERE patient_id IS NOT NULL inside subquery, or use NOT EXISTS
```

### Q5. What Does ANALYZE TABLE Do?

---

📌 **Opening line:**
> *"`ANALYZE TABLE` collects statistical metadata about a table — row counts, column cardinality, value distributions, min/max values — and stores it so the query optimizer can make better decisions about join order, join strategy, and index usage."*

---

🗣️ **Script to say:**

*"The query optimizer is essentially a cost estimator. Before it can choose between, say, a hash join or a sort-merge join, it needs to know — how many rows does this table have? How many distinct values does this column have? What's the distribution — is it skewed toward certain values?*

*Without accurate statistics, the optimizer is guessing. And when it guesses wrong — say it thinks a table has 1,000 rows but it actually has 50 million — it picks the wrong join strategy, wrong index, wrong order. That leads to queries that are 10-100x slower than they should be.*

*`ANALYZE TABLE` fixes this by scanning the table and computing those statistics: row count, null count, min/max, number of distinct values, and optionally histogram buckets for value distribution. These are written to the system catalog — and the optimizer reads them the next time it plans a query.*

*When should you run it?*

*After loading large amounts of data — especially after a bulk insert or a full load. The stats are stale because the table changed significantly.*

*After partitioning changes — new partitions have no stats.*

*When you notice the execution plan looks wrong — you expected an index scan but got a full table scan. Check stats first before adding indexes.*

*In Spark/Hive specifically, `ANALYZE TABLE` also computes column statistics that enable the Catalyst optimizer to make better decisions — especially useful when you have skewed joins and want Spark to correctly size the broadcast threshold.*"

---

✅ **Closing hook:**
> *"At AbbVie, after full loads into Hive or when building the Gold layer, I always ran `ANALYZE TABLE ... COMPUTE STATISTICS FOR ALL COLUMNS`. This was critical for Spark to correctly estimate which joins could be broadcast joins vs sort-merge joins — directly impacting query performance on our 20 TB dataset."*

---

**Code:**
```sql
-- Hive / Spark SQL
-- Basic: row count only
ANALYZE TABLE silver.patient_events COMPUTE STATISTICS;

-- Full: row count + all column statistics (needed for Catalyst optimizer)
ANALYZE TABLE silver.patient_events COMPUTE STATISTICS FOR ALL COLUMNS;

-- Specific columns (faster than ALL COLUMNS on wide tables)
ANALYZE TABLE silver.patient_events
COMPUTE STATISTICS FOR COLUMNS patient_id, territory_id, event_date, region;

-- Per partition (important for partitioned tables)
ANALYZE TABLE silver.patient_events
PARTITION (load_date='2024-03-14', region='US')
COMPUTE STATISTICS FOR ALL COLUMNS;

-- Check if stats exist
DESCRIBE EXTENDED silver.patient_events;
-- Look for: numRows, numFiles, rawDataSize in Table Parameters

-- Oracle equivalent
EXEC DBMS_STATS.GATHER_TABLE_STATS('ABBVIE_PROD', 'PATIENT_MASTER');

-- PostgreSQL equivalent
ANALYZE patient_events;
VACUUM ANALYZE patient_events;  -- also cleans dead rows
```

```python

""")

# Check stats in Spark
```

### P1. Write SQL to Rank Customers/Patients Based on Purchases

**Scenario:** Rank patients by visit count within each territory, and also globally. Show top 3 per territory.

```sql
-- ── Full ranking query ────────────────────────────────────────────────
WITH patient_visit_summary AS (
    -- Step 1: aggregate visits per patient per territory
    SELECT
        p.patient_id,
        p.patient_name,
        t.territory_id,
        t.territory_name,
        t.region,
        COUNT(e.event_id)               AS total_visits,
        SUM(e.visit_duration_mins)      AS total_visit_minutes,
        MAX(e.event_date)               AS last_visit_date,
        MIN(e.event_date)               AS first_visit_date
    FROM patients p
    JOIN patient_events e  ON p.patient_id   = e.patient_id
    JOIN territories t     ON p.territory_id = t.territory_id
    WHERE e.event_date >= '2024-01-01'        -- YTD filter
      AND p.status = 'ACTIVE'
    GROUP BY p.patient_id, p.patient_name,
             t.territory_id, t.territory_name, t.region
),

patient_ranked AS (
    -- Step 2: apply multiple ranking functions
    SELECT
        *,
        -- Global rank across ALL patients
        RANK()       OVER (ORDER BY total_visits DESC)
            AS global_rank,

        -- Rank within each territory (resets per territory)
        RANK()       OVER (PARTITION BY territory_id ORDER BY total_visits DESC)
            AS rank_in_territory,

        -- Dense rank (no gaps: 1,2,2,3 not 1,2,2,4)
        DENSE_RANK() OVER (PARTITION BY region ORDER BY total_visits DESC)
            AS dense_rank_in_region,

        -- Percentile: which % tier is this patient in?
        NTILE(4)     OVER (PARTITION BY territory_id ORDER BY total_visits DESC)
            AS visit_quartile,           -- 1=top 25%, 4=bottom 25%

        -- Running total within territory (ordered by visits)
        SUM(total_visits) OVER (
            PARTITION BY territory_id
            ORDER BY total_visits DESC
            ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
        )            AS running_total_in_territory,

        -- What % of territory's total visits does this patient represent?
        ROUND(
            total_visits * 100.0 /
            SUM(total_visits) OVER (PARTITION BY territory_id),
            2
        )            AS pct_of_territory_visits
    FROM patient_visit_summary
)

-- Step 3: Get top 3 patients per territory
SELECT
    territory_name,
    region,
    rank_in_territory,
    patient_id,
    patient_name,
    total_visits,
    pct_of_territory_visits,
    global_rank,
    visit_quartile,
    last_visit_date
FROM patient_ranked
WHERE rank_in_territory <= 3        -- top 3 per territory
ORDER BY region, territory_name, rank_in_territory;
```

**Key differences between ranking functions — interviewers love this:**
```sql
-- Example data: visits = [10, 10, 8, 5]

SELECT
    patient_id,
    total_visits,
    RANK()       OVER (ORDER BY total_visits DESC) AS rnk,        -- 1, 1, 3, 4 (skips 2)
    DENSE_RANK() OVER (ORDER BY total_visits DESC) AS dense_rnk,  -- 1, 1, 2, 3 (no gaps)
    ROW_NUMBER() OVER (ORDER BY total_visits DESC) AS row_num,     -- 1, 2, 3, 4 (always unique)
    NTILE(2)     OVER (ORDER BY total_visits DESC) AS quartile     -- 1, 1, 2, 2 (bucket)
FROM patient_visit_summary;

-- RANK:       ties get same rank, NEXT rank skips (1,1,3)
-- DENSE_RANK: ties get same rank, next rank does NOT skip (1,1,2)
-- ROW_NUMBER: every row gets unique number (ties broken arbitrarily)
-- NTILE(n):   divides rows into n equal buckets
```

### P2. Optimize a Slow SQL with Joins on Big Tables

**Scenario:** This query runs for 45 minutes on the AbbVie data. Find and fix all problems.

```sql
-- ❌ SLOW ORIGINAL QUERY — 45 minutes
SELECT
    p.patient_id,
    p.patient_name,
    t.territory_name,
    t.region,
    d.diagnosis_description,
    COUNT(e.event_id)    AS visit_count,
    SUM(e.revenue)       AS total_revenue
FROM patient_events e
JOIN patients p
    ON e.patient_id = p.patient_id
JOIN territories t
    ON p.territory_id = t.territory_id
JOIN diagnosis_codes d
    ON e.diagnosis_code = d.code
WHERE YEAR(e.event_date) = 2024             -- ❌ Problem 1
  AND UPPER(p.status) = 'ACTIVE'            -- ❌ Problem 2
  AND t.region IN (
        SELECT region FROM territories      -- ❌ Problem 3
        WHERE country = 'US'
      )
GROUP BY p.patient_id, p.patient_name,
         t.territory_name, t.region, d.diagnosis_description
HAVING SUM(e.revenue) > (                  -- ❌ Problem 4
    SELECT AVG(total_rev)
    FROM (
        SELECT patient_id, SUM(revenue) AS total_rev
        FROM patient_events
        GROUP BY patient_id
    ) sub
)
ORDER BY total_revenue DESC;               -- ❌ Problem 5 (sort on big result)
```

**Problems identified and fixed:**
```sql
-- ❌ Problem 1: YEAR(e.event_date) = 2024
--    Function on column → prevents index/partition use
--    Fix: use range filter on the raw column

-- ❌ Problem 2: UPPER(p.status) = 'ACTIVE'
--    Function wrapping column → index can't be used
--    Fix: standardize at write time, filter on raw column

-- ❌ Problem 3: IN (SELECT ...) correlated subquery on large table
--    Executed once per row → O(n) subquery executions
--    Fix: pre-filter territories before joining

-- ❌ Problem 4: HAVING with correlated aggregate subquery
--    Computes AVG over entire table on every row evaluation
--    Fix: compute once in CTE

-- ❌ Problem 5: ORDER BY on huge unsorted result
--    Forces full sort after aggregation
--    Fix: limit rows before sorting, or sort only final small result

-- ✅ OPTIMIZED QUERY
WITH
-- Pre-compute the average revenue once (not once per row)
avg_revenue AS (
    SELECT AVG(total_rev) AS avg_patient_revenue
    FROM (
        SELECT patient_id, SUM(revenue) AS total_rev
        FROM patient_events
        WHERE event_date >= '2024-01-01'         -- ✅ range filter, partition-friendly
          AND event_date <  '2025-01-01'
        GROUP BY patient_id
    ) sub
),

-- Pre-filter territories to US only (small result → use in join)
us_territories AS (
    SELECT territory_id, territory_name, region
    FROM territories
    WHERE country = 'US'                         -- ✅ filter before join
),

-- Main aggregation
patient_summary AS (
    SELECT
        p.patient_id,
        p.patient_name,
        t.territory_name,
        t.region,
        d.diagnosis_description,
        COUNT(e.event_id)    AS visit_count,
        SUM(e.revenue)       AS total_revenue
    FROM patient_events e
    -- ✅ Join with pre-filtered territory (smaller table)
    JOIN us_territories t    ON e.territory_id  = t.territory_id
    -- ✅ diagnosis_codes is small → broadcast hint
    JOIN diagnosis_codes d   ON e.diagnosis_code = d.code
    -- ✅ patients joined after territory filter reduces rows
    JOIN patients p          ON e.patient_id    = p.patient_id
    WHERE e.event_date >= '2024-01-01'           -- ✅ range, no function wrapping
      AND e.event_date <  '2025-01-01'
      AND p.status = 'ACTIVE'                    -- ✅ standardized at write time
    GROUP BY p.patient_id, p.patient_name,
             t.territory_name, t.region, d.diagnosis_description
)

-- Apply threshold filter using pre-computed CTE
SELECT
    ps.patient_id,
    ps.patient_name,
    ps.territory_name,
    ps.region,
    ps.diagnosis_description,
    ps.visit_count,
    ps.total_revenue
FROM patient_summary ps
CROSS JOIN avg_revenue ar                        -- ✅ single row cross join (cheap)
WHERE ps.total_revenue > ar.avg_patient_revenue
ORDER BY ps.total_revenue DESC
LIMIT 1000;                                      -- ✅ limit before returning
```

**Optimization summary:**
```
Problem                          Fix                               Impact
────────────────────────────────────────────────────────────────────────────
YEAR(event_date) = 2024          range filter: >= AND <            Partition pruning enabled
UPPER(status) = 'ACTIVE'         standardize at write time         Index/stats usable
IN (SELECT ...) per row          pre-filter in CTE                 O(n) → O(1) subquery
HAVING + correlated subquery     pre-compute AVG in CTE            Computed once, not per row
JOIN order: large first          join small filtered tables first  Less data shuffled
No table size hints              broadcast small tables in Spark   Eliminates join shuffle
Full result ORDER BY             LIMIT before ORDER BY             Sort on 1000 rows not 1M
```

**In Spark (PySpark equivalent):**
```python
from pyspark.sql.functions import broadcast, col, count, sum as spark_sum, avg

# Pre-compute average (collect as scalar)
avg_revenue = patient_events_df \
    .filter((col("event_date") >= "2024-01-01") & (col("event_date") < "2025-01-01")) \
    .groupBy("patient_id").agg(spark_sum("revenue").alias("total_rev")) \
    .agg(avg("total_rev")).collect()[0][0]

# Pre-filter territories
us_territories = territories_df.filter(col("country") == "US")

# Main query with broadcast hints
result = patient_events_df \
    .filter(col("event_date") >= "2024-01-01") \
    .filter(col("event_date") <  "2025-01-01") \
    .join(broadcast(us_territories),    on="territory_id",  how="inner") \
    .join(broadcast(diagnosis_codes_df), on="diagnosis_code", how="inner") \
    .join(patients_df.filter(col("status") == "ACTIVE"), on="patient_id") \
    .groupBy("patient_id", "patient_name", "territory_name", "region", "diagnosis_description") \
    .agg(
        count("event_id").alias("visit_count"),
        spark_sum("revenue").alias("total_revenue")
    ) \
    .filter(col("total_revenue") > avg_revenue) \
    .orderBy(col("total_revenue").desc()) \
    .limit(1000)
```

### P3. Convert a Correlated Subquery into a JOIN

**Scenario:** Classic optimization — correlated subqueries execute once per outer row (O(n)), joins execute once (O(n log n) or better).

```sql
-- ❌ CORRELATED SUBQUERY VERSION (slow — executes N times)
-- "Find patients whose visit count is above their territory's average"
SELECT
    p.patient_id,
    p.territory_id,
    p.total_visits
FROM patient_summary p
WHERE p.total_visits > (
    SELECT AVG(p2.total_visits)             -- ← this runs ONCE PER ROW of outer query
    FROM patient_summary p2
    WHERE p2.territory_id = p.territory_id  -- ← correlated condition
);

-- How it executes:
-- Row 1: patient P001, territory T1 → run inner query for T1 → compare
-- Row 2: patient P002, territory T1 → run inner query for T1 AGAIN → compare
-- Row 3: patient P003, territory T2 → run inner query for T2 → compare
-- ... repeat for ALL 5 million patients → 5 million inner query executions ❌
```

```sql
-- ✅ REWRITTEN AS JOIN (fast — executes once)

-- Step 1: Pre-compute the aggregate for each territory ONCE
WITH territory_averages AS (
    SELECT
        territory_id,
        AVG(total_visits)   AS avg_visits,
        MAX(total_visits)   AS max_visits,
        COUNT(patient_id)   AS patient_count
    FROM patient_summary
    GROUP BY territory_id
    -- Executes ONCE and result is materialized
)

-- Step 2: Join pre-computed result (one row per territory) back to patients
SELECT
    p.patient_id,
    p.territory_id,
    p.total_visits,
    ta.avg_visits              AS territory_avg,
    p.total_visits - ta.avg_visits AS above_avg_by
FROM patient_summary p
JOIN territory_averages ta
    ON p.territory_id = ta.territory_id   -- ← simple equi-join, executed once
WHERE p.total_visits > ta.avg_visits;

-- How it executes:
-- Step 1: compute territory_averages once → ~500 rows (500 territories)
-- Step 2: hash join patient_summary (5M rows) with territory_averages (500 rows)
-- → territory_averages is broadcast (500 rows is tiny) → no shuffle on patient side ✅
```

**More correlated subquery patterns to join rewrites:**

```sql
-- ❌ CORRELATED: "Latest event per patient" (classic N+1 problem)
SELECT *
FROM patient_events e1
WHERE e1.event_date = (
    SELECT MAX(e2.event_date)
    FROM patient_events e2
    WHERE e2.patient_id = e1.patient_id  -- runs once per row
);

-- ✅ REWRITTEN: Window function (much better)
SELECT *
FROM (
    SELECT *,
           ROW_NUMBER() OVER (PARTITION BY patient_id ORDER BY event_date DESC) AS rn
    FROM patient_events
) ranked
WHERE rn = 1;

-- ✅ ALTERNATIVE: CTE + JOIN
WITH latest_per_patient AS (
    SELECT patient_id, MAX(event_date) AS latest_date
    FROM patient_events
    GROUP BY patient_id
)
SELECT e.*
FROM patient_events e
JOIN latest_per_patient l
    ON e.patient_id = l.patient_id
    AND e.event_date = l.latest_date;
```

```sql
-- ❌ CORRELATED: EXISTS check (common reconciliation pattern)
SELECT p.patient_id
FROM patients p
WHERE EXISTS (
    SELECT 1
    FROM patient_events e
    WHERE e.patient_id = p.patient_id    -- runs once per patient
      AND e.event_date >= '2024-01-01'
);

-- ✅ REWRITTEN: Semi-join via CTE + INNER JOIN
WITH active_2024_patients AS (
    SELECT DISTINCT patient_id           -- deduplicate
    FROM patient_events
    WHERE event_date >= '2024-01-01'
)
SELECT p.patient_id
FROM patients p
JOIN active_2024_patients a ON p.patient_id = a.patient_id;
-- Or: just use INNER JOIN directly
SELECT DISTINCT p.patient_id
FROM patients p
JOIN patient_events e
    ON p.patient_id = e.patient_id
    AND e.event_date >= '2024-01-01';
```

**Rule of thumb — when to convert:**
```
Correlated subquery is safe when:
  → Returns a scalar (single value) AND outer table is small (< 10k rows)
  → Database optimizer auto-flattens it (modern optimizers do this sometimes)

Always convert to join when:
  → Outer table is large (millions of rows)
  → Correlated condition is on a big table
  → Query is being run repeatedly (reports, dashboards)
  → Subquery contains GROUP BY or aggregate (use CTE + JOIN)
  → Subquery is EXISTS/NOT EXISTS with large tables (use JOIN / ANTI JOIN)
```

## 📊 Quick Reference Cheat Sheet

### Window Function Syntax
```sql
function_name(column)
OVER (
    PARTITION BY grouping_column    -- defines the "window group" (like GROUP BY)
    ORDER BY ordering_column        -- defines order within the window
    ROWS BETWEEN                    -- optional: frame definition
        UNBOUNDED PRECEDING         --   from the start of the window
        AND CURRENT ROW             --   to the current row
)
```

### Execution Plan Reading Guide
```
Key things to look for          What it means
─────────────────────────────────────────────────────────
Seq Scan (full scan)           No index used → add index or partition filter
Index Scan                     Index being used ✅
BroadcastHashJoin              Small table broadcasted ✅ (Spark)
SortMergeJoin                  Large shuffle happening → can we broadcast?
Hash Join                      Good for large-large equi-joins
Nested Loop                    ❌ O(n²) — only OK for tiny tables
High cost node                 Optimize THIS operation first
Estimated rows << Actual rows  Stale statistics → run ANALYZE TABLE
```

### SQL Join Decision Guide
```
Need rows from BOTH tables only?          → INNER JOIN
Need ALL left rows + matching right?      → LEFT JOIN
Need to find unmatched left rows?         → ANTI JOIN (NOT EXISTS)
Need ALL rows from both sides?            → FULL OUTER JOIN
Need to cross-multiply (rare)?            → CROSS JOIN
Need a single scalar from right side?     → CROSS JOIN with aggregated CTE
```

### Correlated → JOIN Conversion Recipe
```
1. Take the correlated subquery
2. Turn it into a CTE with GROUP BY (pre-compute the aggregate)
3. JOIN the CTE back to the main query on the correlated column
4. Filter using the JOIN result instead of the subquery
```
