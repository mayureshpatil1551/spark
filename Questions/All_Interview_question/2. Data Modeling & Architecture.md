### Q1. What is a Data Lake vs Data Warehouse vs Lakehouse?

**One-line answer:**
> Data Lake = raw everything, cheap storage, schema-on-read. Data Warehouse = structured, optimized for queries, schema-on-write. Lakehouse = best of both — raw storage + warehouse-grade performance and ACID transactions.

---

“A Data Lake, Data Warehouse, and Lakehouse are different architectures used for storing and analyzing data.

A Data Lake is designed to store large volumes of raw structured, semi-structured, and unstructured data such as JSON, XML, logs, images, and streaming data. It follows a schema-on-read approach, meaning data is stored first and interpreted later during processing. Technologies like Amazon S3, Azure Data Lake, and Hadoop HDFS are commonly used. Data Lakes are cost-effective and highly scalable, making them suitable for big data and machine learning workloads, but traditional lakes have challenges around governance, ACID transactions, and query performance.

A Data Warehouse is built specifically for structured analytical reporting and business intelligence workloads. It follows a schema-on-write approach, where data is cleaned and transformed before loading into the warehouse. Technologies like Snowflake, Redshift, BigQuery, and Teradata are common examples. Data Warehouses provide fast SQL performance, strong governance, and optimized reporting capabilities, but they are generally more expensive and less flexible for handling semi-structured or unstructured data.

A Lakehouse combines the flexibility and scalability of a Data Lake with the performance and reliability features of a Data Warehouse. It stores data in open formats like Parquet on object storage such as S3 while adding capabilities like ACID transactions, schema enforcement, time travel, and optimized query execution. Technologies such as Apache Iceberg, Delta Lake, and Apache Hudi are examples of Lakehouse implementations. Lakehouses support analytics, BI, streaming, and machine learning on a single platform, reducing the need for separate storage systems.

In simple terms:

* Data Lake stores raw scalable data.
* Data Warehouse provides optimized structured analytics.
* Lakehouse combines both by bringing warehouse capabilities directly into the data lake architecture.

In modern data engineering, Lakehouse architecture is becoming the preferred approach because it offers scalability, governance, performance, and flexibility in a unified platform.”

---

**Detailed Explanation:**

#### 🔹 Data Lake

A Data Lake stores **raw data in its native format** (CSV, JSON, Parquet, XML, images, logs) at massive scale on cheap cloud storage (S3, ADLS).

| Property | Detail |
|---|---|
| **Storage** | Object storage (S3, ADLS Gen2) |
| **Schema** | Schema-on-read (no schema at write time) |
| **Format** | Any format — CSV, JSON, Parquet, Avro |
| **Cost** | Very cheap (cents per GB) |
| **Performance** | Slower for queries (no indexing) |
| **ACID** | ❌ None (data can be corrupt/partial) |
| **Users** | Data Engineers, Data Scientists |

```
Raw Zone:  s3://abbvie-raw/veeva/patients.csv        ← just dropped, no schema
           s3://abbvie-raw/api/ravex_response.json   ← raw API response
           s3://abbvie-raw/finops/report.xml         ← raw XML
```

**Problem with pure Data Lakes ("Data Swamp"):**
- No schema enforcement → garbage data gets in
- No ACID → partial writes cause corruption
- No versioning → can't roll back bad data
- Poor query performance → analysts can't use it directly

---

#### 🔹 Data Warehouse

A Data Warehouse stores **structured, modeled, cleansed data** optimized for analytics queries.

| Property | Detail |
|---|---|
| **Storage** | Columnar (Parquet-like internally) |
| **Schema** | Schema-on-write (enforced at load) |
| **Format** | Proprietary or columnar |
| **Cost** | Expensive (compute + storage bundled) |
| **Performance** | Very fast for analytics (indexes, stats) |
| **ACID** | ✅ Full ACID |
| **Users** | Analysts, BI tools, Business users |
| **Examples** | Snowflake, Redshift, BigQuery, Synapse |

**Problem with pure Data Warehouses:**
- Expensive at scale (especially compute)
- Can't store unstructured/semi-structured data
- Rigid schema — hard to evolve as business changes
- ETL pipeline required before data is usable

---

#### 🔹 Lakehouse

A Lakehouse combines **cheap data lake storage** with **data warehouse features** (ACID, schema enforcement, fast queries).

| Property | Detail |
|---|---|
| **Storage** | Object storage (S3, ADLS) — cheap |
| **Schema** | Schema-on-write (enforced via table format) |
| **Format** | Open format: Delta Lake, Apache Iceberg, Hudi |
| **Cost** | Cheap storage, pay-as-you-go compute |
| **Performance** | Near-warehouse speed via metadata + data skipping |
| **ACID** | ✅ Full ACID (via transaction log) |
| **Users** | Everyone — Engineers, Scientists, Analysts |
| **Examples** | Databricks (Delta), AbbVie stack (Iceberg) |

```
Lakehouse on Iceberg (AbbVie):
Bronze: s3://abbvie/iceberg/bronze/veeva_patients/   ← raw but tracked
Silver: s3://abbvie/iceberg/silver/patient_master/   ← cleansed, ACID, schema-enforced
Gold:   s3://abbvie/iceberg/gold/patient_summary/    ← analytics-ready
```

#### 🔹 Side-by-Side Comparison

| Feature | Data Lake | Data Warehouse | Lakehouse |
|---|---|---|---|
| Storage cost | 💰 Cheap | 💰💰💰 Expensive | 💰 Cheap |
| Schema | Read-time | Write-time | Write-time |
| ACID | ❌ | ✅ | ✅ |
| Unstructured data | ✅ | ❌ | ✅ |
| ML/AI workloads | ✅ | ❌ | ✅ |
| BI/SQL queries | ❌ Slow | ✅ Fast | ✅ Fast |
| Time travel | ❌ | Limited | ✅ |
| Open format | ✅ | ❌ Proprietary | ✅ |

**Interview tip:** "At AbbVie, we used a Lakehouse architecture. Raw data from Veeva, Ravex, Smartsheet APIs landed on S3 (data lake). Apache Iceberg provided ACID transactions, schema enforcement, and metadata-driven file skipping — giving us warehouse-grade performance on cheap S3 storage. That's the Lakehouse pattern."

### Q2. Explain Star Schema vs Snowflake Schema

**One-line answer:**
> Star schema = one central fact table joined directly to denormalized dimension tables (fast queries, simple joins). Snowflake schema = dimensions are further normalized into sub-dimensions (saves storage, more complex joins).

---

**Detailed Explanation:**

#### 🔹 Star Schema

```
                    ┌─────────────┐
                    │  DIM_DATE   │
                    │  date_key   │
                    │  year       │
                    │  quarter    │
                    │  month      │
                    └──────┬──────┘
                           │
┌──────────────┐   ┌───────┴────────┐   ┌──────────────────┐
│ DIM_TERRITORY│   │  FACT_PATIENT  │   │  DIM_PATIENT     │
│ territory_key├───┤  territory_key │   │  patient_key     │
│ territory_id │   │  patient_key   ├───┤  patient_id      │
│ region       │   │  date_key      │   │  name            │
│ country      │   │  diagnosis_key │   │  age             │
│ manager      │   │  visit_count   │   │  gender          │
└──────────────┘   │  drug_spend    │   │  enrollment_date │
                   └───────┬────────┘   └──────────────────┘
                           │
                    ┌──────┴──────┐
                    │ DIM_DIAGNOS │
                    │ diag_key    │
                    │ icd_code    │
                    │ description │
                    │ category    │
                    └─────────────┘
```

**Characteristics:**
- Fact table in the center, dimensions around it (like a star ⭐)
- Dimensions are **denormalized** (all attributes in one table)
- **Simple, fast queries** — typically 1 or 2 joins to get full context
- **More storage** — repeated values in dimension tables
- Used in: **Reporting, BI dashboards, Tableau/Power BI**

```sql
-- Simple star schema query — 2 joins max
SELECT t.region, d.month, SUM(f.drug_spend)
FROM fact_patient f
JOIN dim_territory t ON f.territory_key = t.territory_key
JOIN dim_date d      ON f.date_key = d.date_key
GROUP BY t.region, d.month;
```

---

#### 🔹 Snowflake Schema

```
┌──────────────┐     ┌──────────────┐
│  DIM_COUNTRY │     │  DIM_REGION  │
│  country_key │◄────┤  region_key  │
│  country_name│     │  region_name │
└──────────────┘     │  country_key │
                     └──────┬───────┘
                            │
┌──────────────┐   ┌────────┴───────┐
│ DIM_TERRITORY│   │  FACT_PATIENT  │
│ territory_key├───┤  territory_key │
│ territory_id │   │  patient_key   │
│ territory_nm │   │  ...           │
│ region_key   │   └────────────────┘
└──────────────┘
```

**Characteristics:**
- Dimensions are **normalized** into sub-dimensions
- Saves storage — no repeated region/country values
- **More complex queries** — more joins needed
- Used in: **OLTP-influenced warehouses, storage-sensitive systems**

#### 🔹 Star vs Snowflake — When to Use

| | Star Schema | Snowflake Schema |
|---|---|---|
| **Query speed** | ✅ Faster (fewer joins) | ❌ Slower (more joins) |
| **Storage** | More (denormalized) | Less (normalized) |
| **Maintenance** | Easier | Harder |
| **Best for** | Analytics / BI / Reporting | Storage efficiency |
| **Used with** | Spark, Tableau, Power BI | Data marts |

**Interview tip:** "In healthcare analytics at AbbVie, we used a Star schema for the Gold layer. Fact table was patient visit events; dimensions were territory, date, diagnosis, and product. Analysts using Power BI needed fast, simple joins — Star schema delivered that."

### Q3. What is Dimensional Modeling?

**One-line answer:**
> Dimensional modeling is a technique to design analytical databases using **fact tables** (measurements/events) and **dimension tables** (context/who/what/when/where) — optimized for business intelligence queries.

---

Dimensional Modeling is a data modeling technique used in Data Warehousing to design data structures optimized for analytical queries, reporting, and business intelligence.

It organizes data into:

           Fact tables
           Dimension tables

Fact tables store measurable business metrics such as:

           sales amount
           quantity
           revenue
           transactions

Dimension tables store descriptive business attributes such as:

           customer details
           product information
           date
           region

For example, in a retail system:

           sales_fact may contain sales_amount and quantity
           customer_dim stores customer details
           product_dim stores product information
           date_dim stores calendar attributes

The main goal of dimensional modeling is to make data easy to understand and optimize query performance for analytics.

The two most common dimensional models are:
           
           Star Schema
           Snowflake Schema

Dimensional modeling is widely used in:

           Data Warehouses
           BI dashboards
           Reporting systems
           OLAP analytics

Its major advantages are:

           Faster query performance
           Simplified business reporting
           Easy-to-understand structure
           Better aggregation and analytical capabilities

In modern data engineering, dimensional modeling is still heavily used in Lakehouse architectures for building Gold-layer business reporting tables.

One-line summary:
‘Dimensional Modeling is a technique used in Data Warehousing where business data is organized into fact and dimension tables to optimize analytical querying and reporting.’

---


**Detailed Explanation:**

#### 🔹 Core Concepts

**Fact Table:**
- Stores **measurable, quantitative data** (metrics/events)
- Each row = one business event
- Contains **foreign keys** to dimension tables
- Contains **measures** (numbers you aggregate)
- Examples: sales, visits, prescriptions, transactions

```
FACT_PATIENT_VISIT
├── visit_key        (surrogate key)
├── patient_key      (FK → DIM_PATIENT)
├── territory_key    (FK → DIM_TERRITORY)
├── date_key         (FK → DIM_DATE)
├── diagnosis_key    (FK → DIM_DIAGNOSIS)
├── visit_count      ← MEASURE (additive)
├── drug_spend       ← MEASURE (additive)
└── avg_duration_min ← MEASURE (semi-additive)
```

**Dimension Table:**
- Stores **descriptive, contextual data** (who, what, when, where)
- Each row = one entity (one patient, one territory, one date)
- Contains **surrogate key** (system-generated integer)
- Contains **business key** (natural key from source)
- Contains **descriptive attributes** used for grouping/filtering

```
DIM_PATIENT
├── patient_key         (surrogate key — PK)
├── patient_id          (business/natural key from Veeva)
├── patient_name        
├── date_of_birth       
├── gender              
├── enrollment_date     
├── insurance_type      
└── effective_from/to   (if SCD Type 2)
```

#### 🔹 Types of Facts/Measures

| Type | Description | Example |
|---|---|---|
| **Additive** | Can SUM across all dimensions | Revenue, visit count |
| **Semi-additive** | Can SUM across some, not all | Account balance (not time) |
| **Non-additive** | Cannot SUM at all | Ratios, percentages, rates |

#### 🔹 Types of Dimension Tables

| Type | Description | Example |
|---|---|---|
| **Conformed** | Shared across multiple fact tables | DIM_DATE, DIM_TERRITORY |
| **Junk** | Miscellaneous flags/indicators | Enrollment flags, Y/N fields |
| **Degenerate** | Dimension stored in fact table | Order number in sales fact |
| **Role-playing** | Same dimension used multiple times | DIM_DATE as order_date + ship_date |

**Interview tip:** "When designing the eCDP (healthcare) schema at AbbVie, I applied dimensional modeling. Patient visits were facts, territory/diagnosis/date were dimensions. This made Power BI/Tableau queries fast and self-explanatory to business analysts."

### Q4. What is a Surrogate Key?

**One-line answer:**
> A surrogate key is a **system-generated, meaningless integer or UUID** used as the primary key of a dimension table — replacing the natural/business key from the source system.

---

**Detailed Explanation:**

#### 🔹 Why Not Just Use the Natural Key?

Natural keys (business keys) from source systems have problems:

| Problem | Example |
|---|---|
| Can change over time | Patient ID reassigned in source system |
| Not unique across systems | Same patient in Veeva and Oracle has different IDs |
| Long strings (slow joins) | "ABBVIE-VEEVA-US-PATIENT-2024-001234" |
| May contain business meaning that changes | Product code includes region code |

#### 🔹 Surrogate Key Properties

```
DIM_PATIENT
├── patient_key    INTEGER  (surrogate — generated by data warehouse: 1, 2, 3...)
├── patient_id     STRING   (natural/business key from Veeva: "VEV-001234")
├── patient_name   STRING
└── ...

FACT_PATIENT_VISIT
├── patient_key    INTEGER  ← joins to surrogate key (fast integer join)
├── ...
```

✅ **Surrogate key advantages:**
- **Integer joins** are much faster than string joins (patient_id = "VEV-001234")
- **Insulates** the warehouse from source system changes
- **Enables SCD Type 2** — same patient can have multiple rows with different surrogate keys for different time periods
- **Unique across all sources** — even if Veeva and Oracle both have "P001"

#### 🔹 Surrogate Key in SCD Type 2

```
DIM_PATIENT (with SCD Type 2)
patient_key | patient_id | territory_id | effective_from | effective_to | is_current
1           | P001       | T-US-01      | 2023-01-01     | 2024-03-15   | FALSE  ← old record
2           | P001       | T-US-05      | 2024-03-16     | 9999-12-31   | TRUE   ← current record

# Same business key (P001) — two different surrogate keys
# FACT table references patient_key 1 for visits before 2024-03-16
# FACT table references patient_key 2 for visits after 2024-03-16
```

#### 🔹 Generating Surrogate Keys in PySpark

```python
from pyspark.sql.functions import monotonically_increasing_id, sha2, concat_ws, col

# Option 1: Integer surrogate key (simple, but not stable across runs)
df_dim = df.withColumn("patient_key", monotonically_increasing_id())

# Option 2: Hash-based surrogate key (deterministic, reproducible)
df_dim = df.withColumn(
    "patient_key",
    sha2(concat_ws("||", col("patient_id"), col("source_system")), 256)
)
# Same patient_id + source_system always generates the same hash
# Safe to use as stable surrogate key
```

**Interview tip:** "In the eCDP data model at AbbVie, we used hash-based surrogate keys on patient_id + source_system. This allowed us to merge data from Veeva and Oracle into the same patient dimension while maintaining uniqueness and fast joins."

### Q5. What is OLTP vs OLAP?

**One-line answer:**
> OLTP (Online Transaction Processing) = many small, fast read/write operations (operational systems). OLAP (Online Analytical Processing) = complex queries over large historical datasets (analytics systems).

---

**Detailed Explanation:**

| Feature | OLTP | OLAP |
|---|---|---|
| **Purpose** | Day-to-day operations | Analytics and reporting |
| **Query type** | INSERT/UPDATE/DELETE on few rows | SELECT aggregations over millions of rows |
| **Data volume** | GB range (current data) | TB–PB range (historical data) |
| **Response time** | Milliseconds | Seconds to minutes |
| **Schema** | Normalized (3NF) | Denormalized (star/snowflake) |
| **Indexing** | Row-based (B-tree index) | Columnar, partitioning, Z-ordering |
| **Users** | Application users, transactions | Analysts, data scientists |
| **Examples** | Salesforce, Oracle ERP, MySQL | Snowflake, Redshift, Databricks SQL |
| **Concurrent users** | Thousands | Tens to hundreds |
| **Data freshness** | Real-time | Batch (daily/hourly) |

#### 🔹 OLTP Example (Oracle at AbbVie)
```sql
-- OLTP: Quick lookup + update for one patient
UPDATE patient_master
SET status = 'INACTIVE', updated_at = SYSDATE
WHERE patient_id = 'VEV-001234';
-- Fast, index on patient_id, touches 1 row
```

#### 🔹 OLAP Example (Iceberg/Spark at AbbVie)
```sql
-- OLAP: Aggregate over millions of rows
SELECT territory_id,
       COUNT(DISTINCT patient_id) AS patient_count,
       SUM(drug_spend)            AS total_spend
FROM fact_patient_events
WHERE event_date BETWEEN '2024-01-01' AND '2024-12-31'
  AND region = 'US'
GROUP BY territory_id
ORDER BY total_spend DESC;
-- Touches 50M rows, benefits from partition pruning + columnar storage
```

#### 🔹 How Data Flows Between Them

```
OLTP Systems (Source)              OLAP Systems (Target)
┌────────────────────┐             ┌───────────────────────────┐
│ Veeva CRM (Oracle) │──ETL/ELT──▶│ Iceberg Lakehouse (AbbVie)│
│ Ravex API          │             │ Bronze → Silver → Gold     │
│ Smartsheet         │             │ Power BI / Tableau on top  │
└────────────────────┘             └───────────────────────────┘
  Operational data                   Analytical data
```

**Interview tip:** "At AbbVie, the Oracle database was our OLTP source — Veeva and other systems wrote operational data there. My job as a Data Engineer was to extract from OLTP sources and build the OLAP layer on Iceberg — designed for analytical queries, not row-level transactions."

### Q6. What is Normalization vs Denormalization?

**One-line answer:**
> Normalization = split data into many tables to remove redundancy (OLTP-friendly). Denormalization = combine tables to reduce joins (OLAP-friendly).

---

**Detailed Explanation:**

#### 🔹 Normalization (1NF → 2NF → 3NF)

Goal: **Eliminate data redundancy**, ensure data integrity.

**Without normalization (redundancy problem):**
```
patient_visits (denormalized — BAD for OLTP)
patient_id | patient_name | patient_age | territory_id | territory_name | region | visit_date
P001       | John Smith   | 45          | T-US-01      | Chicago North  | US     | 2024-01-10
P001       | John Smith   | 45          | T-US-01      | Chicago North  | US     | 2024-02-15
P001       | John Smith   | 45          | T-US-01      | Chicago North  | US     | 2024-03-20
```
Problems: "John Smith" stored 3 times. Update his age → update 3 rows. Delete visits → lose patient data.

**With normalization (3NF):**
```sql
-- Patients table
patients: patient_id | patient_name | patient_age

-- Territories table
territories: territory_id | territory_name | region

-- Visits table (junction)
visits: visit_id | patient_id (FK) | territory_id (FK) | visit_date
```
Each piece of data stored ONCE. ✅

#### 🔹 Normal Forms Summary

| Form | Rule | Example |
|---|---|---|
| **1NF** | No repeating groups, atomic values | No comma-separated IDs in one column |
| **2NF** | 1NF + no partial dependencies | Remove columns dependent on part of PK |
| **3NF** | 2NF + no transitive dependencies | Remove columns dependent on non-key column |

#### 🔹 Denormalization

For analytics (OLAP), we deliberately **reverse normalization** to:
- Reduce join complexity (fewer tables to JOIN)
- Improve query speed (less shuffle in Spark)
- Make data self-explanatory for analysts

```
gold_patient_events (denormalized — GOOD for analytics)
patient_id | patient_name | age | territory_id | territory_name | region | 
visit_date | diagnosis_code | diagnosis_desc | drug_name | drug_spend
```
Yes, territory_name and region are repeated — but queries are simple, fast, analyst-friendly.

#### 🔹 When to Use Each

| Scenario | Approach | Why |
|---|---|---|
| OLTP / operational DB | Normalization (3NF) | Data integrity, update efficiency |
| Bronze Layer | Preserve source structure | Don't alter raw data |
| Silver Layer | Light normalization | Split out lookup tables |
| Gold Layer | Denormalization | Fast queries, simple for BI tools |
| Dimensional model | Star schema denormalization | Analytics performance |

**Interview tip:** "In my AbbVie pipeline, Bronze was raw (source structure preserved), Silver was lightly normalized (patient master separate from events), Gold was denormalized — territory name, region, diagnosis description all included in one wide table for direct Power BI consumption."

### Q7. What is Partitioning Strategy in Delta/Iceberg?

**One-line answer:**
> Partitioning divides table data into separate folders/files based on column values — so queries that filter on that column scan only relevant partitions, skipping the rest (partition pruning).

---

**Detailed Explanation:**

#### 🔹 How Partitioning Works Physically

```
iceberg.silver.patient_events
partitioned by (region, load_date)

S3 folder structure:
s3://abbvie/silver/patient_events/
├── region=US/
│   ├── load_date=2024-01-01/  ← 5 Parquet files, 2 GB
│   ├── load_date=2024-01-02/  ← 5 Parquet files, 2 GB
│   └── load_date=2024-01-03/  ← 5 Parquet files, 2 GB
├── region=EU/
│   ├── load_date=2024-01-01/  ← 3 Parquet files, 1.2 GB
│   └── ...
└── region=APAC/
    └── ...

Query: WHERE region = 'US' AND load_date = '2024-01-01'
→ Spark opens ONLY: region=US/load_date=2024-01-01/ ← 2 GB out of 50 GB total ✅
```

#### 🔹 Good Partitioning Strategy

**Rules of thumb:**

| Rule | Detail |
|---|---|
| **Cardinality** | Low-medium cardinality (5–500 unique values) |
| **Query filter** | Columns used in WHERE clauses most often |
| **Data volume** | Partition size: aim for 100 MB – 1 GB per partition |
| **Time column** | Dates are the most common partition column |
| **Write pattern** | Matches how data is ingested (daily batch → partition by date) |

#### 🔹 Common Patterns

```python
# Pattern 1: Date-only (most common for time-series data)
df.writeTo("iceberg.silver.patient_events") \
    .partitionedBy("load_date") \
    .append()

# Pattern 2: Region + Date (hierarchical — good for geo + time filters)
df.writeTo("iceberg.silver.patient_events") \
    .partitionedBy("region", "load_date") \
    .append()

# Pattern 3: Iceberg hidden partitions (no column needed in data)
# Iceberg can partition by YEAR/MONTH/DAY of a timestamp column
CREATE TABLE iceberg.silver.patient_events (
    patient_id STRING,
    event_ts   TIMESTAMP,
    ...
) USING iceberg
PARTITIONED BY (months(event_ts))  -- Iceberg hidden partition by month
# Data still has event_ts column — no separate partition column needed!

# Pattern 4: Bucket partitioning (for high-cardinality joins)
PARTITIONED BY (bucket(100, patient_id))  -- Hash into 100 buckets
```

#### 🔹 Partition + Z-Order Together

```
Partitioning = coarse-grained data organization (folder level)
Z-Ordering  = fine-grained data organization (file level within partition)

Strategy: Partition on low-cardinality column + Z-Order on high-cardinality column

PARTITION BY: region, load_date        (5 regions × 365 days = manageable)
ZORDER BY:    patient_id, territory_id (millions of unique values)
```

**Interview tip:** "In my AbbVie Iceberg tables, I partitioned by region and load_date — the two most common filter columns in downstream queries. This reduced scan size from 20 TB to ~500 GB for typical daily reports. Partition design was key to achieving the 50% processing time reduction."

### Q8. What is Clustering / Z-Ordering and Why Do We Use It?

**One-line answer:**
> Clustering/Z-Ordering physically sorts and co-locates similar data within files using a Z-curve algorithm — so Spark can skip entire files when filtering on clustered columns (data skipping), even within a partition.

---

**Detailed Explanation:**

#### 🔹 The Problem it Solves

Even after partitioning, within a single partition, data can be randomly scattered across many files:

```
region=US / load_date=2024-01-01 / (without Z-Order)
File 1: patient_ids: P001, P500, P9000, P2341, P7723  ← random mix
File 2: patient_ids: P002, P456, P8001, P3344, P6612  ← random mix
File 3: patient_ids: P003, P789, P1001, P4455, P5521  ← random mix

Query: WHERE patient_id = 'P9000'
→ Must scan ALL files (P9000 could be in any file)
```

With Z-Order on `patient_id`:
```
File 1: patient_ids: P001  – P2999  ← sorted range
File 2: patient_ids: P3000 – P5999  ← sorted range
File 3: patient_ids: P6000 – P9999  ← sorted range

Query: WHERE patient_id = 'P9000'
→ Scan ONLY File 3 (stats say max is P9999) ← File skipping! ✅
```

#### 🔹 How Z-Curve Works for Multiple Columns

For single column: simple sort works.
For **multiple columns**: Z-curve interleaves bits of both columns to create a 1D ordering that keeps related values in the same files.

```
Z-Order by (territory_id, patient_id):
Rows with similar territory_id AND patient_id → stored in same files
Query filtering both columns → skips even more files
```

#### 🔹 Data Skipping

Delta/Iceberg stores **min/max statistics** for each column in every file:
```
File metadata:
File 1: patient_id MIN=P001, MAX=P2999 | territory_id MIN=T001, MAX=T099
File 2: patient_id MIN=P3000, MAX=P5999 | territory_id MIN=T100, MAX=T199
...

Query: WHERE patient_id >= 'P8000'
→ Skip File 1 (max P2999 < P8000)
→ Skip File 2 (max P5999 < P8000)
→ Open only files where max >= P8000
```

#### 🔹 How to Apply

```sql
-- Delta: Run OPTIMIZE + ZORDER
OPTIMIZE silver.patient_events
ZORDER BY (patient_id, territory_id);

-- Run after significant data changes (schedule as maintenance job)
-- Typically run after OPTIMIZE completes compaction

-- Iceberg: Set sort order at table creation (applied at write time)
CREATE TABLE iceberg.silver.patient_events
USING iceberg
TBLPROPERTIES (
  'write.sort-order' = 'patient_id ASC NULLS LAST, territory_id ASC NULLS LAST'
)
```

#### 🔹 Partitioning vs Clustering vs Z-Ordering

| | Partitioning | Bucketing | Z-Ordering |
|---|---|---|---|
| **How** | Separate folders | Hash to N buckets | Sort by Z-curve |
| **Best column** | Low cardinality | Join columns | High cardinality |
| **Granularity** | Folder-level skip | File-level skip | File-level skip |
| **Storage overhead** | None | None | Rewrite files |
| **Write cost** | Low | Low | Higher (sort on write) |
| **Iceberg feature** | ✅ | ✅ (bucket) | ✅ (sort order) |

**Interview tip:** "For my 20 TB AbbVie Iceberg tables, I used a two-level strategy: partition by region + load_date for coarse filtering, Z-Order by patient_id + territory_id for fine-grained file skipping. This combination meant typical queries accessing one territory's recent data went from reading 20 TB to reading < 50 GB."

### S2. How Do You Decide Which Column to Partition On?

**One-line answer:**
> Choose a column that: (1) is commonly used in WHERE filters, (2) has low-to-medium cardinality (not millions of unique values), (3) creates balanced partition sizes of 100 MB – 1 GB, and (4) matches your data ingestion pattern.

---

**Decision Framework:**

#### 🔹 Step-by-Step Decision Process

```
Step 1: What columns appear in WHERE clauses 80% of the time?
        → Candidates: event_date, region, status, load_date

Step 2: What is the cardinality of each candidate?
        event_date: ~365 values/year     ← ✅ Good
        region:     5 values             ← ✅ Good
        status:     2 values             ← ⚠️ Too few (large partitions)
        patient_id: 50 million values    ← ❌ Too many (tiny partitions)

Step 3: What size will each partition be?
        Table: 5 TB, 365 days
        Partition by day: 5TB / 365 = ~14 GB per partition  ← OK
        Partition by month: 5TB / 12 = ~417 GB per partition ← Too large
        Partition by hour: 5TB / 8760 = ~570 MB per partition ← ✅ Ideal

Step 4: Does it match how data is ingested?
        Daily batch ingestion → partition by DAY ✅ (writes one partition at a time)
        Streaming ingestion   → partition by HOUR ✅ (finer granularity)
```

#### 🔹 Cardinality Guide

| Cardinality | Value Count | Recommendation |
|---|---|---|
| **Too low** | < 5 values (True/False, M/F) | Don't partition — too few, large partitions |
| **Low** ✅ | 5–50 (regions, status, year) | Good partition candidate |
| **Medium** ✅ | 50–500 (month+region, country) | Good — combine with another column |
| **High** ⚠️ | 500–10,000 (product codes) | Use bucketing or Z-Order instead |
| **Very high** ❌ | > 10,000 (customer_id, order_id) | Never partition — use Z-Ordering |

#### 🔹 Partition Size Guide

```
Target partition size: 100 MB – 1 GB (Parquet files)

Too small (< 10 MB):  Too many files → metadata overhead → slow queries
Too large (> 5 GB):   Few partitions → poor parallelism → slow writes
Sweet spot (512 MB):  Good parallelism + manageable metadata

How to check:
DESCRIBE DETAIL iceberg.silver.patient_events;
-- Check: numFiles, sizeInBytes per partition
```

#### 🔹 Common Patterns at AbbVie

```python
# Pattern 1: Date partition (most common — daily batch ingestion)
df.writeTo("iceberg.silver.patient_events") \
    .partitionedBy("load_date") \
    .append()

# Pattern 2: Region + Date (geo-distributed data)
df.writeTo("iceberg.silver.patient_events") \
    .partitionedBy("region", "load_date") \
    .append()

# Pattern 3: Iceberg hidden partition by MONTH (avoids too many daily partitions)
# No need for separate month column — Iceberg derives it from timestamp
spark.sql("""
    CREATE TABLE iceberg.silver.patient_events
    USING iceberg
    PARTITIONED BY (region, months(event_ts))
""")

# Pattern 4: Status not as partition (bad — only 2 values)
# ❌ df.writeTo(...).partitionedBy("status")  # Creates huge ACTIVE partition
# ✅ Use Z-Order on status, or filter at query time
```

#### 🔹 Red Flags

```
❌ Partition column with NULL values → creates null partition folder
❌ Partition column updated frequently → old partition not updated, new one created
❌ Partition on string that looks like date ("2024-01-15") but is VARCHAR → no pruning
❌ Partition on computed column (YEAR(event_date)) in Spark SQL → doesn't prune

✅ Always use the actual column that appears in WHERE clauses
✅ Match partition granularity to query granularity
✅ Test with EXPLAIN to verify partition pruning is happening
```

**Interview tip:** "At AbbVie, I partitioned the 20 TB patient events table by region and load_date. Region gave me geographic pruning (US queries don't touch EU data), and load_date gave me temporal pruning (today's report only reads today's partition). I validated pruning was working by checking EXPLAIN output and Spark UI file scan counts."

### S3. Design a Model to Store 20+ TB Iceberg Data Efficiently

**One-line answer:**
> Use Iceberg's layered architecture with smart partitioning, sort ordering, regular compaction, and metadata management to ensure fast queries, efficient writes, and manageable metadata on 20+ TB tables.

---

**Full Architecture Design:**

#### 🔹 Table Design Principles at 20 TB Scale

```
20 TB Table Challenges:
1. Too many small files → slow metadata operations
2. Poor partition strategy → full scans even with filters
3. No sort order → file skipping doesn't work within partitions
4. Metadata bloat → snapshot management required
5. Concurrent writes → needs proper ACID handling

Solutions:
1. Target ~512 MB Parquet files (balance between parallelism + metadata)
2. Two-level partitioning (region + time-bucket)
3. Sort order on high-cardinality query columns
4. Regular OPTIMIZE + EXPIRE SNAPSHOTS
5. Iceberg's optimistic concurrency control
```

#### 🔹 Physical Storage Design

```python
# Table creation with full optimization settings
spark.sql("""
    CREATE TABLE IF NOT EXISTS iceberg.silver.patient_events_large (
        patient_id         STRING,
        territory_id       STRING,
        region             STRING,
        event_type         STRING,
        event_ts           TIMESTAMP,
        event_date         DATE,
        diagnosis_code     STRING,
        product_id         STRING,
        event_value        DECIMAL(18,2),
        record_hash        STRING,
        _ingest_ts         TIMESTAMP
    )
    USING iceberg
    PARTITIONED BY (
        region,                    -- 5 values: coarse geographic split
        months(event_ts)           -- ~12/year: manageable time partitions
    )
    TBLPROPERTIES (
        -- File format & size
        'write.format.default'                  = 'parquet',
        'write.parquet.compression-codec'       = 'snappy',
        'write.target-file-size-bytes'          = '536870912',   -- 512 MB target

        -- Sort order (Z-Order equivalent for Iceberg)
        'write.sort-order'                      = 'patient_id ASC NULLS LAST,
                                                  territory_id ASC NULLS LAST',

        -- Metadata management
        'write.metadata.compression-codec'      = 'gzip',
        'write.metadata.metrics.max-inferred-column-defaults' = '32',

        -- Snapshot retention
        'history.expire.max-snapshot-age-ms'    = '604800000',  -- 7 days
        'history.expire.min-snapshots-to-keep'  = '10'
    )
""")
```

#### 🔹 Write Strategy for 20 TB

```python
from pyspark.sql.functions import col, current_timestamp

# Strategy 1: Partition-aligned writes (minimize small files)
# Always write data sorted by partition columns
df.sortWithinPartitions("region", "event_date", "patient_id") \
    .writeTo("iceberg.silver.patient_events_large") \
    .option("write.distribution-mode", "hash") \  # Hash distribute on partition cols
    .append()

# Strategy 2: Control output file count
# If input has 1000 partitions but you want 100 output files per partition:
df.repartition(100, col("region"), col("event_date")) \
    .writeTo("iceberg.silver.patient_events_large") \
    .append()

# Strategy 3: Fan-out writes for large datasets
# Split large loads into manageable batches by partition
for region in ["US", "EU", "APAC", "LATAM", "NA"]:
    df_region = source_df.filter(col("region") == region)
    df_region.writeTo("iceberg.silver.patient_events_large") \
        .option("write.merge.mode", "append") \
        .append()
    print(f"✅ Loaded region: {region}")
```

#### 🔹 Maintenance Strategy (Critical at 20 TB)

```python
# 1. Regular compaction (run nightly)
spark.sql("""
    CALL iceberg.system.rewrite_data_files(
        table => 'iceberg.silver.patient_events_large',
        strategy => 'sort',
        sort_order => 'patient_id ASC, territory_id ASC',
        options => map(
            'target-file-size-bytes', '536870912',    -- 512 MB
            'min-input-files', '5',                   -- Only compact if 5+ small files
            'rewrite-all', 'false'
        )
    )
""")

# 2. Expire old snapshots (run weekly — keeps 7 days of time travel)
spark.sql("""
    CALL iceberg.system.expire_snapshots(
        table => 'iceberg.silver.patient_events_large',
        older_than => TIMESTAMP '2024-01-01 00:00:00',
        retain_last => 10
    )
""")

# 3. Remove orphan files (run monthly)
spark.sql("""
    CALL iceberg.system.remove_orphan_files(
        table => 'iceberg.silver.patient_events_large',
        older_than => TIMESTAMP '2024-01-01 00:00:00'
    )
""")

# 4. Rewrite manifests (keeps metadata lean)
spark.sql("""
    CALL iceberg.system.rewrite_manifests('iceberg.silver.patient_events_large')
""")
```

#### 🔹 Query Optimization at 20 TB

```python
# Always provide partition filters — never do full scans
df = spark.read.format("iceberg") \
    .load("iceberg.silver.patient_events_large") \
    .filter(col("region") == "US") \              # ← Partition pruning (20 TB → 4 TB)
    .filter(col("event_ts") >= "2024-01-01") \    # ← Month partition pruning (4 TB → 200 GB)
    .filter(col("territory_id").isin(["T001","T002"]))  # ← File skipping via sort order

# Check what's being scanned
df.explain()  # Look for: FileScan with PartitionFilters and DataFilters
```

Key decisions: region + month(event_ts) partitioning, patient_id + territory_id sort order, 512 MB target file size, and nightly rewrite_data_files to maintain compaction. These choices reduced typical query scan from 20 TB to 200 GB."

### S4. What Happens if Your Partition Column Has High Cardinality?

**One-line answer:**
> High-cardinality partition columns (like patient_id or order_id) create **millions of tiny partitions** — causing metadata explosion, too many small files, slow query planning, and degraded write performance.

---

**Detailed Explanation:**

#### 🔹 The Problem Visualized

```
Table: 100 GB patient data
Partition by: patient_id (50 million unique patients)

Result:
50,000,000 partition folders in S3
Each folder: 1–2 KB of data (100GB / 50M = 2 KB per "partition")

S3 structure:
s3://abbvie/silver/patients/
├── patient_id=P000001/  ← 2 KB file
├── patient_id=P000002/  ← 2 KB file
├── patient_id=P000003/  ← 2 KB file
... (50 million folders)
```

#### 🔹 Cascading Problems

| Problem | Detail | Impact |
|---|---|---|
| **Metadata explosion** | 50M entries in Hive Metastore/Iceberg catalog | Driver OOM, slow query planning |
| **Small files** | 2 KB per partition vs ideal 512 MB | 250,000x more file open operations |
| **Slow writes** | Each write must update 50M partition entries | Minutes to write what should take seconds |
| **S3 API costs** | Millions of LIST/GET operations per query | Significant cloud cost increase |
| **Spark planning** | Driver must process 50M partition specs | Job hangs at planning stage |
| **Vacuum/Optimize** | Operates on 50M folders | Maintenance jobs take hours |

#### 🔹 Solutions

**Solution 1: Don't partition on high-cardinality columns — use Z-Order instead**

```python
# ❌ BAD — Partitioning on patient_id (50M unique values)
df.writeTo("iceberg.silver.patients") \
    .partitionedBy("patient_id") \   # Creates 50M tiny partitions!
    .append()

# ✅ GOOD — Partition on low-cardinality, Z-Order on high-cardinality
df.writeTo("iceberg.silver.patients") \
    .partitionedBy("region") \       # Only 5 partitions
    .append()

# Then apply sort order for file skipping within partitions
spark.sql("""
    CALL iceberg.system.rewrite_data_files(
        table => 'iceberg.silver.patients',
        sort_order => 'patient_id ASC'   -- File-level skipping, not partition-level
    )
""")
```

**Solution 2: Bucketing (for join optimization)**

```python
# For high-cardinality join columns, use bucketing instead of partitioning
# Hashes patient_id into 100 buckets → 100 files, not 50M
spark.sql("""
    CREATE TABLE silver.patients_bucketed
    USING iceberg
    PARTITIONED BY (bucket(100, patient_id))
""")
# Query joining on patient_id → co-located data, no shuffle needed
```

**Solution 3: Composite Partition to Reduce Cardinality**

```python
# Instead of partitioning by order_id (10M unique values)
# Extract the date from order_id or use a different column entirely

# ❌ Partitioning on raw ID
.partitionedBy("order_id")           # 10M partitions

# ✅ Partition on time component
.partitionedBy("order_date")         # 365 partitions per year
# + Z-Order on order_id within each date partition
```

**Solution 4: If Already Deployed — Remediate**

```python
# Step 1: Create new properly-partitioned table
spark.sql("""
    CREATE TABLE iceberg.silver.patients_v2
    USING iceberg
    PARTITIONED BY (region, months(load_ts))
    TBLPROPERTIES ('write.sort-order' = 'patient_id ASC')
""")

# Step 2: CTAS (Create Table As Select) to migrate data
spark.sql("""
    INSERT INTO iceberg.silver.patients_v2
    SELECT * FROM iceberg.silver.patients   -- Old badly-partitioned table
""")

# Step 3: Validate row counts match
old_count = spark.table("iceberg.silver.patients").count()
new_count = spark.table("iceberg.silver.patients_v2").count()
assert old_count == new_count, f"Row count mismatch: {old_count} vs {new_count}"

# Step 4: Swap table names (zero-downtime approach)
spark.sql("ALTER TABLE iceberg.silver.patients RENAME TO iceberg.silver.patients_old")
spark.sql("ALTER TABLE iceberg.silver.patients_v2 RENAME TO iceberg.silver.patients")
```

The Spark driver was taking 5 minutes just to plan the query. I migrated it to region + load_date partitioning with patient_id Z-Ordering, and query planning dropped to under 5 seconds. That's the real-world cost of high-cardinality partitioning."

## 📊 Quick Reference Cheat Sheet

### Data Modeling Decision Guide

```
Choose your schema type:
├── Building for ANALYTICS (BI/Reporting)? → Star Schema / Dimensional Modeling
├── Building for OPERATIONS (app backend)? → Normalized (3NF)
└── Building for both?                    → Silver=normalized, Gold=denormalized

Choose your partitioning:
├── Cardinality < 5?       → Don't partition (use filter or Z-Order)
├── Cardinality 5–500?     → ✅ Good partition candidate
├── Cardinality > 500?     → Z-Order or Bucketing (not partitioning)
└── Time-series data?      → Always partition by date/month

Choose your storage layer:
├── Raw source data?       → Bronze (schema-on-read, append-only)
├── Cleaned, trusted?      → Silver (schema-enforced, normalized, DQ-checked)
└── Analytics-ready?       → Gold (denormalized, aggregated, BI-optimized)
```

### Architecture Comparison at a Glance

| Concept | What it Is | When to Use |
|---|---|---|
| **Data Lake** | Raw storage, any format | Landing zone for all raw data |
| **Data Warehouse** | Structured, SQL-optimized | Pure BI/analytics, fixed schema |
| **Lakehouse** | ACID + open format on object store | Modern analytics (your AbbVie stack) |
| **Star Schema** | Fact + flat dimensions | Gold layer for BI/Tableau |
| **Snowflake Schema** | Fact + normalized dimensions | Storage-sensitive data marts |
| **OLTP** | Row-level ops, real-time | Source systems (Veeva, Oracle) |
| **OLAP** | Column aggregations, analytics | Your Iceberg/Silver/Gold layers |
| **Surrogate Key** | System-generated integer PK | All dimension tables |
| **Partitioning** | Folder-level data split | Low-cardinality filter columns |
| **Z-Ordering** | File-level data co-location | High-cardinality filter columns |
