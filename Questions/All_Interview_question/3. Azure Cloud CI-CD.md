### Q1. What are ADF Pipelines? Triggers? Linked Services?

---

📌 **Opening line:**
> *"ADF is Azure's managed orchestration and data movement service. Pipelines define the workflow — what to do. Triggers define when to run it. Linked Services are the connection configurations — how to connect to external systems."*

---

🗣️ **Script to say:**

*"So Azure Data Factory has three core building blocks — and they each answer a different question.*

*A **Pipeline** answers 'what work needs to happen.' It's a logical container of activities — like Copy Data, Run a Databricks Notebook, Lookup a value, call a Web API, loop with ForEach, or branch with If Condition. You chain these activities together to build a data workflow. For example — at AbbVie, one pipeline would look up the last watermark, call the Veeva REST API, copy the response to ADLS Bronze, then trigger a Databricks notebook to transform it to Silver.*

*A **Trigger** answers 'when does this pipeline run.' There are four main types. A Schedule Trigger is the simplest — run every day at 2 AM. A Tumbling Window Trigger is more powerful — it runs on hourly or daily windows and tracks each window independently, so if the 2 AM window fails, it retries just that window without skipping it or conflating it with the 3 AM run. That's critical for incremental loads. A Storage Event Trigger fires when a file lands in ADLS — event-driven ingestion. And a Custom Event Trigger uses Azure Event Grid for application events.*

*A **Linked Service** answers 'how do I connect to this system.' It's essentially a connection string stored in ADF — you define it once and reuse it across multiple datasets and pipelines. Linked Services for Oracle, ADLS Gen2, Databricks, Key Vault, Salesforce — all defined once. And critically, the actual secrets — passwords, API keys — are never stored in the Linked Service itself. They're fetched from Azure Key Vault at runtime.*"

---

✅ **Closing hook:**
> *"At AbbVie, I used Tumbling Window Triggers for hourly Veeva API calls — each window was independently tracked so missed windows auto-retried. Storage Event Triggers for Oracle daily exports landing in ADLS. And I had one master Linked Service per system — Oracle, ADLS, Databricks — reused across all 12+ pipelines."*

---

**Quick visual to draw if asked:**

```
ADF Building Blocks:
─────────────────────────────────────────────────────────────
TRIGGER (When?)         PIPELINE (What?)       LINKED SERVICE (How?)
─────────────────       ────────────────────   ─────────────────────
Schedule: daily 2AM  →  Activity 1: Lookup  →  LS_Oracle (JDBC URL + KV secret)
Tumbling: hourly     →  Activity 2: Copy    →  LS_ADLS   (ABFS URL + KV key)
File arrival         →  Activity 3: Notebook→  LS_Databricks (workspace + token)
Event Grid           →  Activity 4: If/Else →  LS_KeyVault   (vault URL + MSI)
```

---

**Key trigger comparison — interviewers ask this:**

| Trigger | Use Case | Backfill? | Best For |
|---|---|---|---|
| Schedule | Run at fixed time | ❌ No | Daily batch, weekly reports |
| Tumbling Window | Hourly incremental | ✅ Yes | Incremental loads, SLA tracking |
| Storage Event | React to file arrival | ❌ No | Event-driven, file-based ETL |
| Custom Event | Application events via Event Grid | ❌ No | Microservices integration |

### Q2. Explain Delta Live Tables (DLT)

---

📌 **Opening line:**
> *"Delta Live Tables is a Databricks-managed ETL framework where you declare what your data should look like — using Python or SQL — and Databricks automatically handles the orchestration, dependencies, retries, data quality enforcement, and monitoring."*

---

🗣️ **Script to say:**

*"The core idea is shifting from imperative to declarative. In a traditional pipeline, I'm writing code that says 'do this, then do that, if it fails do this, then check for errors.' With DLT, I declare 'this table should contain cleansed patient data from this source, and it must meet these quality rules.' Databricks figures out the rest — how to orchestrate it, what order to run it in, when to reprocess.*

*There are two types of tables in DLT. A **Streaming Live Table** reads from a continuously arriving source — like Auto Loader reading new JSON files dropped in ADLS. It processes only new data incrementally. A **Live Table** reads from another DLT table and applies batch transformations.*

*The most powerful feature for data engineering is **Expectations** — built-in data quality rules. There are three modes: `expect` just warns you and tracks violations in metrics. `expect_or_drop` silently removes invalid rows so they don't pollute downstream. `expect_or_fail` halts the entire pipeline on any violation — useful for critical compliance checks.*

*DLT also auto-discovers the dependency graph from your `dlt.read()` calls and builds the pipeline DAG automatically. No manual dependency management.*

*Now — an honest comparison: DLT is excellent for teams that want managed pipelines with built-in DQ. But for our AbbVie work, we used standard Databricks Jobs with custom notebooks because we had Iceberg-specific operations — DLT had stronger Delta Lake bias at the time and didn't support all Iceberg APIs natively."*

---

✅ **Closing hook:**
> *"DLT's expectations feature is closest to what I built manually for AbbVie — null checks on mandatory fields, referential integrity checks, quarantining bad records to a DQ table. DLT packages that pattern natively. If we were starting fresh today I'd seriously consider it for the Silver layer."*

---

```python
import dlt
from pyspark.sql.functions import col, current_timestamp, upper, trim

# STREAMING LIVE TABLE — Bronze: raw intake from ADLS
@dlt.table(
    name="bronze_veeva_patients",
    comment="Raw Veeva API data — append only, no transforms",
    table_properties={"quality": "bronze"}
)
def bronze_veeva_patients():
    return (
        spark.readStream
            .format("cloudFiles")                          # Auto Loader
            .option("cloudFiles.format", "json")
            .option("cloudFiles.schemaLocation", "/mnt/schemas/veeva/")
            .load("abfss://raw@abbvie.dfs.core.windows.net/veeva/")
            .withColumn("_ingest_ts", current_timestamp())
    )

# LIVE TABLE — Silver: cleanse + quality rules
@dlt.table(name="silver_patients", comment="Cleansed, validated patient master")
@dlt.expect("valid_patient_id",  "patient_id IS NOT NULL")       # warn only
@dlt.expect_or_drop("valid_region", "region IN ('US','EU','APAC','LATAM')")  # drop row
@dlt.expect_or_fail("known_source", "source_system IN ('VEEVA','ORACLE')")   # halt pipeline
def silver_patients():
    return (
        dlt.read("bronze_veeva_patients")
            .withColumn("status", upper(trim(col("status"))))
            .withColumn("region", upper(trim(col("region"))))
            .filter(col("status") != "DELETED")
    )

# Auto-built DAG: bronze_veeva_patients → silver_patients ✅
```

### Q3. What is Data Lake Gen2 Hierarchy?

---

📌 **Opening line:**
> *"ADLS Gen2 has a 3-level hierarchy — Storage Account, then Containers as root filesystems, then directories and files inside them. The key difference from regular Blob Storage is the Hierarchical Namespace — it makes folder operations atomic and efficient, which is critical for Spark and Delta/Iceberg."*

---

ADLS Gen2 hierarchy refers to the Hierarchical Namespace (HNS) feature, which organizes data into directories and subdirectories like a traditional file system instead of flat blob storage. It enables efficient folder-level operations such as rename, move, and delete, and improves performance for big data workloads in Spark, Databricks, and Hadoop.

In regular Blob Storage, the structure is flat and folders are only virtual, whereas ADLS Gen2 provides a true hierarchical directory structure with optimized file-system operations and better analytics performance.

---

🗣️ **Script to say:**

*"Let me build this top to bottom. At the top is the **Storage Account** — this is the globally unique Azure resource, like `abbviedatalake`. Everything lives inside it.*

*Inside the storage account, you have **Containers** — also called Filesystems when Hierarchical Namespace is enabled. A Container is like a root drive. A common pattern is one container per data layer — `raw`, `bronze`, `silver`, `gold`, `dq` — or per environment — `dev`, `prod`. This gives you separate access control and lifecycle policies per layer.*

*Inside containers you have **Directories** — real folders, not simulated ones. And inside directories, files — Parquet, JSON, Avro, whatever.*

*Now — the critical thing about Gen2 vs regular Blob Storage is the **Hierarchical Namespace**. In regular Blob Storage, folders are just naming conventions — there's no real folder structure. So renaming a folder means copying every single file, then deleting originals — for a million files that takes hours and isn't atomic. In Gen2, folders are first-class objects. Rename is a single metadata operation — milliseconds, and fully atomic. This is what makes Delta Lake and Iceberg work correctly — they rely on atomic rename for their transaction commits.*

*Access works via the ABFS driver — `abfss://containername@accountname.dfs.core.windows.net/path`. The double-S in abfss means SSL encrypted. In Databricks, you can mount containers to `/mnt/silver` so notebooks use simple file paths.*

*Security has two layers — Azure RBAC for coarse-grained access at the container level, and POSIX ACLs for fine-grained per-folder and per-file permissions. Both can be scoped to users, groups, or service principals."*

---

✅ **Closing hook:**
> *"At AbbVie, we had separate containers for Bronze, Silver, Gold, and DQ in ADLS Gen2. Our Databricks service principal had write access only to its own layer — so the ingestion service principal could write Bronze but not Silver. POSIX ACLs on the patient data directories ensured analysts only had read access to Gold, never to raw Bronze with unmasked data."*

### Q4. Difference Between ADF vs Databricks Workflows

---

📌 **Opening line:**
> *"ADF is Azure's ETL orchestration platform — best for data movement between heterogeneous systems using 200+ native connectors. Databricks Workflows is compute orchestration — best for scheduling Spark notebooks and Python jobs within the Databricks ecosystem. In practice, you use both together."*

---

🗣️ **Script to say:**

*"Think of it this way — ADF excels at the boundaries. Connecting to Oracle, pulling from Salesforce, reacting to a file arriving in ADLS, calling a REST API, moving data from on-premises to cloud using a Self-Hosted Integration Runtime. These are things where ADF has native connectors, retry policies, monitoring, and no-code designers.*

*Databricks Workflows excels inside the Databricks ecosystem. Once the data is in cloud storage, all the heavy Spark transformations — Bronze to Silver to Gold, data quality checks, ML feature pipelines, dbt runs — these are better orchestrated by Databricks Workflows because it manages cluster lifecycle, supports multi-task DAGs where each task can be a notebook or Python script, has git integration, and handles cluster reuse with warm pools.*

*The key difference in terms of what they're good at: ADF has 200-plus connectors to external systems. Databricks Workflows doesn't — to read from Oracle in a Databricks workflow, you're writing JDBC code manually. On the flip side, ADF can trigger Databricks notebooks but can't introspect what happens inside them. Databricks Workflows gives you full observability — task-level logs, retry per task, dependency graphs.*

*The most common enterprise pattern combines both — ADF handles ingestion and triggering, Databricks Workflows handles transformation. ADF Databricks Notebook Activity calls a Databricks job, and inside that job, Databricks Workflows manages the multi-step DAG of notebooks."*

---

✅ **Closing hook:**
> *"At AbbVie this was exactly our pattern. ADF owned the outer orchestration — it had the Oracle JDBC connector, the Veeva REST API linked service, file-arrival triggers. ADF's Databricks Notebook Activity called into Databricks. Inside Databricks, a multi-task workflow ran Bronze ingest → Silver transform → Gold aggregate → DQ validation as a DAG with proper dependencies."*

---

### Q5. Explain Key Vault Usage

---

📌 **Opening line:**
> *"Azure Key Vault is a centralized secrets management service. Instead of hardcoding passwords, connection strings, or API keys anywhere in your code, notebooks, or pipeline configs — you store them once in Key Vault and retrieve them at runtime. Secrets never appear in code, logs, Git history, or deployment artifacts."*

---

🗣️ **Script to say:**

*"The core problem Key Vault solves is credential sprawl. Without it, a team of 10 engineers over 2 years ends up with Oracle passwords in 15 notebooks, in ADF linked service JSON exports, in environment variable files, maybe even in a Confluence page. When you need to rotate the password, you hunt down every place it appears. And if any of those sources leak — Git history, screenshot, log file — you have a security incident.*

*Key Vault centralizes all of this. You store the Oracle password in Key Vault once. Then in Databricks, you create a Secret Scope backed by Key Vault — and in your notebooks you call `dbutils.secrets.get(scope, key)`. The value is fetched at runtime, and importantly, Databricks redacts it in all logs and notebook output — you literally cannot print the value accidentally.*

*In ADF, Linked Services don't store passwords directly. They reference a Key Vault secret by name — ADF fetches it at pipeline execution time using its Managed Identity. No human ever needs to know the password.*

*Access control is through Azure RBAC — the Databricks cluster's Managed Identity gets Get permission on secrets. A CI/CD service principal gets Get and Set for deployments. Developers only get List — they can see secret names but never values. And every single access is logged in Key Vault's audit log — which gives you a complete trail for HIPAA or SOC2 compliance.*

*The rotation story is where this really shines — when Oracle forces a password rotation, you update it in Key Vault once, and every pipeline, notebook, and ADF linked service that references that secret automatically picks up the new value on the next run. Zero code changes, zero downtime.*"

---

✅ **Closing hook:**
> *"At AbbVie, Key Vault was mandatory by enterprise security policy. Every credential — Oracle passwords, Veeva API keys, Smartsheet tokens, ADLS account keys — lived in Key Vault. Our Databricks clusters had KV-backed secret scopes. In three years, not a single credential appeared in any notebook or Git commit. The KV audit logs also satisfied our HIPAA compliance requirement for access audit trails."*

---

```python
# ── In Databricks Notebooks ──────────────────────────────────────────

### Q6. How Does ADF Handle Incremental Loads?

---

📌 **Opening line:**
> *"ADF handles incremental loads through three main patterns — watermark-based filtering using a control table, Tumbling Window Triggers that track each time window independently, and file-based incremental loading for file-arrival scenarios. The key is that only new or changed data is processed on each run."*

---

🗣️ **Script to say:**

*"The most common pattern is the **watermark approach**. You maintain a control table — a small metadata table in Azure SQL or even a JSON file in ADLS — that stores the last successfully processed timestamp per pipeline. At the start of each run, ADF does a Lookup Activity to read that watermark. The Copy Activity then queries the source with `WHERE updated_at > last_watermark`. After successful copy, a Script Activity updates the watermark to the current time. If the copy fails, the watermark is NOT updated — so the next retry reprocesses the same window.*

*The second pattern is **Tumbling Window Triggers**, which is more robust. Instead of you manually tracking watermarks, the trigger itself owns the window. Each hour or day is a discrete window — if the 3 AM window fails, the trigger retries just that window. And if you need to backfill — say you want to reprocess last week's data — you can retroactively trigger specific past windows. This is why Tumbling Window is better than Schedule Trigger for incremental loads.*

*The third pattern is **file-based incremental loading**. A Storage Event Trigger fires when a new file arrives in ADLS. The pipeline processes only that file. This is common for Oracle exports that land as daily CSV files — each file arrival triggers ingestion of just that file.*

*In Databricks — after ADF copies new data to Bronze, the PySpark transformation uses MERGE INTO on the Silver table. New records get inserted, changed records get updated. That's the complete incremental pipeline — ADF handles the extraction and scheduling, Databricks handles the transformation upsert.*"

---

✅ **Closing hook:**
> *"At AbbVie, Veeva had a modified_since API parameter — perfect for watermark-based incremental. I stored the watermark in an ADLS control file, fetched it with Lookup Activity, called the API with that timestamp, wrote Bronze, then updated the watermark. Tumbling Window Triggers handled the scheduling. Oracle extracts used the file-arrival pattern — daily CSV dropped in ADLS triggered the pipeline."*

---

```
Incremental Load Patterns:

Pattern 1: Watermark (timestamp-based)
─────────────────────────────────────
ADF Lookup Activity
  → reads: control_table WHERE pipeline = 'veeva_patients'
  → gets: last_run_ts = "2024-03-14 02:00:00"
        ↓
ADF Copy Activity
  → source query: WHERE updated_at > '2024-03-14 02:00:00'
  → only NEW/CHANGED rows fetched
        ↓
ADF Script Activity
  → UPDATE control_table SET last_run_ts = pipeline run start time
  → (only runs if copy SUCCEEDED — ensures no gap on retry)


Pattern 2: Tumbling Window Trigger
────────────────────────────────────────
Trigger fires → @trigger().outputs.windowStartTime = "2024-03-14T02:00:00Z"
                @trigger().outputs.windowEndTime   = "2024-03-14T03:00:00Z"
Copy source query: WHERE event_ts BETWEEN windowStart AND windowEnd
→ Each window tracked independently
→ Missed window auto-retried
→ Backfill: manually trigger past windows


Pattern 3: File Arrival (Storage Event)
────────────────────────────────────────
Oracle extract lands: /raw/oracle/patient_master_2024-03-14.csv
      ↓
Storage Event Trigger fires
      ↓
ADF gets: @trigger().outputs.body.fileName = "patient_master_2024-03-14.csv"
      ↓
Copy Activity processes ONLY this file
```

### Q7. What is CI/CD? What Tools Have You Used?

---

📌 **Opening line:**
> *"CI/CD — Continuous Integration and Continuous Deployment — is an automated process that takes code from a developer's commit all the way to production without manual steps. CI runs tests and checks on every change. CD deploys passing changes to environments automatically, with approvals at critical gates."*

---

🗣️ **Script to say:**

*"Let me break down what each part actually means in practice, not just the textbook definition.*

***Continuous Integration** means every time a developer pushes code or opens a pull request, an automated pipeline immediately runs — linting the code, running unit tests, checking for security issues. The goal is catching bugs as early as possible, before they ever reach a shared environment. For a data engineering team, CI also runs PySpark unit tests using a local SparkSession — testing transformation logic, data quality functions, null handling — without needing a full cluster.*

***Continuous Deployment** means once CI passes and the code merges to main, the deployment to the next environment happens automatically — no one manually copying notebooks or updating job configs. For Databricks, that means notebooks are synced to the workspace, job configurations are updated via the Databricks CLI or Asset Bundles, and the job is ready to run with the new code.*

*The tools I've used: **Git and GitHub** for version control — feature branches, pull requests, code review. **GitHub Actions** for the CI/CD pipelines — YAML-based workflows that run on GitHub-hosted runners. For Databricks deployment specifically — the **Databricks CLI** to push notebooks and update job configs, and more recently **Databricks Asset Bundles** which lets you define your entire Databricks workspace — jobs, clusters, notebooks — as code in a `databricks.yml` file.*

*Environment promotion is key — code goes DEV first, automatically. Then UAT, with manual approval from a lead. Then PROD, with a second approval gate. GitHub Environments handle this — you assign required reviewers to the production environment, and the workflow pauses and waits for that approval before deploying."*

---

✅ **Closing hook:**
> *"At AbbVie, we used GitHub and GitHub Actions. Every PR triggered unit tests — pytest with a local SparkSession for PySpark logic, testing null handling, schema validation, deduplication logic. Merges to main auto-deployed to DEV Databricks workspace. PROD deployment required lead approval. This eliminated environment drift — DEV, UAT, and PROD always had exactly the same notebook version."*

---

### Scenario 1. Create a Pipeline to Ingest Salesforce API Data Every Hour

**Full Architecture:**

```
Salesforce REST API
       │  (OAuth2 client credentials)
       ▼
ADF Tumbling Window Trigger (every 1 hour)
       │
       ▼
ADF Pipeline: SF_Hourly_Ingestion
  ├─ Activity 1: Web Activity         → GET OAuth token from Salesforce
  ├─ Activity 2: Copy Activity        → Salesforce contacts → ADLS Bronze JSON
  ├─ Activity 3: Databricks Notebook  → Bronze → Silver (cleanse + upsert)
  └─ Activity 4: If Condition         → success/failure notification
       │
       ▼
ADLS Gen2 Bronze  (raw JSON, partitioned by hour)
       │
       ▼
Databricks PySpark (MERGE INTO Iceberg Silver)
       │
       ▼
Iceberg Silver: salesforce_contacts (upserted, DQ-checked)
```

# Parameters passed from ADF
window_start = dbutils.widgets.get("window_start")   # "2024-03-14T02:00:00Z"
window_end   = dbutils.widgets.get("window_end")     # "2024-03-14T03:00:00Z"

from pyspark.sql.functions import (
    col, current_timestamp, lit, explode,
    upper, trim, sha2, concat_ws
)

# ── Read raw Bronze JSON ────────────────────────────────────────────────
bronze_path = f"abfss://bronze@abbviedatalake.dfs.core.windows.net/salesforce/"
df_raw = spark.read.option("multiLine", True).json(
    f"{bronze_path}window={window_start[:13]}/"
)

# ── Flatten nested Salesforce structure ────────────────────────────────
df_flat = df_raw \
    .withColumn("record", explode(col("records"))) \
    .select(
        col("record.Id").alias("sf_contact_id"),
        col("record.AccountId").alias("sf_account_id"),
        col("record.Territory__c").alias("territory_id"),
        col("record.Status__c").alias("status"),
        col("record.LastModifiedDate").alias("sf_last_modified"),
        current_timestamp().alias("_ingest_ts"),
        lit(window_start).alias("_window_start"),
        lit("SALESFORCE").alias("_source_system")
    )

# ── DQ: quarantine bad records ─────────────────────────────────────────
df_bad   = df_flat.filter(col("sf_contact_id").isNull() | col("territory_id").isNull())
df_valid = df_flat.filter(col("sf_contact_id").isNotNull() & col("territory_id").isNotNull())

if df_bad.count() > 0:
    df_bad.writeTo("iceberg.dq.failed_records").append()

# ── Standardize + hash ─────────────────────────────────────────────────
df_silver = df_valid \
    .withColumn("status", upper(trim(col("status")))) \
    .withColumn("record_hash",
        sha2(concat_ws("||", col("territory_id"), col("status")), 256)
    )

# ── MERGE INTO Silver (upsert) ─────────────────────────────────────────
df_silver.createOrReplaceTempView("sf_incoming")

spark.sql("""
    MERGE INTO iceberg.silver.salesforce_contacts AS tgt
    USING sf_incoming AS src
    ON tgt.sf_contact_id = src.sf_contact_id
    WHEN MATCHED AND tgt.record_hash != src.record_hash
        THEN UPDATE SET *
    WHEN NOT MATCHED
        THEN INSERT *
""")

print(f"✅ SF ingestion | Window: {window_start} | Valid: {df_valid.count()} | Failed DQ: {df_bad.count()}")
dbutils.notebook.exit("SUCCESS")
```

### Scenario 2. ADF Pipeline is Failing — How Do You Debug?

**Structured debugging approach — walk the interviewer through this in order:**

---

**Step 1 — ADF Monitor: Find the failing activity**
```
ADF Portal → Monitor → Pipeline Runs → Click the failed run
  ✅ Find: Which activity failed?
  ✅ Find: What is the error message?
  ✅ Find: What time did it fail?
  ✅ Find: Input/Output of the failed activity
```

---

**Step 2 — Read error messages and what they mean**

```
ERROR                                              ROOT CAUSE + FIX
────────────────────────────────────────────────────────────────────────────────
"Cannot connect to Oracle server"                  → Oracle down / SHIR offline
                                                   → Check: Integration Runtimes
                                                   → Check: SHIR node status, firewall

"Invalid connection string"                        → KV secret rotated, ADF cached old
                                                   → Fix: update KV secret, restart LS

"Databricks job run failed (exit code: 1)"         → Notebook threw an exception
                                                   → Fix: Databricks → Job Runs → notebook logs

"Activity timeout"                                 → Source slow / data volume increased
                                                   → Fix: increase activity timeout setting

"Schema mismatch at sink"                          → Source added/removed a column
                                                   → Fix: update type mappings in Copy activity

"Quota exceeded" / "Rate limit hit"                → Too many API calls (Salesforce, Veeva)
                                                   → Fix: add retry with exponential backoff

"Access denied on container"                       → Service principal lost ADLS permission
                                                   → Fix: re-assign Storage Blob Contributor role

"Self-Hosted IR offline"                           → SHIR VM stopped / network policy changed
                                                   → Fix: restart SHIR service on VM
```

---

**Step 3 — Test Linked Service connectivity**
```
ADF → Manage → Linked Services → Select failing LS → Test Connection
  ✅ Success → problem is in the pipeline logic / data
  ❌ Failure → problem is in connection / network / credentials
```

---

**Step 4 — Use Debug mode for deeper investigation**
```python
# ADF Canvas → Debug (not Trigger Now)
# → Runs interactively, shows each activity output live
# → Can modify parameters per run
# → Can re-run from a specific activity (without re-running the whole pipeline)

# Check activity Input tab:
{
  "source": { "query": "SELECT * FROM PATIENT WHERE UPDATED > '2024-03-14'" },
  "copyDuration": 0,
  "rowsRead": 0,        ← if 0, source returned nothing (wrong watermark?)
  "rowsWritten": 0      ← if rowsRead > 0 but rowsWritten = 0, sink issue
}
```

---

**Step 5 — Check Databricks side if notebook activity failed**
```python
# Go to Databricks workspace
# Jobs → Job Runs → Find the failed run ID (from ADF error output)
# Open the run → click the failed task → see notebook output

# Common Databricks errors from ADF-triggered runs:
# AnalysisException: Table not found
#   → Iceberg table doesn't exist in the catalog yet
#   → Fix: run table creation DDL first

# OOM (OutOfMemoryError)
#   → Data volume grew, cluster too small
#   → Fix: increase executor memory or optimize partition strategy

# Schema mismatch / cannot cast
#   → Source API changed a field's type
#   → Fix: add explicit cast in transformation

# Access denied on ADLS container
#   → Cluster MSI lost permission (can happen after RBAC policy refresh)
#   → Fix: re-assign Storage Blob Data Contributor role to cluster MSI
```

---

**Step 6 — Full debug checklist**
```
□  ADF Monitor → identify failing activity + error message
□  Check activity Input/Output tabs (rowsRead, rowsWritten, error detail)
□  ADF → Manage → Linked Services → Test Connection
□  ADF → Manage → Integration Runtimes → SHIR node status (if on-prem source)
□  Databricks → Job Runs → notebook logs (if Databricks activity failed)
□  ADLS → Check permissions (if access denied errors)
□  Source system → Did schema change? Is the DB up? Rate limit hit?
□  Key Vault → Was the secret rotated? Does ADF have Get permission?
□  Run ADF Debug mode with same parameters to reproduce interactively
```

**Step 1: Configure S3 credentials in Databricks (no hardcoding)**

```python
# Method 1: IAM Role via Instance Profile (preferred — no keys in code)

# The cluster assumes the IAM role — no access keys needed at all ✅

AWS_KEY    = dbutils.secrets.get(scope="abbvie-kv", key="aws-access-key")

```

---

**Step 2: Read from S3**

```python

SILVER    = "abfss://silver@abbviedatalake.dfs.core.windows.net"

# Read raw from S3 (CSV / Parquet / JSON)

    .option("header", True) \
    .option("inferSchema", True) \
    .csv(S3_PATH)

df_s3.printSchema()
```

---

**Step 3: Write to Bronze (preserve raw)**

```python
# Land raw on ADLS Bronze first — preserve source fidelity
df_s3 \
    .withColumn("_source", lit("S3")) \
    .withColumn("_source_path", lit(S3_PATH)) \
    .withColumn("_ingest_ts", current_timestamp()) \
    .write \
    .format("delta") \
    .mode("append") \
    .partitionBy("load_date") \
    .save(f"{BRONZE}/patient_master_s3/")
```

---

**Step 4: Transform and write to Delta Silver**

```python
df_bronze = spark.read.format("delta").load(f"{BRONZE}/patient_master_s3/")

df_silver = df_bronze \
    .filter(col("patient_id").isNotNull()) \
    .withColumn("status",  upper(trim(col("status")))) \
    .withColumn("region",  upper(trim(col("region")))) \
    .withColumn("record_hash",
        sha2(concat_ws("||", col("patient_id"), col("territory_id"), col("status")), 256)
    ) \
    .withColumn("_silver_ts", current_timestamp())

# Write to Delta on ADLS
df_silver.write \
    .format("delta") \
    .mode("overwrite") \
    .option("overwriteSchema", "true") \
    .partitionBy("region", "load_date") \
    .save(f"{SILVER}/patient_master/")

print(f"✅ Written to Delta Silver: {df_silver.count()} rows")
```

---

**Step 5: Validate the migration**

```python
from pyspark.sql.functions import sum as spark_sum, hash as spark_hash

# Count validation
s3_count    = spark.read.csv(S3_PATH, header=True).count()
delta_count = spark.read.format("delta").load(f"{SILVER}/patient_master/").count()
assert s3_count == delta_count, f"Count mismatch: S3={s3_count}, Delta={delta_count}"

# Checksum validation (catch silent data corruption)
def checksum(df):
    return df.select(spark_sum(spark_hash(col("patient_id"), col("status")))).collect()[0][0]

assert checksum(spark.read.csv(S3_PATH, header=True)) == \
       checksum(spark.read.format("delta").load(f"{SILVER}/patient_master/")), \
    "Checksum mismatch — data integrity issue!"

print("✅ S3 → Delta migration validated successfully")
```

## 📊 Quick Reference Cheat Sheet

### ADF Trigger Selection Guide
```
Fixed time (daily 2 AM)?                  → Schedule Trigger
Hourly with backfill + window tracking?   → Tumbling Window Trigger  ← most common for incremental
React to file landing in ADLS?            → Storage Event Trigger
Application event via Event Grid?         → Custom Event Trigger
```

### ADF vs Databricks Workflows — One Line Each
```
ADF:                 Orchestrate across systems, 200+ connectors, data movement
Databricks Workflows: Orchestrate inside Databricks, Spark-native, cluster management
Combined pattern:    ADF triggers → Databricks Notebook Activity → Workflows DAG
```

### Key Vault — What Goes Where
```
Passwords / API keys / tokens    → Key Vault Secrets
Encryption keys (AES, RSA)       → Key Vault Keys
TLS certificates                 → Key Vault Certificates
Databricks access                → dbutils.secrets.get()
ADF access                       → LinkedService references KV secret by name
CI/CD access                     → GitHub Secrets (for workspace tokens)
```

feature-branch → PR → CI (tests pass?) → merge to main
                                               ↓
                                         auto-deploy DEV
                                               ↓
                                         smoke test passes?
