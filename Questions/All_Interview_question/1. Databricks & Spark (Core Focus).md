### Q1. What is the architecture of Spark? Explain Driver, Executors, Cluster Manager.

**One-line answer:**
> Spark follows a master-worker architecture where the **Driver** plans the job, the **Cluster Manager** allocates resources, and **Executors** run the actual tasks.

---

**Detailed Explanation:**

#### 🔹 Driver
- The **brain** of a Spark application
- Runs your `main()` / notebook code
- Creates the `SparkSession` / `SparkContext`
- Converts your code into a **DAG (Directed Acyclic Graph)**
- Splits the DAG into **stages** and **tasks**
- Tracks the progress of each task
- **There is only ONE driver per Spark application**

#### 🔹 Executors
- Worker processes that run on **cluster nodes**
- Execute the tasks assigned by the driver
- Store data in **memory or disk** (for caching)
- Report results and status back to the driver
- Each Spark application gets its **own set of executors** (isolated)
- If an executor crashes, the driver reschedules those tasks

#### 🔹 Cluster Manager
- Responsible for **allocating resources** (CPU, RAM) to Spark apps
- Spark supports:
  - **YARN** — Hadoop ecosystem
  - **Kubernetes** — containerized environments
  - **Databricks-managed** — fully abstracted (what you used at AbbVie)
  - **Standalone** — Spark's built-in manager

#### 🔹 Flow Diagram
```
Your PySpark Code
       ↓
   SparkSession (Driver)
       ↓
   Cluster Manager  ← requests resources
       ↓
   Worker Nodes → Executors
       ↓
   Tasks run on partitions → Results returned to Driver
```

**Interview tip:** "In Databricks, the driver node is the cluster's master node. When I submitted Veeva API ingestion jobs at AbbVie, the driver parsed my transformation plan, Databricks Cluster Manager allocated executor nodes, and each executor processed a partition of the API data."

### Q2. Difference Between RDD vs DataFrame vs Dataset

**One-line answer:**
> RDD is the low-level, type-safe but slow API. DataFrame is the high-level, schema-aware, Catalyst-optimized API. Dataset combines both — type-safe + optimized, but only in Scala/Java.

---

**Detailed Comparison:**

| Feature | RDD | DataFrame | Dataset |
|---|---|---|---|
| **Introduced** | Spark 1.0 | Spark 1.3 | Spark 1.6 |
| **Type Safety** | ✅ Compile-time | ❌ Runtime only | ✅ Compile-time |
| **Optimization** | ❌ Manual only | ✅ Catalyst + Tungsten | ✅ Catalyst + Tungsten |
| **Schema** | ❌ No schema | ✅ Yes | ✅ Yes |
| **Language** | All | All | Scala / Java only |
| **Performance** | Slowest | Fast | Fast |
| **API Level** | Low-level | High-level | High-level |
| **Use Today?** | Rarely | ✅ Primary | Scala/Java only |

#### 🔹 RDD (Resilient Distributed Dataset)
```python
# RDD — you control everything manually
rdd = spark.sparkContext.parallelize([("Alice", 30), ("Bob", 25)])
rdd_filtered = rdd.filter(lambda x: x[1] > 28)
# No optimization — Spark just does what you say
```

#### 🔹 DataFrame
```python
# DataFrame — schema-aware, Catalyst-optimized
df = spark.read.csv("s3://abbvie-raw/patients.csv", header=True, inferSchema=True)
df_filtered = df.filter(col("age") > 28)
# Catalyst pushes this filter to file scan level — reads less data
```

#### 🔹 Why PySpark uses DataFrame (not Dataset)
- Python has no compile-time type system → Dataset not available in Python
- **DataFrame IS a Dataset[Row]** internally in Spark
- In Python jobs (like your AbbVie pipelines), always use **DataFrame**

**Interview tip:** "In all my PySpark work at AbbVie — CSV/XML ingestion from S3, Veeva API pipelines — I exclusively used DataFrames. The Catalyst optimizer gave me automatic predicate pushdown into Iceberg files without any extra code."

### Q3. Explain the Catalyst Optimizer

**One-line answer:**
> Catalyst is Spark's query optimization engine that automatically rewrites and optimizes your query plan before execution — similar to how a database query planner works.

---

**Detailed Explanation:**

Catalyst works in **4 phases**:

#### Phase 1 — Analysis (Resolve)
- Resolves column names, checks if tables exist
- Uses the **catalog** (Hive Metastore / Unity Catalog) to validate schema
- Outputs: **Analyzed Logical Plan**

#### Phase 2 — Logical Optimization
- Applies **rule-based** optimizations:
  - **Predicate Pushdown** — move filters as early as possible (closer to data source)
  - **Column Pruning** — drop unused columns before reading
  - **Constant Folding** — evaluate `1 + 1` → `2` at plan time
  - **Partition Pruning** — skip irrelevant partitions

#### Phase 3 — Physical Planning
- Converts logical plan to one or more **physical plans**
- Chooses the best using a **cost model**:
  - Should this join be a **BroadcastHashJoin** or **SortMergeJoin**?
  - Which partition order is best?

#### Phase 4 — Code Generation (Tungsten)
- Generates **optimized JVM bytecode** for the physical plan
- Eliminates virtual function calls → direct CPU execution

```python
# You write this:
df.filter(col("region") == "US").select("patient_id", "territory_id")

# Catalyst rewrites it to:
# → Push filter to Iceberg scan → only open "region=US" files
# → Column pruning → only read patient_id, territory_id columns
# → 200 files to scan becomes 3 files to scan
```

**Interview tip:** "When I optimized Spark pipelines at AbbVie and achieved 50% reduction in processing time, Catalyst's predicate pushdown into Iceberg metadata was one of the biggest contributors — it avoided full table scans."

### Q4. What is Tungsten? How Does It Help Performance?

**One-line answer:**
> Tungsten is Spark's CPU and memory optimization engine that bypasses the JVM's garbage collector, uses compact binary data formats, and generates fast bytecode — making Spark dramatically faster at the hardware level.

---

**Detailed Explanation:**

Tungsten solves the **JVM bottleneck** in Spark. Three main optimizations:

#### 🔹 1. Off-Heap Memory Management
- JVM has Garbage Collection (GC) which causes **pause-the-world** stops
- Tungsten stores data **outside the JVM heap** in raw binary format
- Result: No GC pauses, faster memory access
```
JVM Heap (with GC pauses)    → Tungsten Off-Heap (no GC) ✅
Object overhead: 16 bytes     → Binary format: 8 bytes per int ✅
```

#### 🔹 2. Cache-Aware Computation
- CPU caches (L1/L2) are 100x faster than RAM
- Tungsten stores data in **column-oriented binary format** that fits in CPU cache lines
- Processes data in batches aligned to cache → massive speedup

#### 🔹 3. Whole-Stage Code Generation
- Instead of: function call → function call → function call (slow)
- Tungsten generates a **single compiled Java function** for the entire stage
- Eliminates virtual function dispatch overhead

#### 🔹 4. Vectorized Execution (Arrow)
- When you enable Arrow-based PySpark:
  ```python
  spark.conf.set("spark.sql.execution.arrow.pyspark.enabled", "true")
  ```
- Data is transferred JVM ↔ Python as **Arrow columnar batches** instead of row-by-row
- This is critical for Pandas UDFs and PySpark DataFrames

**Interview tip:** "At AbbVie, enabling Arrow-based execution was one of my configuration tuning steps. It reduced JVM-to-Python data transfer overhead significantly when processing high-volume Veeva API payloads."

### Q5. Explain Wide vs Narrow Transformations

**One-line answer:**
> Narrow = each partition produces output from exactly one input partition (no data movement). Wide = output partitions need data from multiple input partitions (causes a shuffle across the network).

---

**Detailed Explanation:**

#### 🔹 Narrow Transformations
- Each input partition → **exactly one** output partition
- **No data movement** across the network
- Can be **pipelined** (executed together in one stage)
- Fast ✅

| Transformation | Description |
|---|---|
| `filter()` | Keep rows matching condition |
| `select()` | Choose columns |
| `withColumn()` | Add/modify column |
| `map()` | Transform each row |
| `union()` | Stack two DataFrames |
| `sample()` | Sample rows |

#### 🔹 Wide Transformations
- Output requires data from **multiple input partitions**
- Causes a **shuffle** — data moved across the network
- Creates a **stage boundary** in the DAG
- Expensive ❌

| Transformation | Description |
|---|---|
| `groupBy()` | Group rows by key |
| `join()` | Combine two DataFrames |
| `distinct()` | Remove duplicates |
| `repartition()` | Change partition count |
| `orderBy()` | Sort globally |
| `reduceByKey()` | Aggregate by key |

```python
# Narrow — 100% local, no shuffle
df_us = df.filter(col("region") == "US")          # Narrow
df_cols = df_us.select("patient_id", "territory") # Narrow

# Wide — triggers shuffle, stage boundary
df_count = df.groupBy("territory_id").count()     # WIDE ← shuffle
df_joined = patient_df.join(territory_df, "id")   # WIDE ← shuffle
```

#### 🔹 Why This Matters for Performance
```
Stage 1 (narrow): filter → select → withColumn  [all pipelined, no network]
    ↓ SHUFFLE BOUNDARY
Stage 2 (wide): groupBy → agg               [network transfer, disk write]
    ↓ SHUFFLE BOUNDARY  
Stage 3 (narrow): filter result → write     [pipelined again]
```

**Interview tip:** "When analyzing slow jobs in Spark UI at AbbVie, I always checked how many stages existed and which shuffle operations were the bottleneck. Reducing wide transformations — like replacing groupBy-dedup with dropDuplicates where possible — helped reduce stage count."

### Q6. What is a Shuffle? Why is it Expensive?

**One-line answer:**
> A shuffle is the process of redistributing data across partitions/executors so that related data is grouped together — it involves disk writes, network transfer, and serialization, making it the most expensive operation in Spark.

---

**Detailed Explanation:**

#### 🔹 What Happens During a Shuffle

```
Stage 1 Executors          Network          Stage 2 Executors
┌──────────────┐                          ┌──────────────┐
│ Executor 1   │──── partition A data ───▶│ Executor 3   │
│ (writes to   │──── partition B data ───▶│ (reads from  │
│  local disk) │                          │  network)    │
├──────────────┤                          ├──────────────┤
│ Executor 2   │──── partition A data ───▶│ Executor 4   │
│              │──── partition B data ───▶│              │
└──────────────┘                          └──────────────┘
     WRITE                                      READ
```

#### 🔹 Why It's Expensive — 4 Reasons

**1. Disk I/O (Write Phase)**
- Before sending data, Spark writes shuffle output to **local disk**
- Disk reads/writes are 100x slower than memory access

**2. Network Transfer**
- Data is **serialized** and sent across the network to other executors
- Network is the bottleneck for large datasets

**3. Serialization/Deserialization**
- Data must be **serialized** (JVM object → bytes) before network transfer
- Then **deserialized** (bytes → JVM object) on the receiving side
- CPU-intensive process

**4. Barrier Synchronization**
- The next stage **cannot start** until ALL shuffle data is written and transferred
- One slow executor (stragglers) delays the entire next stage

#### 🔹 How to Reduce Shuffles

```python
# ❌ BAD — Two shuffles: one for join, one for groupBy
result = df_patients.join(df_territory, "territory_id") \
                    .groupBy("region").count()

# ✅ BETTER — Use broadcast to eliminate join shuffle
from pyspark.sql.functions import broadcast
result = df_patients.join(broadcast(df_territory), "territory_id") \
                    .groupBy("region").count()
# Now only ONE shuffle (for groupBy)

# ✅ ALSO — Tune shuffle partitions
spark.conf.set("spark.sql.shuffle.partitions", "100")  # Default 200, tune to data size
```

**Interview tip:** "In my AbbVie pipelines, shuffle was the biggest performance bottleneck. I reduced it by: (1) using broadcast joins for small lookup tables, (2) pre-partitioning data on the join key before multi-step pipelines, (3) using AQE to dynamically coalesce shuffle partitions."

### Q7. What is Lazy Evaluation?

**One-line answer:**
> Lazy evaluation means Spark does NOT execute any transformation when you call it — it just builds an execution plan (DAG). Execution only happens when you call an action like `.count()`, `.show()`, or `.write()`.

---

**Detailed Explanation:**

#### 🔹 Two Types of Operations

| Type | Examples | Behavior |
|---|---|---|
| **Transformation** | filter, select, join, groupBy, withColumn | **Lazy** — builds DAG, no execution |
| **Action** | show(), count(), collect(), write(), save() | **Triggers** full DAG execution |

#### 🔹 Why Lazy?

```python
# NOTHING executes yet — just building a plan
df = spark.read.csv("s3://abbvie-raw/patients.csv")   # Lazy
df = df.filter(col("region") == "US")                  # Lazy
df = df.withColumn("load_ts", current_timestamp())    # Lazy
df = df.select("patient_id", "region", "load_ts")     # Lazy
df = df.dropDuplicates(["patient_id"])                 # Lazy

# THIS triggers everything above in one optimized plan ↓
df.writeTo("iceberg.bronze.patients").append()        # ACTION ← executes now
```

#### 🔹 The Power: Optimization Before Execution

Because Spark waits until an action, it can **optimize the entire plan**:
- Combine `filter` + `select` into a single pass
- Push `filter` down to the file scan (Iceberg/Parquet level)
- Remove unused columns before reading from disk
- Choose the best join strategy

```python
# Without lazy eval: 3 full passes over data
step1 = df.filter(col("region") == "US")   # Pass 1
step2 = step1.select("patient_id")          # Pass 2  
step3 = step2.dropDuplicates()             # Pass 3

# With lazy eval: Spark combines into 1 optimized pass
# filter + select + dedup → single file scan with column pruning
```

#### 🔹 Debugging Lazy Evaluation
```python
df.explain()          # Show execution plan (without running)
df.explain(True)      # Show all 4 plan phases
df.explain("formatted")  # Pretty formatted plan
```

**Interview tip:** "Lazy evaluation is why Spark is powerful for pipeline chaining. In my S3 → Iceberg pipelines, I'd chain 6-8 transformations before the write action, and Spark optimized the whole thing as one plan — including predicate pushdown into S3 file metadata."

### Q8. What are Jobs, Stages, and Tasks?

**One-line answer:**
> An **Action** creates a **Job**. A Job is split into **Stages** at shuffle boundaries. Each Stage contains **Tasks** — one Task per partition.

---

**Detailed Explanation:**

#### 🔹 Hierarchy
```
Application
└── Job (one per Action: count, write, show)
    └── Stage (split at each shuffle boundary)
        └── Task (one per partition)
```

#### 🔹 Job
- Created every time you call an **action** (`count()`, `write()`, `show()`, `collect()`)
- One Spark application can have many jobs
```python
df.count()   # Job 1
df.write.parquet("s3://...")  # Job 2
```

#### 🔹 Stage
- A Job is divided into stages at every **wide transformation (shuffle)**
- All operations within a stage are **narrow** (pipelined, no network)
- Stages execute **sequentially** (next stage waits for shuffle to complete)

```
Job: write patient data
  ├── Stage 1: read CSV → filter → withColumn   [narrow, pipelined]
  │      ↓ SHUFFLE (repartition by region)
  ├── Stage 2: groupBy region → agg             [wide, network transfer]
  │      ↓ SHUFFLE (sort for output)
  └── Stage 3: sort → write Iceberg             [narrow, pipelined]
```

#### 🔹 Task
- The **smallest unit** of work in Spark
- One task = one partition processed by one executor core
- If a stage has 200 partitions → 200 tasks run in parallel
- Each task runs the same code but on different data

```python
df.rdd.getNumPartitions()  # e.g., 200
# → Stage will have 200 tasks
# → Spark will run them in parallel across your executors
```

#### 🔹 Where to See This
- **Spark UI → Jobs tab** — see all jobs and their duration
- **Spark UI → Stages tab** — see each stage, task count, shuffle read/write
- **Databricks UI → Spark UI** — built-in, available for every job

**Interview tip:** "I used Spark UI heavily for optimization. When I saw one stage taking 80% of total time, I'd drill into the tasks — if one task was 10x longer, that pointed to data skew. That's when I applied salting or AQE skew join optimization."

### Q9. What is Broadcast Join? When Should You Use It?

**One-line answer:**
> A broadcast join sends a **copy of the small table to every executor** so the join can happen locally — eliminating the need to shuffle the large table across the network.

---

**Detailed Explanation:**

#### 🔹 How Normal Join Works (SortMergeJoin)
```
Large Table (50M rows)    Small Table (300 rows)
Executor 1: rows 1-10M    → Both tables SHUFFLED to group matching keys
Executor 2: rows 10-20M   → Expensive network transfer + disk I/O
Executor 3: rows 20-30M   → ...
```

#### 🔹 How Broadcast Join Works
```
Small Table (300 rows)
   → Serialized on Driver
   → Broadcast (copied) to ALL Executors
   → Each executor has full copy in memory

Large Table (50M rows)    ← NO SHUFFLE, stays local
Executor 1: joins locally using its in-memory copy of small table ✅
Executor 2: joins locally using its in-memory copy of small table ✅
```

#### 🔹 When to Use

✅ **Use broadcast join when:**
- One table is **small** (default threshold: 10 MB, you can increase to 50-100 MB)
- Joining a large fact table with a **dimension / lookup table**
- You want to **eliminate shuffle entirely**

❌ **Don't use broadcast join when:**
- Both tables are large (OOM risk)
- The "small" table is actually medium-large (executor memory pressure)

#### 🔹 Code
```python
from pyspark.sql.functions import broadcast

# Auto broadcast (if small table < 10MB, Spark does it automatically)
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", 10 * 1024 * 1024)  # 10 MB

# Manual broadcast (when you know table is small)
result = large_patient_df.join(
    broadcast(territory_lookup_df),   # 300 rows — broadcast this
    on="territory_id",
    how="left"
)

# Increase threshold for slightly larger tables
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", 50 * 1024 * 1024)  # 50 MB

# Disable broadcast (force sort-merge join for testing)
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", -1)
```

#### 🔹 Verify in Execution Plan
```python
result.explain()
# Look for: BroadcastHashJoin ✅  (good)
# Vs:       SortMergeJoin      (means shuffle is happening)
```

**Interview tip:** "In my AbbVie Veeva API pipelines, joining 50M patient records with a 300-row territory lookup using broadcast join was one of the single biggest performance wins. The join went from minutes (with shuffle) to seconds (broadcast)."

### Q10. What is Delta Lake? Explain ACID in Delta.

**One-line answer:**
> Delta Lake is an open-source storage layer on top of Parquet files that adds ACID transactions, versioning (time travel), schema enforcement, and scalable metadata to data lakes.

---

**Detailed Explanation:**

#### 🔹 What Problem Delta Solves

Without Delta, data lakes had problems:
- ❌ No transactions — partial writes corrupted data
- ❌ No schema enforcement — wrong data got in
- ❌ No versioning — can't roll back mistakes
- ❌ Small file problem — many small Parquet files
- ❌ Concurrent write conflicts

Delta Lake solves ALL of these.

#### 🔹 How Delta Works — The Transaction Log
```
Delta Table folder:
├── part-0001.parquet
├── part-0002.parquet
├── _delta_log/
│   ├── 00000000000000000000.json  ← commit 0: table creation
│   ├── 00000000000000000001.json  ← commit 1: first write
│   ├── 00000000000000000002.json  ← commit 2: update
│   └── 00000000000000000010.checkpoint.parquet  ← checkpoint
```

Every write = a new commit JSON in `_delta_log/`. Reading the log tells Spark exactly which Parquet files are valid for the current version.

#### 🔹 ACID in Delta

| ACID | What it means | How Delta achieves it |
|---|---|---|
| **Atomicity** | All or nothing — partial writes don't exist | Transaction log: commit written only if write succeeds fully |
| **Consistency** | Data always matches schema rules | Schema enforcement rejects invalid writes |
| **Isolation** | Concurrent reads/writes don't interfere | Optimistic concurrency control + snapshot isolation |
| **Durability** | Committed data survives crashes | Transaction log on cloud storage (S3/ADLS) is durable |

#### 🔹 Key Delta Features
```python
# Time Travel — query previous version
df_yesterday = spark.read.format("delta") \
    .option("versionAsOf", 5) \
    .load("/mnt/silver/patients")

df_yesterday = spark.read.format("delta") \
    .option("timestampAsOf", "2024-01-15") \
    .load("/mnt/silver/patients")

# See table history
from delta.tables import DeltaTable
dt = DeltaTable.forPath(spark, "/mnt/silver/patients")
dt.history().show()
```

**Interview tip:** "At AbbVie we used Apache Iceberg (not Delta), but Iceberg has the same ACID guarantees through its own metadata layer. When my multithreaded API ingestion wrote concurrently from Veeva and Ravex, Iceberg's isolation ensured readers never saw partial writes."

### Q11. What is Z-Ordering? When Should You Apply It?

**One-line answer:**
> Z-Ordering is a data co-location technique that physically sorts data in files so that rows with similar values on the Z-Order columns are stored together — enabling Spark to skip entire files when filtering on those columns.

---

**Detailed Explanation:**

#### 🔹 The Problem Z-Ordering Solves

Imagine a Delta table with 1000 Parquet files. Query:
```sql
SELECT * FROM patients WHERE patient_id = 'P123456'
```
Without Z-Order: Spark **opens all 1000 files** to find rows with that patient_id.
With Z-Order on `patient_id`: Spark reads **file statistics** → skips 998 files → opens only 2.

This is called **Data Skipping**.

#### 🔹 How Z-Ordering Works
- Uses a **Z-curve (space-filling curve)** to map multi-dimensional data to 1D
- Data physically sorted so similar values are in the same files
- Each file's **min/max statistics** are stored in Delta log
- Query engine reads stats first → skips files that can't contain matching rows

```
Without Z-Order:           With Z-Order by (patient_id, region):
File 1: patient_ids mixed  File 1: P100000–P199999, US patients
File 2: patient_ids mixed  File 2: P200000–P299999, US patients
File 3: patient_ids mixed  File 3: P100000–P199999, EU patients
...                        ...
Query P123456 → scan 1000 files  Query P123456 → scan 1 file ✅
```

#### 🔹 When to Apply Z-Ordering

✅ **Good candidates:**
- Columns used frequently in **WHERE / JOIN / GROUP BY**
- **High cardinality** columns: `patient_id`, `order_id`, `transaction_id`
- Multiple columns that are often filtered together

❌ **Bad candidates:**
- Columns already used for **partitioning** (redundant — already co-located)
- **Low cardinality** columns: `gender` (M/F), `status` (Active/Inactive) — use partitioning instead
- Rarely queried columns

#### 🔹 Code
```sql
-- Apply after OPTIMIZE
OPTIMIZE silver.patient_data
ZORDER BY (patient_id, load_date);
```

#### 🔹 Partitioning vs Z-Ordering

| | Partitioning | Z-Ordering |
|---|---|---|
| Best for | Low cardinality (region, year) | High cardinality (patient_id) |
| Creates | Separate folders per value | Co-located data within files |
| Skipping | Folder-level | File-level (within partition) |

**Interview tip:** "On my AbbVie Iceberg tables, I applied sort ordering on `patient_id` and `load_date` — the two most common filter columns in downstream queries. This enabled significant file skipping and improved query performance."

### Q12. What is OPTIMIZE in Delta?

**One-line answer:**
> `OPTIMIZE` compacts many small Parquet files into fewer, larger files — solving the "small file problem" that degrades read performance.

---

**Detailed Explanation:**

#### 🔹 The Small File Problem

Every incremental write creates new Parquet files:
```
Day 1 append: 50 files (batch was small)
Day 2 append: 80 files
Day 3 append: 120 files
...
After 1 month: 3000 files in your table!
```

Reading this table = opening 3000 files = 3000 metadata requests = SLOW.

#### 🔹 What OPTIMIZE Does
- Reads many small files
- Combines them into **fewer large files** (~1 GB each is ideal)
- Writes new compacted files
- Updates the Delta log to point to new files
- **Does not delete old files** (that's VACUUM's job)

```sql
-- Basic compaction
OPTIMIZE delta.`/mnt/silver/patients`;

-- Compaction + Z-Ordering (do both together)
OPTIMIZE delta.`/mnt/silver/patients`
ZORDER BY (patient_id);

-- Target file size (default 1 GB)
-- Configurable via: spark.databricks.delta.optimize.maxFileSize
```

#### 🔹 When to Run OPTIMIZE
- After **frequent small incremental writes** (API ingestion jobs that run hourly)
- Before **heavy reads / analytics queries**
- As a scheduled **maintenance job** (daily/weekly)
- In Databricks: **Auto Optimize** can be enabled to do this automatically

```python
# Enable Auto Optimize on a Delta table (Databricks)
spark.sql("""
    ALTER TABLE silver.patients
    SET TBLPROPERTIES (
        'delta.autoOptimize.optimizeWrite' = 'true',
        'delta.autoOptimize.autoCompact' = 'true'
    )
""")
```

**Interview tip:** "In my AbbVie pipelines, hourly API ingestion created many small files. I scheduled a nightly OPTIMIZE job to compact them. This made morning analytics queries 3x faster."

### Q13. What is VACUUM in Delta? What Does Retention Threshold Mean?

**One-line answer:**
> `VACUUM` physically deletes old data files that are no longer referenced by any version of the Delta table — freeing up storage. The **retention threshold** controls how far back in time you still want to be able to time-travel.

---

**Detailed Explanation:**

#### 🔹 Why VACUUM is Needed

Delta never deletes old Parquet files automatically (for time travel). Without VACUUM:
```
Initial write:  100 files
After 1 update: 100 new files (old 100 still on disk!)
After 10 updates: 1100 files on disk, only 100 are "current"
```
Storage costs grow forever.

#### 🔹 What VACUUM Does
- Scans the Delta log for **files no longer referenced** by any valid version
- Deletes those old Parquet files from storage (S3/ADLS)
- **Cannot be undone** — deleted files are gone permanently

#### 🔹 Retention Threshold
- Default: **7 days (168 hours)**
- After VACUUM, you can only time-travel to versions within the retention window
- Example: `RETAIN 168 HOURS` → you can still query versions from the past 7 days

```sql
-- Default: delete files older than 7 days
VACUUM silver.patients;

-- Custom: keep 30 days for compliance
VACUUM silver.patients RETAIN 720 HOURS;

-- Dry run: preview what WOULD be deleted (safe to run anytime)
VACUUM silver.patients RETAIN 168 HOURS DRY RUN;

-- Force lower retention (override safety check — USE CAREFULLY)
SET spark.databricks.delta.retentionDurationCheck.enabled = false;
VACUUM silver.patients RETAIN 0 HOURS;  -- ⚠️ Deletes ALL old versions
```

#### 🔹 OPTIMIZE vs VACUUM

| | OPTIMIZE | VACUUM |
|---|---|---|
| **Purpose** | Compact small files → fewer large files | Delete unreferenced old files |
| **Deletes old files?** | ❌ No | ✅ Yes |
| **Affects time travel?** | ❌ No | ✅ Yes (reduces window) |
| **Run frequency** | Daily/weekly | Weekly/monthly |

**Interview tip:** "For AbbVie's compliance requirements, I was careful with VACUUM — pharmaceutical data may need 30-day audit trails. We set RETAIN 720 HOURS to keep 30 days of history before vacuuming."

### Q14. Schema Enforcement vs Schema Evolution in Delta

**One-line answer:**
> **Schema enforcement** rejects writes that don't match the table's schema (ON by default). **Schema evolution** automatically updates the table schema when new columns appear (opt-in with `mergeSchema=true`).

---

**Detailed Explanation:**

#### 🔹 Schema Enforcement (Default ON)

Protects your table from bad writes:
```python
# Table schema: patient_id STRING, region STRING, status STRING

# New data has an extra column 'priority'
new_df = source.withColumn("priority", lit("HIGH"))

# This FAILS ✅ (enforcement working correctly)
new_df.write.format("delta").mode("append").save("/mnt/silver/patients")
# → AnalysisException: A schema mismatch detected when writing to the Delta table

# This also FAILS — wrong data type
wrong_type_df = df.withColumn("patient_id", col("patient_id").cast("integer"))
wrong_type_df.write.format("delta").mode("append").save("/mnt/silver/patients")
# → Type mismatch: expected STRING but got INTEGER
```

#### 🔹 Schema Evolution (Opt-in)

Allows schema to grow when new columns arrive:
```python
# New data has extra column 'priority' — auto-add to table
new_df.write.format("delta") \
    .mode("append") \
    .option("mergeSchema", "true") \  # ← enable evolution
    .save("/mnt/silver/patients")

# Table now has: patient_id, region, status, priority (new column) ✅
# Old rows have NULL for 'priority'
```

#### 🔹 Where to Use Each

| Layer | Strategy | Reason |
|---|---|---|
| **Bronze** (Raw) | Schema Evolution ON | Accept all source data as-is, even if schema changes |
| **Silver** (Cleansed) | Schema Enforcement ON | Protect downstream consumers from unexpected changes |
| **Gold** (Analytics) | Schema Enforcement ON (strict) | BI tools break if schema changes unexpectedly |

```python
# Bronze — accept everything
raw_df.write.format("delta") \
    .option("mergeSchema", "true") \
    .mode("append") \
    .save("/mnt/bronze/veeva_raw")

# Silver — strict enforcement (no option needed, it's the default)
clean_df.write.format("delta") \
    .mode("append") \
    .save("/mnt/silver/patients")  # Fails if schema doesn't match ✅
```

**Interview tip:** "In my AbbVie pipelines, Veeva API occasionally added new fields. Bronze layer accepted them via mergeSchema. But Silver layer had strict enforcement — if a new field appeared, the job would fail and alert me to review the change before letting it propagate downstream."

### Q15. What is Checkpointing in Spark?

**One-line answer:**
> Checkpointing saves a DataFrame/RDD to **reliable distributed storage** (HDFS/S3), breaking the DAG lineage chain — preventing recomputation failures and enabling fault-tolerant streaming.

---

**Detailed Explanation:**

#### 🔹 The Problem Checkpointing Solves

Spark tracks lineage (how each DataFrame was created). Long lineage = risk:
```
df1 = read()
df2 = df1.join(huge_table)
df3 = df2.groupBy(...)
df4 = df3.join(another_table)
df5 = df4.filter(...)
... (50 more transformations)
dfN = ...

# If dfN partition fails:
# Spark must recompute from df1 → all the way through 50+ steps
# With complex/expensive steps, this is very slow
```

Checkpoint breaks the chain:
```
df25.checkpoint()  # Saved to S3
# Now dfN failure → recompute from df25, not from df1 ✅
```

#### 🔹 Two Types of Checkpointing

**Batch Checkpointing:**
```python
# Set checkpoint directory (reliable storage)
spark.sparkContext.setCheckpointDir("s3://abbvie-checkpoints/batch/")

# Checkpoint a DataFrame
df_complex = df.join(huge_lookup).groupBy("region").agg(...)
df_complex.checkpoint()          # Async (doesn't trigger action)
df_complex_eager = df_complex.checkpoint(eager=True)  # Triggers action immediately
```

**Streaming Checkpointing:**
```python
# REQUIRED for Structured Streaming
# Stores: offsets, state, metadata
query = df_stream \
    .writeStream \
    .format("iceberg") \
    .option("checkpointLocation", "s3://abbvie-checkpoints/streaming/veeva_stream/") \
    .start("iceberg.silver.events")

# If query restarts → resumes from last checkpoint, no data loss
```

#### 🔹 Checkpoint vs Cache vs Persist

| | Cache | Persist | Checkpoint |
|---|---|---|---|
| Storage | Memory | Memory + Disk | Reliable storage (S3/HDFS) |
| Lineage broken? | ❌ No | ❌ No | ✅ Yes |
| Survives restart? | ❌ No | ❌ No | ✅ Yes |
| Use case | Reuse in same job | Large DFs | Long lineage / Streaming |

### Q16. Explain Caching vs Checkpointing vs Persistence

**One-line answer:**
> **Cache** = store in memory (reuse in same job, fast). **Persist** = cache with storage level options (memory + disk). **Checkpoint** = save to reliable storage and break lineage (for fault tolerance and streaming).

---

**Detailed Explanation:**

#### 🔹 Cache
```python
df.cache()
# Equivalent to: df.persist(StorageLevel.MEMORY_ONLY)

# Trigger cache population (cache is lazy too)
df.count()  # ← This forces actual caching

# Un-cache when done
df.unpersist()
```
- Stored in **executor memory**
- Lineage still **intact**
- If executor crashes → partition **recomputed from lineage**
- Best for: DataFrames reused **multiple times in one session**

#### 🔹 Persist
```python
from pyspark import StorageLevel

df.persist(StorageLevel.MEMORY_AND_DISK)     # Memory first, spill to disk
df.persist(StorageLevel.DISK_ONLY)           # Only disk (large DFs, less RAM)
df.persist(StorageLevel.MEMORY_AND_DISK_SER) # Serialized (less memory, more CPU)
df.persist(StorageLevel.OFF_HEAP)            # Off-heap memory (Tungsten)
```

| Storage Level | Memory | Disk | Serialized | Use When |
|---|---|---|---|---|
| MEMORY_ONLY | ✅ | ❌ | ❌ | Small DFs, fast access needed |
| MEMORY_AND_DISK | ✅ | ✅ (overflow) | ❌ | Medium DFs |
| DISK_ONLY | ❌ | ✅ | ✅ | Very large DFs, rare reuse |
| MEMORY_ONLY_SER | ✅ | ❌ | ✅ | Save memory, OK with CPU cost |

#### 🔹 Checkpoint
```python
spark.sparkContext.setCheckpointDir("s3://abbvie-checkpoints/")
df.checkpoint()   # Saves to S3, breaks lineage
```
- Stored in **reliable distributed storage (S3/HDFS)**
- Lineage **broken** — Spark forgets how the DF was created
- Survives executor/driver restart
- Best for: **Streaming**, iterative ML algorithms, very long DAG chains

#### 🔹 Decision Guide
```
Is the DF used multiple times in the same job?
  → YES: Use cache() or persist()
  → Is it larger than available memory?
    → YES: persist(MEMORY_AND_DISK)
    → NO: cache()

Is this a streaming job or iterative algorithm?
  → YES: Use checkpoint()

Is the DAG lineage very long and recomputation is expensive?
  → YES: Use checkpoint()
```

**Interview tip:** "At AbbVie, I cached the territory lookup DataFrame that was joined with 5 different source DataFrames in the same pipeline. Without cache, Spark would re-read and re-process it 5 times. With cache, it was read once and reused."

### Q17. What Does Adaptive Query Execution (AQE) Do?

**One-line answer:**
> AQE re-optimizes query plans **at runtime** based on actual data statistics collected during execution — fixing problems that the static planner couldn't predict (like data skew, incorrect partition counts, and suboptimal join strategies).

---

**Detailed Explanation:**

#### 🔹 Why AQE is Needed

Static planning happens BEFORE data is read:
- Spark **estimates** partition sizes → often wrong
- Spark **assumes** uniform data distribution → often skewed
- Spark **pre-decides** join strategies → may not be optimal

AQE fixes plans **during execution** using real statistics.

#### 🔹 Three Key AQE Features

**1. Dynamic Coalescing of Shuffle Partitions**
```
Default: 200 shuffle partitions (spark.sql.shuffle.partitions)
Reality: After filter, only 5 partitions have data

Without AQE: 200 tasks, 195 nearly empty = waste of resources
With AQE:   Merges 195 tiny partitions → runs 5 meaningful tasks ✅
```

**2. Dynamic Switch to Broadcast Join**
```
Plan said: SortMergeJoin (because table stats said "large")
Reality: After WHERE filter, right side has only 500 rows

Without AQE: Full SortMergeJoin with shuffle = slow
With AQE:   Detects small result → switches to BroadcastHashJoin ✅
```

**3. Skew Join Optimization**
```
Partition 1: 100 rows   → Task finishes in 5 seconds
Partition 2: 100 rows   → Task finishes in 5 seconds
Partition 3: 5M rows    → Task takes 10 minutes ← SKEW ❌

Without AQE: Entire job waits for slow task
With AQE:   Splits partition 3 into smaller sub-partitions ✅
```

#### 🔹 How to Enable AQE

```python
# Enable AQE (default ON in Spark 3.2+, Databricks runtime)
spark.conf.set("spark.sql.adaptive.enabled", "true")

# Feature 1: Coalesce partitions
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")
spark.conf.set("spark.sql.adaptive.coalescePartitions.minPartitionSize", "64MB")

# Feature 2: Switch to broadcast join dynamically
spark.conf.set("spark.sql.adaptive.localShuffleReader.enabled", "true")

# Feature 3: Skew join
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionFactor", "5")   # 5x median = skew
spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionThresholdInBytes", "256MB")
```

**Interview tip:** "AQE's skew join optimization was key for my AbbVie pipelines. Some US territories had 10x more patient records than others. Without AQE, those tasks were straggling. Enabling AQE skew join automatically split oversized partitions and brought execution time down significantly."

### Q18. What is Cluster Mode vs Client Mode?

**One-line answer:**
> In **Client Mode**, the Driver runs on the machine that submitted the job (your laptop/edge node). In **Cluster Mode**, the Driver runs inside the cluster on a worker node — more fault-tolerant for production.

---

**Detailed Explanation:**

#### 🔹 Client Mode
```
Your Machine / Edge Node / Databricks Notebook
┌─────────────────────┐
│  DRIVER runs here   │  ← SparkContext, DAG, task scheduling
│  (SparkSession)     │
└─────────────────────┘
          ↕ network
  ┌───────────────────────────────────────┐
  │           CLUSTER                     │
  │  Worker1: Executor  Worker2: Executor │
  │  Worker3: Executor  Worker4: Executor │
  └───────────────────────────────────────┘
```

✅ **Pros:** Easy to debug, logs visible locally, good for interactive development
❌ **Cons:** Job fails if submitting machine goes down, network latency between driver and executors

#### 🔹 Cluster Mode
```
  ┌────────────────────────────────────────────┐
  │               CLUSTER                       │
  │  ┌──────────────────┐                       │
  │  │  DRIVER runs here│  ← on a worker node   │
  │  └──────────────────┘                       │
  │  Worker2: Executor   Worker3: Executor       │
  │  Worker4: Executor   Worker5: Executor       │
  └────────────────────────────────────────────┘
           ↑
   You submit job and disconnect — cluster manages everything
```

✅ **Pros:** Driver close to executors (lower latency), fault-tolerant, submitting machine can go offline
❌ **Cons:** Harder to access driver logs directly

#### 🔹 Where Each is Used

| Mode | Where Used | Example |
|---|---|---|
| **Client Mode** | Databricks Notebooks, Jupyter, interactive dev | Your development work at AbbVie |
| **Cluster Mode** | `spark-submit`, ADF-triggered jobs, production pipelines | Scheduled ingestion jobs at AbbVie |

```bash
# Cluster mode
spark-submit --master yarn --deploy-mode cluster my_job.py

# Client mode
spark-submit --master yarn --deploy-mode client my_job.py
```

**Interview tip:** "At AbbVie, I used Databricks notebooks (client mode) for development and debugging. Production jobs triggered by ADF ran as Databricks Jobs in cluster mode — the driver was managed by Databricks and didn't depend on any user session staying active."

### Q19. How Does Spark Handle Faults Internally?

**One-line answer:**
> Spark handles faults through: **task retry** (re-run failed tasks on other executors), **lineage-based recovery** (recompute lost partitions), and **checkpointing** for streaming fault tolerance.

---

**Detailed Explanation:**

#### 🔹 Level 1: Task Failure
- A task failing = one partition failed on one executor
- Spark **automatically retries** the task (default: 4 attempts)
- Retried on a **different executor** (in case the original executor was faulty)
- If all retries fail → Stage fails → Job fails

```python
spark.conf.set("spark.task.maxFailures", "4")  # Default: 4 retries per task

# Speculative execution (run duplicate of slow tasks)
spark.conf.set("spark.speculation", "true")
spark.conf.set("spark.speculation.multiplier", "1.5")  # Task taking 1.5x median = speculate
```

#### 🔹 Level 2: Executor Failure
- Executor process crashes on a worker node
- **Cluster Manager** detects the failure and launches a new executor
- Tasks that were running on the failed executor are **rescheduled**
- Lost **cached data** on that executor is **recomputed from lineage**

#### 🔹 Level 3: Lineage-Based Recovery
- The key to Spark's fault tolerance is **RDD/DataFrame lineage**
- If partition P is lost → Spark traces back through the DAG to find the parent partition → recomputes

```
df1 = read(S3)          # Parent
df2 = df1.filter(...)   # Derived
df3 = df2.groupBy(...)  # Derived

# If df3 partition 5 is lost:
# → Recompute: read partition 5 from S3, filter, group → recreate partition 5
# No data lost ✅
```

#### 🔹 Level 4: Driver Failure
- **Client Mode**: Job completely fails if driver (your machine) crashes
- **Cluster Mode**: Some cluster managers (YARN, Kubernetes) can restart the driver
- **Streaming**: Checkpoint + driver restart = resume from last committed offset

#### 🔹 Level 5: Blacklisting
- Executor repeatedly fails → Spark **blacklists** it
- No more tasks sent to that executor
- Avoids sending tasks to a consistently faulty node

```python
spark.conf.set("spark.blacklist.enabled", "true")
spark.conf.set("spark.blacklist.task.maxTaskAttemptsPerExecutor", "1")
```

**Interview tip:** "In my AbbVie role, I did root cause analysis on connectivity failures. When Oracle JDBC connections timed out mid-task, Spark retried the task. But if the connection issue was persistent, all 4 retries failed. My fix was to add connection retry logic in the JDBC options and increase the connection timeout settings."

### Q20. What is Dynamic Partition Pruning (DPP)?

**One-line answer:**
> Dynamic Partition Pruning uses the result of filtering one table in a join to skip partitions in the other table — evaluated at runtime, not at planning time.

---

**Detailed Explanation:**

#### 🔹 The Problem

```sql
SELECT p.patient_id, p.load_date, t.region
FROM patient_events p
JOIN territories t ON p.territory_id = t.territory_id
WHERE t.region = 'US'
```

`patient_events` is partitioned by `region`.

**Without DPP:**
1. Spark reads ALL `patient_events` partitions (US, EU, APAC, LATAM...)
2. Joins with territories
3. Filters for region = 'US'
→ Read 100% of data, use 25%

**With DPP:**
1. Spark first evaluates `WHERE t.region = 'US'` on `territories`
2. Gets the territory_ids for region = 'US'
3. **Passes those IDs back** to prune `patient_events` partitions
4. Only reads `patient_events` partitions for US territories
→ Read 25% of data, use 25% ✅

#### 🔹 Requirements for DPP to Work
1. The table being pruned must be **partitioned**
2. The join must be on the **partition column**
3. The filtering side must be a **small table** (or produce a small result)
4. `spark.sql.optimizer.dynamicPartitionPruning.enabled = true` (default ON in Spark 3.x)

```python
# Enable DPP
spark.conf.set("spark.sql.optimizer.dynamicPartitionPruning.enabled", "true")

# Example: patient_events partitioned by load_date
# territory table filtered by region
result = spark.sql("""
    SELECT p.*, t.region
    FROM iceberg.silver.patient_events p
    JOIN iceberg.silver.territories t
        ON p.territory_id = t.territory_id
    WHERE t.region = 'US'
""")
# DPP: Only load_dates with US territory data are scanned in patient_events
```

#### 🔹 DPP vs Static Partition Pruning

| | Static Pruning | Dynamic Partition Pruning |
|---|---|---|
| When applied | Planning time | Runtime |
| Filter on | The partitioned table directly | On the join's other table |
| Example | `WHERE region = 'US'` | `WHERE t.region = 'US'` (t = dimension table) |

**Interview tip:** "On my 20+ TB Iceberg tables at AbbVie, DPP was critical. Tables were partitioned by load_date. Queries that joined with a filtered dimension table would previously scan all partitions. With DPP enabled, only relevant date partitions were opened — reducing scan size from 20 TB to 2 TB for typical date-range filtered queries."

### P2. Optimize Joins Between Two Large Tables

**Scenario:** Optimize joins between 50M patient records and various lookup/fact tables.

```python
from pyspark.sql import SparkSession
from pyspark.sql.functions import broadcast, col, concat, lit, rand, floor, explode, array

spark = SparkSession.builder \
    .config("spark.sql.adaptive.enabled", "true") \
    .config("spark.sql.adaptive.skewJoin.enabled", "true") \
    .config("spark.sql.autoBroadcastJoinThreshold", str(50 * 1024 * 1024)) \
    .getOrCreate()

patient_df = spark.read.format("iceberg").load("iceberg.silver.patients")      # 50M rows
territory_df = spark.read.format("iceberg").load("iceberg.silver.territories") # 300 rows
events_df = spark.read.format("iceberg").load("iceberg.silver.events")         # 30M rows

# ── Strategy 1: Broadcast Join (small table) ──────────────────────────────
# Eliminates shuffle for the large table entirely
result_1 = patient_df.join(
    broadcast(territory_df),   # Copies territory_df to every executor
    on="territory_id",
    how="left"
)
# Verify: result_1.explain() → should show BroadcastHashJoin

# ── Strategy 2: Repartition Before Multi-Step Pipeline ────────────────────
# When joining two large tables multiple times on the same key
# Pre-partition once → reuse partition layout
patient_rp = patient_df.repartition(200, col("territory_id"))
events_rp  = events_df.repartition(200, col("territory_id"))

# Both DFs now co-partitioned → join has much less shuffle
result_2 = patient_rp.join(events_rp, on="territory_id") \
                      .groupBy("territory_id") \
                      .count()

# ── Strategy 3: Salted Join for Skewed Data ───────────────────────────────
# Use when some join keys have 10x more rows (data skew)
SALT = 10

# Add random salt to large table
patient_salted = patient_df \
    .withColumn("salt", (floor(rand() * SALT)).cast("string")) \
    .withColumn("salted_key", concat(col("territory_id"), lit("_"), col("salt")))

# Replicate small table across all salt values
territory_salted = territory_df \
    .withColumn("salt_array", array([lit(str(i)) for i in range(SALT)])) \
    .withColumn("salt", explode(col("salt_array"))) \
    .withColumn("salted_key", concat(col("territory_id"), lit("_"), col("salt")))

result_3 = patient_salted.join(territory_salted, on="salted_key", how="left") \
    .drop("salt", "salted_key", "salt_array")  # Clean up salt columns

# ── Strategy 4: Tune Shuffle Partitions ──────────────────────────────────
# Default 200 is often wrong — tune to actual data size
# Rule of thumb: partition size = 100-200 MB
data_size_gb = 20
target_partition_mb = 128
num_partitions = int((data_size_gb * 1024) / target_partition_mb)  # ~160

spark.conf.set("spark.sql.shuffle.partitions", str(num_partitions))
```

### P3. PySpark Job to Ingest Data from Oracle to Delta/Iceberg

```python
from pyspark.sql import SparkSession
from pyspark.sql.functions import current_timestamp, lit, col
from datetime import datetime, timedelta

spark = SparkSession.builder \
    .appName("AbbVie-Oracle-Ingestion") \
    .config("spark.jars", "/opt/spark/jars/ojdbc8.jar") \
    .getOrCreate()

# ── Credentials from Key Vault (NEVER hardcode) ────────────────────────
# In Databricks:
ORACLE_URL  = "jdbc:oracle:thin:@//oracle-abbvie.host:1521/PRODDB"
ORACLE_USER = dbutils.secrets.get(scope="abbvie-kv", key="oracle-username")
ORACLE_PASS = dbutils.secrets.get(scope="abbvie-kv", key="oracle-password")

JDBC_OPTS = {
    "url":      ORACLE_URL,
    "user":     ORACLE_USER,
    "password": ORACLE_PASS,
    "driver":   "oracle.jdbc.driver.OracleDriver",
    "fetchsize": "5000",              # Rows fetched per JDBC batch
    "sessionInitStatement": "BEGIN DBMS_APPLICATION_INFO.SET_MODULE('SPARK','INGEST'); END;"
}

# ── Full Load ──────────────────────────────────────────────────────────
def full_load(oracle_table: str, iceberg_table: str, id_col: str,
              lower: int, upper: int, num_partitions: int = 20):
    """
    Parallel JDBC read using partitionColumn for parallelism.
    num_partitions = number of parallel JDBC connections to Oracle.
    """
    df = spark.read.format("jdbc") \
        .options(**JDBC_OPTS) \
        .option("dbtable", oracle_table) \
        .option("numPartitions", num_partitions) \
        .option("partitionColumn", id_col) \
        .option("lowerBound", lower) \
        .option("upperBound", upper) \
        .load()

    df = df.withColumn("_load_type", lit("FULL")) \
           .withColumn("_load_ts", current_timestamp())

    df.writeTo(iceberg_table) \
        .tableProperty("write.format.default", "parquet") \
        .overwritePartitions()

    print(f"✅ Full load complete: {df.count()} rows → {iceberg_table}")


# ── Incremental Load (Watermark) ───────────────────────────────────────
def incremental_load(oracle_table: str, iceberg_table: str,
                     watermark_col: str, last_run_ts: str):
    """
    Load only records changed after last_run_ts.
    watermark_col: timestamp column in Oracle (e.g., UPDATED_AT)
    """
    # Push filter down to Oracle (query Oracle, don't load all then filter)
    query = f"""(
        SELECT *
        FROM {oracle_table}
        WHERE {watermark_col} > TO_TIMESTAMP('{last_run_ts}', 'YYYY-MM-DD HH24:MI:SS')
    ) src"""

    df = spark.read.format("jdbc") \
        .options(**JDBC_OPTS) \
        .option("dbtable", query) \
        .load()
```

### P4. Write a PySpark UDF and Explain Why UDFs Are Expensive

```python
from pyspark.sql import SparkSession
from pyspark.sql.functions import udf, col, regexp_replace, when
from pyspark.sql.types import StringType
import pandas as pd

spark = SparkSession.builder.appName("UDF-Demo").getOrCreate()

# ── ❌ EXPENSIVE: Python UDF (row-by-row) ──────────────────────────────

@udf(returnType=StringType())
def mask_phi_udf(patient_id: str) -> str:
    """Mask last 4 digits of patient ID for GDPR/HIPAA compliance"""
    if patient_id and len(patient_id) > 4:
        return patient_id[:-4] + "****"
    return "****"

df_masked_slow = df.withColumn("masked_id", mask_phi_udf(col("patient_id")))

# ── WHY PYTHON UDFs ARE EXPENSIVE ─────────────────────────────────────
#
# 1. SERIALIZATION: JVM serializes each row → sends to Python process
#    Python processes → JVM deserializes result → repeat for EVERY row
#    50M rows = 50M serialize/deserialize cycles
#
# 2. ROW-BY-ROW: Python UDFs process ONE row at a time, not batches
#    No vectorization = no SIMD CPU optimization
#
# 3. NO CATALYST OPTIMIZATION: Spark's optimizer is blind to UDF logic
#    Cannot push filters inside UDF, cannot reorder operations
#
# 4. PYTHON GIL: Global Interpreter Lock prevents true parallelism
#    within a single Python process
#
# 5. TWO PROCESSES: JVM (Spark) + Python process running side-by-side
#    Context switching and IPC overhead

# ── ✅ BETTER: Built-in Spark Functions (stay in JVM) ─────────────────

from pyspark.sql.functions import concat, substring, length, lit, expr

# Stays entirely in JVM → Catalyst-optimized → Tungsten execution
df_masked_fast = df.withColumn(
    "masked_id",
    concat(
        substring(col("patient_id"), 1, length(col("patient_id")) - 4),
        lit("****")
    )
)
# 10-100x faster than Python UDF

# ── ✅ BEST when UDF is unavoidable: Pandas UDF (vectorized) ──────────

from pyspark.sql.functions import pandas_udf

@pandas_udf(StringType())
def mask_phi_pandas_udf(series: pd.Series) -> pd.Series:
    """
    Receives a pandas Series (batch of rows), not a single value.
    Data transferred as Arrow columnar batch — much more efficient.
    """
    return series.apply(
        lambda x: x[:-4] + "****" if isinstance(x, str) and len(x) > 4 else "****"
    )

df_masked_medium = df.withColumn("masked_id", mask_phi_pandas_udf(col("patient_id")))

# Performance comparison (approximate):
# Built-in functions:  1x    (fastest — JVM + Catalyst)
# Pandas UDF:          5x    (Arrow batches — good)
# Python UDF:          50x+  (row-by-row + serialization — slowest)

# ── Enable Arrow for Pandas UDFs ──────────────────────────────────────
```

### P7. Convert SQL Query to Optimized PySpark

```python
# ── Original SQL Query ─────────────────────────────────────────────────
sql_query = """
SELECT
    t.region,
    t.territory_id,
    COUNT(DISTINCT p.patient_id) AS patient_count,
    SUM(CASE WHEN p.status = 'ACTIVE' THEN 1 ELSE 0 END) AS active_count,
    RANK() OVER (PARTITION BY t.region ORDER BY COUNT(DISTINCT p.patient_id) DESC) AS rank_in_region
FROM patient_events e
JOIN patients p ON e.patient_id = p.patient_id
JOIN territories t ON p.territory_id = t.territory_id
WHERE e.event_date >= '2024-01-01'
  AND t.region IN ('US', 'EU')
GROUP BY t.region, t.territory_id
HAVING COUNT(DISTINCT p.patient_id) > 100
ORDER BY t.region, rank_in_region
"""

# ── Optimized PySpark Version ──────────────────────────────────────────
from pyspark.sql.functions import (
    broadcast, col, countDistinct, sum as spark_sum,
    when, rank, count
)
from pyspark.sql.window import Window

# 1. Load with early filters and column pruning
events = spark.read.format("iceberg") \
    .load("iceberg.silver.patient_events") \
    .filter(col("event_date") >= "2024-01-01") \  # Push filter to scan
    .select("patient_id", "event_date")            # Only needed columns

# 2. Filter territories early (small table → broadcast)
territories = spark.read.format("iceberg") \
    .load("iceberg.silver.territories") \
    .filter(col("region").isin("US", "EU")) \     # Filter before broadcast
    .select("territory_id", "region")             # Only needed columns

patients = spark.read.format("iceberg") \
    .load("iceberg.silver.patients") \
    .select("patient_id", "territory_id", "status") # Column pruning

# 3. Join with broadcasts (territories and patients are smaller tables)
result = events \
    .join(broadcast(patients), on="patient_id", how="inner") \
    .join(broadcast(territories), on="territory_id", how="inner") \
    .groupBy("region", "territory_id") \
    .agg(
        countDistinct("patient_id").alias("patient_count"),
        spark_sum(when(col("status") == "ACTIVE", 1).otherwise(0)).alias("active_count")
    ) \
    .filter(col("patient_count") > 100) \          # HAVING equivalent

# 4. Window function for RANK
window = Window.partitionBy("region").orderBy(col("patient_count").desc())
result = result \
    .withColumn("rank_in_region", rank().over(window)) \
    .orderBy("region", "rank_in_region")

result.show()

# Key optimizations applied:
# ✅ Early filter on event_date (partition pruning)
# ✅ Column pruning on all tables
# ✅ Filter territories before broadcast (smaller broadcast)
# ✅ Broadcast both small dimension tables (no shuffle for events)
# ✅ Push HAVING as filter after aggregation
```

### P10. Create a Medallion Architecture Pipeline (Bronze → Silver → Gold)

```python
from pyspark.sql import SparkSession, DataFrame
from pyspark.sql.functions import (
    col, current_timestamp, lit, sha2, concat_ws,
    upper, trim, when, row_number, count, sum as spark_sum, max as spark_max

from pyspark.sql.window import Window
from pyspark.sql.types import StructType
from delta.tables import DeltaTable

spark = SparkSession.builder \

    .config("spark.sql.adaptive.enabled", "true") \
    .getOrCreate()

# ══════════════════════════════════════════════════════════════════════════════
# BRONZE LAYER — Raw ingestion, exactly as received, zero business logic
# Goal: land data fast, preserve source fidelity, add audit metadata only
# ══════════════════════════════════════════════════════════════════════════════

def bronze_ingest_csv(source_path: str, source_system: str, table_name: str):
    """Ingest raw CSV from S3 to Iceberg Bronze — no transformations."""
    df_raw = spark.read \
        .option("header", True) \
        .option("inferSchema", True) \
        .csv(source_path)

    df_bronze = df_raw \
        .withColumn("_source_system",  lit(source_system)) \
        .withColumn("_source_path",    lit(source_path)) \
        .withColumn("_ingest_ts",      current_timestamp()) \
        .withColumn("_batch_id",       lit(f"{source_system}_{spark.sparkContext.applicationId}"))

    df_bronze.writeTo(f"iceberg.bronze.{table_name}") \
        .option("write.format.default", "parquet") \
        .append()

    print(f"📥 Bronze: {table_name} — {df_bronze.count()} rows ingested")
    return df_bronze.count()


def bronze_ingest_api(records: list, source_system: str, table_name: str):
    """Ingest API response list (from Veeva/Ravex/Smartsheet) to Bronze."""
    df_raw = spark.createDataFrame(records)

    df_bronze = df_raw \
        .withColumn("_source_system", lit(source_system)) \
        .withColumn("_ingest_ts",     current_timestamp())

    df_bronze.writeTo(f"iceberg.bronze.{table_name}").append()
    print(f"📥 Bronze: {table_name} — {df_bronze.count()} API records ingested")


# ══════════════════════════════════════════════════════════════════════════════
# SILVER LAYER — Cleanse, validate, deduplicate, standardize, type-cast
# Goal: trusted, consistent, DQ-checked data ready for analytics
# ══════════════════════════════════════════════════════════════════════════════

def silver_transform(bronze_table: str, silver_table: str, pk_col: str):
    """Transform Bronze → Silver with full DQ pipeline."""

    df = spark.read.format("iceberg").load(f"iceberg.bronze.{bronze_table}")

    # ── DQ Check 1: Null primary key ──────────────────────────────────
    df_null_pk  = df.filter(col(pk_col).isNull())
    df_valid    = df.filter(col(pk_col).isNotNull())

    if df_null_pk.count() > 0:
        df_null_pk.withColumn("_dq_rule", lit(f"NULL_{pk_col.upper()}")) \
                  .writeTo("iceberg.dq.failed_records").append()
        print(f"⚠️  {df_null_pk.count()} records quarantined (null PK)")

    # ── DQ Check 2: Duplicates ────────────────────────────────────────
    # Keep only the latest record per PK (deduplication by ingest time)
    window_spec = Window.partitionBy(pk_col).orderBy(col("_ingest_ts").desc())
    df_deduped = df_valid \
        .withColumn("_rn", row_number().over(window_spec)) \
        .filter(col("_rn") == 1) \
        .drop("_rn")

    dup_count = df_valid.count() - df_deduped.count()
    if dup_count > 0:
        print(f"ℹ️  {dup_count} duplicate records removed")

    # ── DQ Check 3: Standardize string columns ────────────────────────
    string_cols = [f.name for f in df_deduped.schema.fields
                   if str(f.dataType) == "StringType()"]
    df_std = df_deduped
    for c in string_cols:
        df_std = df_std.withColumn(c, upper(trim(col(c))))

    # ── Add Silver metadata ───────────────────────────────────────────
    df_silver = df_std \
        .withColumn("_silver_load_ts", current_timestamp()) \
        .withColumn("_record_hash",
            sha2(concat_ws("||", *[col(c) for c in string_cols[:5]]), 256)
        )

    # Drop Bronze audit columns (optional — keep clean in silver)
    df_silver = df_silver.drop("_source_path", "_batch_id")

    df_silver.writeTo(f"iceberg.silver.{silver_table}") \
        .option("write.format.default", "parquet") \
        .overwritePartitions()

    print(f"✅ Silver: {silver_table} — {df_silver.count()} clean records")
    return df_silver.count()


# ══════════════════════════════════════════════════════════════════════════════
# GOLD LAYER — Business aggregates, wide tables, ready for BI/Dashboards
# Goal: optimized for read performance, aligned to business metrics
# ══════════════════════════════════════════════════════════════════════════════

def gold_patient_summary():
    """Build Gold: Patient counts per territory per region (for dashboards)."""

    patients = spark.read.format("iceberg").load("iceberg.silver.patient_master")
    territories = spark.read.format("iceberg").load("iceberg.silver.territories")
    events = spark.read.format("iceberg") \
        .load("iceberg.silver.patient_events") \
        .filter(col("event_date") >= "2024-01-01")  # YTD only

    # Join (territories is small → broadcast)
    df_enriched = patients \
        .join(broadcast(territories), on="territory_id", how="left") \
        .join(events.groupBy("patient_id") \
              .agg(count("*").alias("ytd_events")),
              on="patient_id", how="left")

    # Aggregate to territory level
    df_gold = df_enriched.groupBy("region", "territory_id", "territory_name") \
        .agg(
            count("patient_id").alias("total_patients"),
            spark_sum(when(col("status") == "ACTIVE", 1).otherwise(0)).alias("active_patients"),
            spark_sum(when(col("status") == "INACTIVE", 1).otherwise(0)).alias("inactive_patients"),
            spark_sum(col("ytd_events")).alias("total_ytd_events"),
            spark_max("_silver_load_ts").alias("last_refreshed")
        ) \
        .withColumn("active_rate",
            (col("active_patients") / col("total_patients") * 100).cast("decimal(5,2)")
        ) \
        .withColumn("_gold_load_ts", current_timestamp())

    df_gold.writeTo("iceberg.gold.patient_territory_summary") \
        .partitionedBy("region") \
        .overwritePartitions()

    print(f"🏆 Gold: patient_territory_summary — {df_gold.count()} territory records")


# ══════════════════════════════════════════════════════════════════════════════
# PIPELINE ORCHESTRATION — Run all layers in sequence
# ══════════════════════════════════════════════════════════════════════════════

def run_full_pipeline():
    print("🚀 Starting Medallion Pipeline...\n")

    # Bronze
    bronze_ingest_csv(
        source_path="s3://abbvie-raw/veeva/patients/",
        source_system="VEEVA",
        table_name="veeva_patients_raw"
    )

    # Silver
    silver_transform(
        bronze_table="veeva_patients_raw",
        silver_table="patient_master",
        pk_col="patient_id"
    )

    # Gold
    gold_patient_summary()

    print("\n✅ Medallion Pipeline Complete!")


# Run
run_full_pipeline()
```

## 📊 Quick Reference Cheat Sheet

### Spark Optimization Checklist
| Issue | Detection | Fix |
|---|---|---|
| Full table scan | Explain plan: no file skipping | Add partition filter + enable DPP |
| Slow join | Explain: SortMergeJoin | Broadcast small table |
| Data skew | Spark UI: one task 10x longer | Enable AQE skewJoin or salt key |
| Too many small files | getNumPartitions() very high | OPTIMIZE + coalesce before write |
| Slow UDF | Python UDF in plan | Replace with built-in functions |
| OOM errors | Executor killed | Increase executor memory / use persist(DISK) |
| Too many shuffle partitions | 200 for small data | Tune spark.sql.shuffle.partitions |

### Key Spark Configs
```python
# AQE (Spark 3.x — enable always)
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")

# Join optimization
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", str(50*1024*1024))  # 50MB

# Shuffle
spark.conf.set("spark.sql.shuffle.partitions", "200")  # Tune to your data

# Arrow for pandas UDFs
spark.conf.set("spark.sql.execution.arrow.pyspark.enabled", "true")

# DPP
spark.conf.set("spark.sql.optimizer.dynamicPartitionPruning.enabled", "true")
```
