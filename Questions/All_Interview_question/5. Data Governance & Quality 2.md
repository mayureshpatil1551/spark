# Data Governance & Quality – Complete Interview Notes

# 1. What are common Data Quality (DQ) checks?

> “Data Quality checks are validations applied to ensure data is accurate, complete, consistent, and reliable before downstream processing.”

## Common DQ Checks

### Null Checks

Ensure mandatory columns are not null.

Example:

* customer_id should not be null

---

### Duplicate Checks

Detect duplicate records based on primary/business keys.

Example:

* duplicate order_id

---

### Referential Integrity Checks

Validate foreign keys exist in parent tables.

Example:

* product_id in sales table must exist in product dimension

---

### Data Type Validation

Ensure values match expected datatype.

Example:

* amount should be numeric

---

### Range Checks

Validate values fall within allowed ranges.

Example:

* age between 0 and 120

---

### Format Checks

Validate data patterns.

Example:

* email format
* phone number format

---

### Uniqueness Checks

Ensure unique values for business keys.

Example:

* employee_id should be unique

---

### Freshness Checks

Ensure latest data is received on time.

Example:

* daily file should arrive before 2 AM

---

# 4. Difference Between Schema Validation vs Schema Evolution

| Feature  | Schema Validation       | Schema Evolution             |
| -------- | ----------------------- | ---------------------------- |
| Purpose  | Enforce expected schema | Allow schema changes         |
| Behavior | Reject invalid schema   | Adapt to new columns         |
| Use Case | Strict governance       | Dynamic/semi-structured data |
| Example  | Reject extra column     | Add new column automatically |

---

## Interview Answer

> “Schema validation ensures incoming data strictly matches the expected schema, while schema evolution allows controlled schema changes such as adding new columns without breaking pipelines.”

# 5. Scenario: How to apply DQ checks in Bronze → Silver layer?

> “In Medallion Architecture, Bronze stores raw ingested data with minimal validation, while Silver applies cleansing and Data Quality rules before downstream consumption.”

## Typical Approach

### Bronze Layer

* Store raw source data
* Preserve original records
* Minimal validation

---

### Silver Layer

Apply DQ validations such as:

* null checks
* duplicate removal
* datatype validation
* referential checks
* standardization

---

## Example

### Bronze

Raw customer data from source

### Silver

* remove duplicate customer_id
* filter invalid emails
* standardize date formats
* enforce schema

# 6. Scenario: If CDC ingestion duplicates records, how do you fix it?

> “CDC pipelines can create duplicates because of retries, late-arriving events, or multiple updates for the same key. We typically handle this using deduplication logic based on primary keys, timestamps, or CDC sequence numbers.”

---

## Common Solutions

### 1. Window Function Deduplication

Keep latest record using:

* updated_timestamp
* sequence_number

Example:

```python
from pyspark.sql.window import Window
from pyspark.sql.functions import row_number, col

windowSpec = Window.partitionBy("customer_id") \
                   .orderBy(col("updated_ts").desc())

df_final = df.withColumn(
    "rn",
    row_number().over(windowSpec)
).filter("rn = 1")
```

---

### 2. MERGE INTO (Upsert)

Use Delta/Iceberg merge operations.

```sql
MERGE INTO target t
USING source s
ON t.id = s.id
WHEN MATCHED THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *
```
