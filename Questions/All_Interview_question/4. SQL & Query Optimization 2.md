# Theory Questions

## 1. Difference between Window Functions vs Aggregate Functions

### Interview Script

“Aggregate functions reduce multiple rows into a single output row per group. They are typically used with GROUP BY.

For example:

```sql
SELECT department, SUM(salary)
FROM employees
GROUP BY department;
```

This returns one row per department.

Window functions, on the other hand, perform calculations across a set of rows without collapsing the result set. They retain the original rows while adding calculated values.

For example:

```sql
SELECT employee_id,
       department,
       salary,
       RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS rank
FROM employees;
```

Here every employee row is preserved while ranking employees within each department.

In simple terms:
- Aggregate functions collapse rows.
- Window functions preserve rows while performing calculations across partitions.”

## 2. Explain Indexes and When Not to Use Them

### Interview Script

“Indexes improve query performance by allowing the database engine to locate rows faster instead of scanning the full table.

They work similarly to a book index where the database can directly jump to the required data.

Indexes are highly useful for:
- WHERE conditions
- JOIN columns
- ORDER BY
- GROUP BY

Common index types include:
- B-tree indexes
- Bitmap indexes
- Composite indexes

However, indexes should not always be used.

We should avoid indexes when:
- Tables are very small
- Columns have very low cardinality
- Frequent INSERT, UPDATE, DELETE operations happen because indexes increase write overhead
- Columns are rarely queried

In large-scale systems, excessive indexing can slow down data ingestion and increase storage usage.”

## 3. What is the Execution Plan?

### Interview Script

“An execution plan is the sequence of operations the database engine uses to execute a SQL query.

It shows:
- Table scans
- Index usage
- Join strategies
- Sorting
- Aggregations
- Data movement operations

Execution plans help identify performance bottlenecks.

For example, we can detect:
- Full table scans
- Expensive joins
- Large shuffle operations
- Missing indexes

In Spark SQL, we commonly use:

```sql
EXPLAIN FORMATTED
```

or:

```python
DataFrame.explain(True)
```

to analyze physical and logical execution plans.

Understanding execution plans is important for query optimization and troubleshooting slow SQL queries.”

## 4. Explain Inner vs Left vs Anti Joins

### Interview Script

“An INNER JOIN returns only matching records from both tables.

Example:

```sql
SELECT *
FROM orders o
INNER JOIN customers c
ON o.customer_id = c.customer_id;
```

Only matching customer records are returned.

A LEFT JOIN returns all records from the left table and matching rows from the right table.

If no match exists, NULL values are returned for the right-side columns.

Example:

```sql
SELECT *
FROM orders o
LEFT JOIN customers c
ON o.customer_id = c.customer_id;
```

An ANTI JOIN returns rows from the left table that do not have matching records in the right table.

Example:

```sql
SELECT *
FROM orders o
LEFT ANTI JOIN customers c
ON o.customer_id = c.customer_id;
```

This is commonly used for:
- Finding missing records
- Incremental loads
- Data quality checks
- CDC pipelines”

## 5. What does ANALYZE TABLE do?

### Interview Script

“ANALYZE TABLE collects table statistics used by the query optimizer.

These statistics include:
- Row counts
- Table size
- Column statistics
- Data distribution

The optimizer uses these statistics to make better decisions for:
- Join strategies
- Partition pruning
- Broadcast joins
- Query execution plans

Example:

```sql
ANALYZE TABLE sales COMPUTE STATISTICS;
```

For column statistics:

```sql
ANALYZE TABLE sales COMPUTE STATISTICS FOR COLUMNS customer_id;
```

Without accurate statistics, the optimizer may choose inefficient execution plans.”

# Practical Questions

## 1. Write SQL to Rank Customers Based on Purchases

### Interview Script

“Here we can use a window function like RANK or DENSE_RANK.”

```sql
SELECT customer_id,
       total_purchase,
       RANK() OVER (ORDER BY total_purchase DESC) AS customer_rank
FROM (
    SELECT customer_id,
           SUM(amount) AS total_purchase
    FROM orders
    GROUP BY customer_id
) t;
```

### Explanation

“This query first calculates total purchases per customer using aggregation and then ranks customers using a window function.”

## 2. Optimize a Slow SQL with Joins on Big Tables

### Interview Script

“To optimize joins on large tables, I generally follow these steps:

1. Filter data early before joins.
2. Select only required columns.
3. Use partition pruning.
4. Ensure join keys are indexed or partitioned.
5. Use broadcast joins for smaller tables.
6. Avoid data skew.
7. Analyze execution plans.

Example optimization in Spark:

```python
from pyspark.sql.functions import broadcast

result = large_df.join(
    broadcast(small_df),
    "customer_id",
    "inner"
)
```

Broadcast joins avoid large shuffle operations and significantly improve performance when one table is small.”

## 3. Convert Correlated Subquery into Join

### Interview Script

“Correlated subqueries can be slow because the inner query executes repeatedly for each row.

We can improve performance by converting them into joins.”

### Correlated Subquery

```sql
SELECT employee_id,
       employee_name
FROM employees e
WHERE salary > (
    SELECT AVG(salary)
    FROM employees
    WHERE department_id = e.department_id
);
```

### Optimized Join Version

```sql
WITH dept_avg AS (
    SELECT department_id,
           AVG(salary) AS avg_salary
    FROM employees
    GROUP BY department_id
)

SELECT e.employee_id,
       e.employee_name
FROM employees e
JOIN dept_avg d
ON e.department_id = d.department_id
WHERE e.salary > d.avg_salary;
```

### Explanation

“The optimized version computes department averages once and joins them back, avoiding repeated subquery execution and improving performance significantly on large datasets.”
