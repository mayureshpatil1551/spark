### Q1. What Was the Biggest Technical Challenge and How Did You Solve It?

**What to say:**

*"The biggest challenge was building a unified identity layer across multiple disconnected pharmaceutical data sources — Oracle, Snowflake, Smartsheet, and Salesforce — where the same patient or customer existed under different IDs in each system.*

*For example, a patient might be 'PAT-001' in Oracle but 'SF-C-9982' in Salesforce. No common key existed across systems. Running analytics across sources meant joining on fuzzy matches — which was slow, inconsistent, and produced duplicates or missed linkages.*

*My solution was engineering a Golden Record identity resolution system using cross-reference mappings — XREFs. The XREF table stored every known identifier for every entity across all source systems, with a master entity ID as the single key. Any incoming record from any source was first matched against the XREF table to resolve its master ID before loading into the Trusted layer.*

*This created a single source of truth that downstream MDM — Master Data Management — workflows could consume reliably. 10M+ records daily flow through this resolution layer with 99.9% uptime.*

*The technical challenges were: handling XREF mismatches when sources used slightly different spellings or ID formats, managing XREF updates when source systems changed IDs, and ensuring the resolution step didn't become a bottleneck at 10M+ daily volume — which I solved by broadcasting the XREF table and caching it across pipeline runs.*"

---

**Code — XREF-based Golden Record resolution:**

```python
from pyspark.sql.functions import broadcast, col, coalesce, sha2, concat_ws, current_timestamp

# XREF table: maps every source system ID to a master entity ID
# source_system | source_id    | master_entity_id | confidence
# ORACLE        | PAT-001      | ME-00123         | EXACT
# SALESFORCE    | SF-C-9982    | ME-00123         | EXACT
# SNOWFLAKE     | SNF-4421     | ME-00123         | FUZZY

xref_df = spark.read.format("iceberg") \
    .load("iceberg.trusted.xref_master") \
    .cache()  # Cache — used across all source pipelines

def resolve_golden_record(df, source_system: str):
    """
    Join incoming source records against XREF to assign master_entity_id.
    Records with no XREF match → quarantine for manual review.
    """
    resolved = df.join(
        broadcast(xref_df.filter(col("source_system") == source_system)),
        on=df["source_id"] == xref_df["source_id"],
        how="left"
    )

    # Records matched to a master ID → load to Trusted layer
    df_matched = resolved.filter(col("master_entity_id").isNotNull()) \
        .withColumn("golden_id", col("master_entity_id")) \
        .withColumn("_resolved_ts", current_timestamp())

    # Unmatched → quarantine for XREF enrichment
    df_unmatched = resolved.filter(col("master_entity_id").isNull()) \
        .withColumn("_unmatched_reason", lit("NO_XREF_MATCH"))

    df_unmatched.writeTo("iceberg.dq.unmatched_records").append()
    print(f"  Resolved: {df_matched.count()} | Unmatched: {df_unmatched.count()}")

    return df_matched

# Usage for each source
oracle_resolved    = resolve_golden_record(oracle_df,    "ORACLE")
salesforce_resolved = resolve_golden_record(salesforce_df, "SALESFORCE")
snowflake_resolved = resolve_golden_record(snowflake_df, "SNOWFLAKE")
```

### Q2. Why Iceberg? Why Not Delta?

**What to say:**

*"Three specific reasons drove the Iceberg choice for this pharma project.*

*First — AWS S3 as the storage layer. Our data lake was on AWS S3, not Azure ADLS. Delta Lake at the time had tighter integration with Databricks and Azure. Iceberg was the better-supported open format for multi-engine access on S3 — we used Spark for ingestion and transformation, but downstream teams needed Trino and Athena for ad-hoc queries on the same tables. Iceberg natively supports all three.*

*Second — regulatory audit trails. In pharmaceutical, you need to prove the state of data at any point in time for compliance audits. Iceberg's snapshot-based time travel meant we could query data as it existed on any audit date — without maintaining separate archive tables. This directly enabled the ACID transactions and time-travel audit capability mentioned in our architecture.*

*Third — XREF and historical snapshot management. We were managing 18+ years of pharmaceutical data with multi-level historical snapshots. Iceberg's partition evolution — changing how data is partitioned without rewriting files — was critical. We changed partition strategy multiple times as data grew without any data rewrite on 20 TB.*"

### Q3. How Did You Reduce Batch Processing Time by 30%?

> **Use 30% — that is what your new resume states.**

**What to say:**

*"The 30% reduction came from three targeted optimizations on our Spark pipelines.*

*First — partitioning strategy. Our Iceberg tables were partitioned suboptimally or not at all. I profiled query patterns and added partition fields aligned with how downstream jobs filtered the data — typically by ingestion date and source system. Daily incremental runs that previously scanned 20 TB were now touching only the relevant day's partition.*

*Second — shuffle reduction. Our pipelines had expensive sort-merge joins between large transaction tables and lookup tables. Analyzing the execution plan with `.explain()`, I identified that lookup tables — XREF, territory master, diagnosis codes — were small enough to broadcast. Replacing those joins with broadcast joins eliminated the shuffle entirely.*

*Third — query execution plan optimization. I tuned Spark partitioning settings and identified shuffle partition counts that were misconfigured — the default 200 was either too high causing empty tasks or too low causing large task sizes. I also enabled AQE to let Spark adapt shuffle partitions and handle skew dynamically.*

*Combined, these brought measurable reduction in end-to-end pipeline execution time and cloud compute cost — which translates directly to AWS EMR or Databricks DBU savings.*"

---

# 1. Aligned partitioning with query patterns
df.writeTo("iceberg.trusted.patient_events") \

# 2. Broadcast small lookups — eliminate shuffle

### Q4. Explain the 300 SQL Dynamic Scripts Optimization

**What to say:**

*"This was about eliminating the manual overhead of environment provisioning. Every time we set up a new environment — Dev, UAT, Prod — someone had to manually write or copy-paste SQL configuration scripts for table DDLs, database links, grants, synonyms, sequence configurations, and schema setups. With 300+ such objects, each with environment-specific values — different schema names, different DB link names, different user accounts — this was taking 15+ hours per setup.*

*The problem beyond time was consistency. Manual copy-paste across 300 scripts meant mistakes — wrong schema name in one script, a missed grant in another. These inconsistencies caused bugs that only appeared in UAT or Prod.*

*My solution was a Python script using Jinja2 templates. Each SQL script became a template with placeholders for environment-specific values. A YAML config file held the values per environment. The Python script read the config, rendered all 300 templates, and output ready-to-execute SQL files.*

*One command replaced 15+ hours of manual work. Environments were now guaranteed identical in structure. All scripts version-controlled in Git — full audit trail.*"

---

```python
# Template: sql_templates/create_table.sql.j2
# CREATE TABLE {{ schema_name }}.PATIENT_MASTER (
#   PATIENT_ID VARCHAR(50),
#   TERRITORY_ID VARCHAR(50),
#   ...
# );
# GRANT SELECT ON {{ schema_name }}.PATIENT_MASTER TO {{ grant_user }};
# CREATE SYNONYM {{ schema_name }}.PM FOR {{ schema_name }}.PATIENT_MASTER@{{ db_link }};

# config/environments.yaml
# dev:
#   schema_name: ABBVIE_DEV
#   db_link:     DEV_DBLINK
#   grant_user:  SVC_DEV_USER
# prod:
#   schema_name: ABBVIE_PROD
#   db_link:     PROD_DBLINK
#   grant_user:  SVC_PROD_USER

from jinja2 import Template
import yaml, os

def generate_sql_scripts(env: str, template_dir: str, output_dir: str):
    with open("config/environments.yaml") as f:
        config = yaml.safe_load(f)[env]

    os.makedirs(output_dir, exist_ok=True)
    count = 0

    for filename in os.listdir(template_dir):
        if not filename.endswith(".sql.j2"):
            continue

        with open(f"{template_dir}/{filename}") as f:
            rendered = Template(f.read()).render(**config)

        out_file = f"{output_dir}/{filename.replace('.j2', '')}"
        with open(out_file, "w") as f:
            f.write(rendered)
        count += 1

    print(f"✅ {count} scripts generated for '{env}' — ready to execute")

# One command → 300 consistent, environment-specific SQL scripts
generate_sql_scripts("prod", "sql_templates/", "output/prod/")
```

### Q6. How Did You Automate Ingestion Frameworks?

**What to say:**

*"The goal was a single reusable ingestion framework that could handle any source without writing new pipeline code each time. We had four different source types: Oracle via JDBC, Snowflake, Smartsheet API, and Salesforce API — each with different connection mechanics, authentication patterns, and pagination logic.*

*I built a config-driven framework where each data source was described in a config table — one row per source — containing: source type, connection reference, load mode (full or incremental), watermark column for incrementals, target Iceberg table, partition columns, and DQ rules to apply.*

*The framework read the config row, instantiated the right connector class for that source type, ran the extract with the right load strategy, applied DQ checks, routed failed records to quarantine, and wrote clean records to the target Iceberg table.*

*For API sources — Smartsheet and Salesforce — the framework also handled OAuth token management, request pagination, and rate limit backoff. For high-volume sources like Salesforce, it used multithreading to call multiple endpoints in parallel.*

*The result: onboarding a new data source meant adding one row to the config table, not writing a new notebook. The same framework handled all 10M+ daily records across all four source systems.*"

---

### Q1. Explain the Pipeline for XML → Hive / Iceberg

**What to say (framed from your actual resume):**

*"In the main project, I developed ingestion workflows in PySpark to load raw file-based data including XML from Amazon S3 into Apache Iceberg tables. The pattern was consistent regardless of XML schema.*

*Step one: read XML using the spark-xml library, configuring rowTag to the repeating element so each occurrence becomes one Spark row. Iceberg auto-infers the schema including nested structs and arrays.*

*Step two: flatten. XML attributes get an underscore prefix. Nested struct fields are accessed with dot notation — `col('Vendor.VendorID')`. Arrays like line items are exploded with `explode_outer` — one row per element, parent fields repeated.*

*Step three: type casting, null handling, basic DQ — filter null transaction IDs, cast amount fields to Decimal, standardize date formats.*

*Step four: write to Iceberg partitioned by ingestion date. For Hive in the FinOps work specifically — same PySpark code, `saveAsTable` targeting a Hive Metastore table instead.*"

---

```python
from pyspark.sql import SparkSession
from pyspark.sql.functions import col, explode_outer, current_timestamp

spark = SparkSession.builder \
    .config("spark.jars.packages", "com.databricks:spark-xml_2.12:0.17.0") \
    .enableHiveSupport() \
    .getOrCreate()

# Read XML from S3
df_raw = spark.read \
    .format("xml") \
    .option("rowTag",       "Transaction") \   # each <Transaction> = one row
    .option("rootTag",      "Transactions") \
    .option("inferSchema",  "true") \
    .load("s3://abbvie-raw/finops/xml/")

# Flatten struct + explode array
df_flat = df_raw.select(
    col("_TransactionID").alias("transaction_id"),
    col("_Date").alias("transaction_date"),
    col("Amount._VALUE").cast("decimal(18,2)").alias("total_amount"),
    col("Amount._Currency").alias("currency"),
    col("Vendor.VendorID").alias("vendor_id"),
    col("Vendor.VendorName").alias("vendor_name"),
    col("LineItems.LineItem").alias("line_items")
)

df_final = df_flat \
    .withColumn("line", explode_outer(col("line_items"))) \
    .select(
        "transaction_id", "transaction_date",
        "total_amount", "currency",
        "vendor_id", "vendor_name",
        col("line.Code").alias("item_code"),
        col("line.Amount").cast("decimal(18,2)").alias("line_amount"),
        current_timestamp().alias("load_ts")
    ) \
    .filter(col("transaction_id").isNotNull())

# Write to Iceberg (standard path)
df_final.writeTo("iceberg.raw.finops_transactions") \
    .partitionedBy("transaction_date").append()

# OR write to Hive (FinOps PoC path)
df_final.write.format("hive").mode("append") \
    .partitionBy("transaction_date") \
    .saveAsTable("finops_db.bronze_transactions")
```

### Q3. What Were the Major Performance Bottlenecks?

**What to say:**

*"Three main bottlenecks in the XML pipeline.*

*First — small file problem. Source XML files were 2-5 MB each, thousands of them. Spark created one task per file — thousands of tiny tasks, massive task scheduling overhead, tiny payloads per task. Fix: coalesce before write to reduce output files, and read all files in a single scan with wildcard path rather than looping.*

*Second — data skew after explode. Some transaction XMLs had thousands of line items, others had five. After explode, some partitions were 100x larger — one task ran 10x longer than all others. Fix: repartition after explode on a balanced column, and enable AQE skew join handling.*

*Third — schema inference overhead. `inferSchema=true` makes spark-xml do a full scan twice — once to infer types, once to load. For thousands of files this doubled read time. Fix: define schema explicitly as a StructType so inference is skipped entirely, single read pass.*"

---

df_final.coalesce(20) \
    .write.format("hive").mode("append") \
    .partitionBy("transaction_date") \
    .saveAsTable("finops_db.bronze_transactions")

# Fix 2: Skew after explode — repartition + AQE
df_exploded = df_flat.withColumn("line", explode_outer(col("line_items")))
df_rebalanced = df_exploded.repartition(100, col("transaction_date"))
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")

# Fix 3: Skip schema inference — define schema explicitly (single read pass)
from pyspark.sql.types import (
    StructType, StructField, StringType, DecimalType, ArrayType
)
xml_schema = StructType([
    StructField("_TransactionID", StringType()),
    StructField("_Date",          StringType()),
    StructField("Amount", StructType([
        StructField("_VALUE",    DecimalType(18, 2)),
        StructField("_Currency", StringType())
    ])),
    StructField("Vendor", StructType([
        StructField("VendorID",   StringType()),
        StructField("VendorName", StringType())
    ])),
    StructField("LineItems", StructType([
        StructField("LineItem", ArrayType(StructType([
            StructField("Code",   StringType()),
            StructField("Amount", DecimalType(18, 2))
        ])))
    ]))
])

df_raw = spark.read \
    .format("xml") \
    .option("rowTag", "Transaction") \
    .schema(xml_schema) \     # ← no inference, single pass ✅
    .load("s3://abbvie-raw/finops/xml/")
