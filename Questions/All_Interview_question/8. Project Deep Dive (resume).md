### Q1. What Was the Biggest Technical Challenge and How Did You Solve It?

**What to say:**

*"The biggest challenge was handling concurrent multi-source API ingestion at scale — specifically Veeva and Ravex APIs — where each API had rate limits, paginated responses, and large payloads. Running them sequentially was taking several hours and wasn't meeting our SLA windows.*

*The root issue: we were calling each API endpoint one at a time inside a Python loop — purely sequential. For Veeva, which had 50+ territory-level endpoints, this meant endpoint 1 finished, then endpoint 2 started, and so on.*

*My solution was multithreading with Python's `ThreadPoolExecutor`. I designed a parallel ingestion framework where multiple API endpoints were called simultaneously in separate threads, each writing results to its own partition in Iceberg Bronze. This reduced the API ingestion time by several hours — which earned me the Emerging Talent Award.*

*The technical challenges within that: thread safety when writing shared audit logs, handling per-thread rate limit errors with retry logic and exponential backoff, and ensuring each thread's failure was isolated — one endpoint failing didn't kill the entire ingestion run.*"

---

**Code — the core pattern:**

```python
from concurrent.futures import ThreadPoolExecutor, as_completed
from pyspark.sql import SparkSession
import requests, time

def ingest_territory(territory_id: str, api_key: str, spark) -> dict:
    """Single thread — fetches one territory's data from Veeva API."""
    url = f"https://api.veeva.com/v1/territories/{territory_id}/patients"
    retries = 3

    for attempt in range(retries):
        try:
            response = requests.get(
                url,
                headers={"Authorization": f"Bearer {api_key}"},
                timeout=30
            )
            response.raise_for_status()
            records = response.json()["data"]

            # Write this thread's data directly to Iceberg Bronze
            df = spark.createDataFrame(records)
            df.withColumn("territory_id", lit(territory_id)) \
              .withColumn("_ingest_ts", current_timestamp()) \
              .writeTo(f"iceberg.bronze.veeva_patients") \
              .append()

            return {"territory_id": territory_id, "status": "SUCCESS", "count": len(records)}

        except Exception as e:
            if attempt == retries - 1:
                return {"territory_id": territory_id, "status": "FAILED", "error": str(e)}
            time.sleep(2 ** attempt)   # exponential backoff


def run_parallel_ingestion(territory_ids: list, api_key: str, spark, max_workers: int = 10):
    results = []
    with ThreadPoolExecutor(max_workers=max_workers) as executor:
        futures = {
            executor.submit(ingest_territory, tid, api_key, spark): tid
            for tid in territory_ids
        }
        for future in as_completed(futures):
            result = future.result()
            results.append(result)
            print(f"  {'✅' if result['status']=='SUCCESS' else '❌'} {result['territory_id']}")

    failed = [r for r in results if r["status"] == "FAILED"]
    print(f"\nCompleted: {len(results)-len(failed)} success, {len(failed)} failed")
    return results
```

### Q2. Why Iceberg? Why Not Delta?

**What to say:**

*"Three specific reasons drove the Iceberg decision at AbbVie.*

*First — multi-engine support. We needed Spark for our ingestion and transformation pipelines, but the analytics team was also using Trino for ad-hoc SQL queries directly on the same tables. Delta Lake had limited Trino support at the time. Iceberg worked natively with both.*

*Second — hidden partitioning. Our source data had a timestamp column event_ts. In Delta, we'd need to add a separate load_date column, populate it manually on every write, and remember to filter on it in every query. With Iceberg's `days(event_ts)` transform, no extra column — Iceberg computes the partition automatically. When we needed to change partition granularity later, it was a metadata-only ALTER TABLE — no data rewrite on 20 TB.*

*Third — vendor neutrality. AbbVie's data strategy was to avoid lock-in to a single vendor's proprietary format. Iceberg is Apache-governed, open format — readable by any engine today or in the future.*"

---

**Quick answer if pressed for one sentence:**
> *"Multi-engine access — Spark for ingestion, Trino for analytics, both on same Iceberg tables. Delta didn't support that combination cleanly. Plus Iceberg's hidden partitioning meant no schema pollution with extra partition columns."*

### Q3. How Did You Reduce Batch Processing Time by 50%?

**What to say:**

*"This was a combination of four specific optimizations, not one silver bullet.*

*One — partitioning strategy. Our Iceberg tables were originally unpartitioned. I added `days(event_ts)` and `region` as partition fields. Daily incremental runs that previously scanned the full table now touched only today's partition — a fraction of the data.*

*Two — broadcast joins. We had several joins between large patient event tables and small lookup tables — territory master (300 rows), diagnosis codes (500 rows). These were causing full SortMergeJoins with shuffles. Wrapping the small tables in `broadcast()` eliminated those shuffles entirely.*

*Three — Adaptive Query Execution. Enabled `spark.sql.adaptive.enabled=true` and `spark.sql.adaptive.skewJoin.enabled=true`. Some US territories had 10x more patient records than others — massive skew. AQE's skew join handling automatically split oversized partitions. Before AQE, one task ran for 20 minutes while 199 others finished in 2 minutes.*

*Four — replacing Python UDFs. We had custom masking and transformation logic written as Python UDFs — row-by-row, JVM-to-Python serialization overhead. I rewrote them using Spark built-in functions — `when`, `regexp_replace`, `substring` — which stay in the JVM and are Catalyst-optimized.*

*Together, these brought pipeline runtime from roughly 4 hours down to under 2 hours.*"

---

**The four levers:**

```python
# 1. Partition strategy (before: full scan, after: partition pruning)
df.writeTo("iceberg.silver.patient_events") \
  .partitionedBy("days(event_ts)", "region") \
  .append()

# 2. Broadcast join (before: SortMergeJoin + shuffle, after: local join)
from pyspark.sql.functions import broadcast
result = patient_df.join(broadcast(territory_lookup), "territory_id")

# 3. AQE — handles skew, coalesces empty partitions automatically
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")

# 4. Replace Python UDF with built-in functions
# Before (slow — JVM → Python per row):
# @udf(StringType())
# def mask_id(x): return x[:-4] + "****"

# After (fast — stays in JVM, Catalyst-optimized):
from pyspark.sql.functions import concat, substring, lit, length
df.withColumn("masked_id",
    concat(substring(col("patient_id"), 1, length(col("patient_id")) - 4), lit("****"))
)
```

### Q4. Explain the 300 SQL Dynamic Scripts Optimization

**What to say:**

*"This was from my earlier role on the same AbbVie project, July 2023 to August 2024.*

*The problem: every time a new environment was set up — Dev, UAT, Prod — a team member would manually write or copy-paste SQL configuration scripts. Things like table DDLs, grants, synonyms, sequence resets, linked server configs. These were environment-specific — different schema names, different connection strings, different user accounts per environment. With 300+ such objects, manual setup was taking 15+ hours per environment provisioning.*

*Beyond the time cost, it was error-prone — copy-paste mistakes, missing a grant, wrong schema name in one script. Environment inconsistency caused bugs that only appeared in UAT or Prod, not Dev.*

*My solution was a Python script that dynamically generated all 300 SQL scripts from a configuration template. You'd pass in the environment name — Dev, UAT, or Prod — and the script would read a config file with environment-specific values, inject them into Jinja2 templates, and output all 300 ready-to-execute SQL files. One command, consistent output every time.*

*Result: environment setup went from 15+ manual hours to under 30 minutes of automated generation and execution. Completely standardized — Dev, UAT, and Prod environments were now guaranteed identical in structure.*"

---

**The pattern:**

```python
# config/environments.yaml
# dev:
#   schema_name: ABBVIE_DEV
#   db_link: DEV_DBLINK
#   grant_user: SVC_DEV_USER
# prod:
#   schema_name: ABBVIE_PROD
#   db_link: PROD_DBLINK
#   grant_user: SVC_PROD_USER

from jinja2 import Template
import yaml, os

def generate_sql_scripts(env: str, template_dir: str, output_dir: str):
    # Load environment config
    with open("config/environments.yaml") as f:
        config = yaml.safe_load(f)[env]

    os.makedirs(output_dir, exist_ok=True)
    generated = 0

    # Loop through all SQL templates
    for template_file in os.listdir(template_dir):
        if not template_file.endswith(".sql.j2"):
            continue

        with open(f"{template_dir}/{template_file}") as f:
            template = Template(f.read())

        # Inject environment-specific values
        rendered_sql = template.render(**config)

        output_file = f"{output_dir}/{template_file.replace('.j2', '')}"
        with open(output_file, "w") as f:
            f.write(rendered_sql)

        generated += 1

    print(f"✅ Generated {generated} SQL scripts for '{env}' environment")
    return generated


# Usage — one command replaces 15+ hours of manual work
generate_sql_scripts(
    env="prod",
    template_dir="sql_templates/",   # 300 .sql.j2 template files
    output_dir="output/prod/"         # 300 ready-to-execute .sql files
)
```

### Q6. How Did You Automate Ingestion Frameworks?

**What to say:**

*"The goal was a config-driven ingestion framework — instead of writing a new PySpark notebook for every new data source, we built one generic framework that could be configured to handle any source.*

*The framework was driven by a config table in Iceberg — one row per source, with columns for: source type (API, S3 file, JDBC), connection details, load mode (full or incremental), watermark column, target Iceberg table, partition columns, and DQ rules to apply.*

*When a new source needed to be onboarded — say, a new Smartsheet sheet — instead of writing new code, we added one row to the config table. The framework read the config, instantiated the right connector (REST client, JDBC reader, S3 reader), applied the right load strategy, ran DQ checks, and wrote to the target Iceberg table.*

*For API sources specifically, the framework handled: OAuth token management, pagination, rate limit backoff, multithreaded parallel calls, and writing each thread's result to Iceberg. This was generic — the same code ran for Veeva, Ravex, and Smartsheet API ingestion.*

*Combined with Modak Nabu for logging — every ingestion run was tracked: source system, records read, records passed DQ, records quarantined, run duration, batch ID. Full audit trail out of the box for every source.*"

### Q1. Explain the Pipeline for XML → Hive

**What to say:**

*"The source was financial XML files — vendor transaction data with nested structures: transaction header, line items array, vendor details as nested elements. These landed in an S3/ADLS folder.*

*Step one was reading XML with the spark-xml library — `com.databricks:spark-xml`. The key configuration was setting `rowTag` to the repeating element — in our case Transaction — so each XML Transaction node became one Spark row. Iceberg inferred the schema including nested structs and arrays.*

*Step two was flattening the nested structure. Transaction had a nested Vendor struct and a LineItems array. I used dot notation for struct fields — `col('Vendor.VendorID')` — and explode for the array — one row per line item. After exploding, each row represented one line item with its parent transaction context.*

*Step three was applying data types, null handling, and basic DQ — validating transaction IDs, casting amount fields to Decimal, standardizing date formats.*

*Step four was writing to a Hive table partitioned by transaction date, using Spark's `saveAsTable` with partitionBy. Downstream users could then query with HiveQL through their existing BI tools.*"

---

**Pipeline:**

```python
from pyspark.sql import SparkSession
from pyspark.sql.functions import col, explode_outer, current_timestamp, lit

spark = SparkSession.builder \
    .appName("FinOps-XML-Ingest") \
    .config("spark.jars.packages", "com.databricks:spark-xml_2.12:0.17.0") \
    .enableHiveSupport() \
    .getOrCreate()

# Step 1: Read XML
# <Transactions>
#   <Transaction TransactionID="T001" Date="2024-01-15">
#     <Amount Currency="USD">5000.00</Amount>
#     <Vendor><VendorID>V100</VendorID><VendorName>PharmaCo</VendorName></Vendor>
#     <LineItems>
#       <LineItem><Code>LAB001</Code><Amount>2500.00</Amount></LineItem>
#     </LineItems>
#   </Transaction>
# </Transactions>

df_raw = spark.read \
    .format("xml") \
    .option("rowTag", "Transaction") \        # each <Transaction> = one row
    .option("rootTag", "Transactions") \
    .option("inferSchema", "true") \
    .option("nullValue", "") \
    .load("abfss://raw@abbviedatalake.dfs.core.windows.net/finops/xml/")

# Step 2: Flatten nested XML
# XML attributes come with underscore prefix: _TransactionID, _Date
df_flat = df_raw.select(
    col("_TransactionID").alias("transaction_id"),
    col("_Date").alias("transaction_date"),
    col("Amount._VALUE").cast("decimal(18,2)").alias("total_amount"),
    col("Amount._Currency").alias("currency"),
    col("Vendor.VendorID").alias("vendor_id"),
    col("Vendor.VendorName").alias("vendor_name"),
    col("LineItems.LineItem").alias("line_items")  # array column
)

# Step 3: Explode line items
df_final = df_flat \
    .withColumn("line", explode_outer(col("line_items"))) \
    .select(
        "transaction_id", "transaction_date",
        "total_amount", "currency",
        "vendor_id", "vendor_name",
        col("line.Code").alias("item_code"),
        col("line.Amount").cast("decimal(18,2)").alias("line_amount"),
        current_timestamp().alias("load_ts")
    ) \
    .filter(col("transaction_id").isNotNull())    # DQ: drop null TXN IDs

# Step 4: Write to Hive (partitioned by transaction_date)
df_final.write \
    .format("hive") \
    .mode("append") \
    .partitionBy("transaction_date") \
    .saveAsTable("finops_db.bronze_transactions")

print(f"✅ Loaded {df_final.count()} line items into Hive")
```

### Q3. What Were the Major Performance Bottlenecks?

**What to say:**

*"Three main bottlenecks in the XML pipeline.*

*First — small file problem from the XML source. Each XML file from the financial system was small — maybe 2-5 MB. We had thousands of them. Spark was spinning up one task per file — thousands of tiny tasks with high overhead and tiny payloads. The fix was using `spark.read.xml` with a wildcard path to read all files in one scan, then coalescing before write to produce fewer, larger output files.*

*Second — explode creating data skew. Some transaction XML files had thousands of line items while others had 5. After explode, some partitions had 100x more rows than others — causing classic skew where one task ran 10x longer than all others. The fix was enabling AQE skew join handling and also repartitioning after explode on transaction_date to spread the load.*

*Third — schema inference on large XML. spark-xml's `inferSchema=true` does a full scan of all input files to infer types — it reads everything twice: once to infer schema, once to load. For thousands of XML files this doubled read time. The fix was defining the schema explicitly — pre-specify the StructType — so inference was skipped.*"

---

**The three fixes:**

```python
# Bottleneck 1: Too many small files — coalesce before write
df_final \
    .coalesce(20) \                           # Merge thousands of small partitions into 20
    .write.format("hive") \
    .mode("append") \
    .partitionBy("transaction_date") \
    .saveAsTable("finops_db.bronze_transactions")


# Bottleneck 2: Skew after explode — repartition after explode
df_exploded = df_flat.withColumn("line", explode_outer(col("line_items")))
df_rebalanced = df_exploded.repartition(100, col("transaction_date"))  # spread evenly
# Also enable AQE
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")


# Bottleneck 3: Schema inference = double read — define schema explicitly
from pyspark.sql.types import StructType, StructField, StringType, DecimalType, ArrayType

xml_schema = StructType([
    StructField("_TransactionID", StringType()),
    StructField("_Date",          StringType()),
    StructField("Amount", StructType([
        StructField("_VALUE",    DecimalType(18,2)),
        StructField("_Currency", StringType())
    ])),
    StructField("Vendor", StructType([
        StructField("VendorID",   StringType()),
        StructField("VendorName", StringType())
    ])),
    StructField("LineItems", StructType([
        StructField("LineItem", ArrayType(StructType([
            StructField("Code",   StringType()),
            StructField("Amount", DecimalType(18,2))
        ])))
    ]))
])

df_raw = spark.read \
    .format("xml") \
    .option("rowTag", "Transaction") \
    .schema(xml_schema) \                     # ← skip inference, single read pass
    .load("abfss://raw@abbviedatalake.dfs.core.windows.net/finops/xml/")
```
