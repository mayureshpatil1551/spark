## 1. What Is Bucketing?

Bucketing is a **write-time optimization** that pre-organizes data into a fixed number of files (buckets) based on the **hash value of a column**. Unlike partitioning (which creates directories), bucketing controls **which rows go into which file** within a table.

```
Core formula: bucket_number = hash(bucket_column) % num_buckets

Example: product_id = 22, num_buckets = 4
  hash(22) % 4 = 2  →  row goes into bucket file 2
```

Bucketing is stored as a **Hive-compatible table** (uses `saveAsTable`, not `.parquet(path)`). Spark remembers the bucket metadata in the metastore and uses it to **skip shuffles** on future reads.

---

## 2. Bucketing vs Partitioning — The Critical Difference

Both organize data at write time, but they solve different problems.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    PARTITIONING                                     │
│                                                                     │
│  orders/                                                            │
│  ├── product_id=P001/   (directory per value)                       │
│  │       part-0.parquet                                             │
│  ├── product_id=P002/                                               │
│  └── product_id=P003/                                               │
│                                                                     │
│  ✅ Great for low–medium cardinality columns (date, region)         │
│  ❌ High cardinality (product_id, customer_id) → millions of dirs   │
└─────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────┐
│                      BUCKETING                                      │
│                                                                     │
│  orders_bucketed/                                                   │
│  ├── part-00000.parquet   (bucket 0 — all rows where hash%4 == 0)  │
│  ├── part-00001.parquet   (bucket 1 — all rows where hash%4 == 1)  │
│  ├── part-00002.parquet   (bucket 2 — all rows where hash%4 == 2)  │
│  └── part-00003.parquet   (bucket 3 — all rows where hash%4 == 3)  │
│                                                                     │
│  ✅ Great for HIGH cardinality columns (product_id, customer_id)   │
│  ✅ Eliminates shuffle on joins and GroupBy                         │
│  ❌ Only works when saved as a table (not raw path)                 │
└─────────────────────────────────────────────────────────────────────┘
```

| Property | Partitioning | Bucketing |
|---|---|---|
| Storage layout | Directories per value | Fixed N files, hash-assigned |
| Best for | Low–medium cardinality | High cardinality |
| Eliminates shuffle? | No | Yes (if both tables bucketed same way) |
| Supports pruning? | Yes (partition pruning) | Yes (bucket pruning) |
| Works with raw path? | Yes | No — requires Hive metastore table |
| Compatible | All formats | Parquet + Hive metastore |
| Can combine both? | ✅ Yes — `partitionBy + bucketBy` | ✅ Yes |

## 3. How Bucketing Works Internally

### At Write Time

```python
df_orders.write \
    .bucketBy(4, "product_id") \    # ← hash(product_id) % 4 assigns each row
    .sortBy("product_id") \          # ← rows WITHIN each bucket are sorted
    .mode("overwrite") \
    .saveAsTable("orders_bucketed")

# Result: 4 Parquet files, pre-sorted by product_id within each file
# Metadata stored in Hive metastore: {num_buckets: 4, bucket_col: product_id}
```

### At Read Time (The Magic)

```
Without bucketing:              With bucketing:
Read all data                   Read all data
      ↓                               ↓
   SHUFFLE                     Spark checks metastore:
(redistribute by key)           "Is this table bucketed?"
      ↓                         YES → rows already co-located
    Sort                        → SKIP SHUFFLE
      ↓                               ↓
   Merge/Join                      Sort (within buckets)
                                      ↓
                                  Merge/Join

Shuffle = read entire dataset   No shuffle = read once, process once
over the network
```

### The Metastore Is Key

Bucketing metadata lives in the **Hive metastore** (or Unity Catalog on Databricks). When Spark plans a query, it looks up whether the tables involved are bucketed, and if so — using the same column and same number of buckets — it **eliminates the Exchange (shuffle) step** from the query plan.

## 4. Bucketing for Filtering

### Why Partitioning Fails for High-Cardinality Filter Columns

```
Dataset: 10 million orders, 500,000 unique product_ids
Filter:  WHERE product_id = 'P00042'

Option 1 — No optimization:
  Full scan of all 10M rows → slow

Option 2 — Partition by product_id:
  500,000 directories × tiny files → small files explosion → NameNode overload

Option 3 — Bucket by product_id (4 buckets):
  hash('P00042') % 4 = 2
  Spark reads ONLY part-00002.parquet
  → 1 out of 4 files scanned instead of all 500,000 directories
```

### Bucket Pruning in Action

```python
df_orders_bucketed = spark.table("orders_bucketed")

df_filtered = df_orders_bucketed \
    .filter(F.col("product_id") == 1) \
    .groupBy("product_id") \
    .agg(F.sum("total_amount").alias("sales"))

df_filtered.explain()

# Physical Plan confirms bucket pruning:
# SelectedBucketsCount: 1 out of 4
# → Only 1 bucket file read, 3 skipped
```

## 5. Bucketing for Joins — The Biggest Win

### Without Bucketing — The Full Shuffle Tax

```python
df_orders_product_details = df_orders.join(df_products, on="product_id", how="inner")
df_orders_product_details.explain()

# Physical Plan:
# SortMergeJoin [product_id], [product_id], Inner
#   Sort [product_id ASC]
#     Exchange hashpartitioning(product_id, 200)   ← SHUFFLE (expensive!)
#       Filter isnotnull(product_id)
```

Every time you run this join:
```
Stage 1: Read orders      → shuffle ALL data across network by product_id hash
Stage 2: Read products    → shuffle ALL data across network by product_id hash
Stage 3: Sort both sides  → sort within each partition
Stage 4: Merge join       → merge sorted partitions
```

### With Bucketing — Shuffle Eliminated

```python
# Write ONCE — both tables with same bucket count and column
df_orders.write \
    .bucketBy(4, "product_id") \
    .sortBy("product_id") \
    .mode("overwrite") \
    .saveAsTable("orders_bucketed")

df_products.write \
    .bucketBy(4, "product_id") \
    .sortBy("product_id") \
    .mode("overwrite") \
    .saveAsTable("products_bucketed")

# Join — no shuffle
df_orders_bucketed   = spark.table("orders_bucketed")
df_products_bucketed = spark.table("products_bucketed")

result = df_orders_bucketed.join(df_products_bucketed, on="product_id", how="inner")
result.explain()

# Physical Plan:
# SortMergeJoin [product_id], [product_id], Inner
#   Sort [product_id ASC]
#     FileScan parquet orders_bucketed ... SelectedBucketsCount: 4 out of 4
#   Sort [product_id ASC]
#     FileScan parquet products_bucketed ... SelectedBucketsCount: 4 out of 4
#
# ← NO Exchange (shuffle) step! Spark reads directly and merges.
```

### Why This Matters at Scale

```
10 GB orders + 5 GB products joined 20 times daily:

Without bucketing:
  20 joins × (10 GB + 5 GB) shuffle = 300 GB network traffic/day
  Each join: ~8 minutes

With bucketing:
  Write cost: one-time setup (a few minutes)
  Each join: ~45 seconds (no shuffle, just read + sort + merge)
  20 joins/day × 7 min saved = 140 minutes saved daily
```

## 6. Bucketing for Aggregations (GroupBy)

### Without Bucketing — Shuffle Required

```python
df_product_sales = df_orders.groupBy("product_id").agg(F.sum("total_amount").alias("sales"))
df_product_sales.explain()

# Physical Plan:
# HashAggregate(keys=[product_id], functions=[sum(total_amount)])
#   Exchange hashpartitioning(product_id, 200)    ← SHUFFLE (expensive!)
#     HashAggregate(keys=[product_id], functions=[partial_sum(total_amount)])
#       FileScan csv [product_id, total_amount]
#
# Two-phase aggregation:
#   Phase 1: Partial aggregate locally per partition
#   Phase 2: SHUFFLE all partial results → final aggregate
```

### With Bucketing — Single-Phase Aggregation

```python
df_product_sales = df_orders_bucketed.groupBy("product_id").agg(F.sum("total_amount").alias("sales"))
df_product_sales.explain()

# Physical Plan:
# HashAggregate(keys=[product_id], functions=[sum(total_amount)])
#   HashAggregate(keys=[product_id], functions=[partial_sum(total_amount)])
#     FileScan parquet orders_bucketed [product_id, total_amount]
#              SelectedBucketsCount: 4 out of 4
#
# ← NO Exchange step! All product_id=X rows already in the same bucket file.
# Spark can aggregate directly — no redistribution needed.
```

```
Without bucketing:        With bucketing:
Read CSV                  Read bucketed Parquet
    ↓                           ↓
Partial Aggregate         Direct full Aggregate
    ↓                           ↓
SHUFFLE                   Write result
    ↓
Final Aggregate
    ↓
Write result

3 stages, network I/O     1 stage, no network I/O
```

## 7. Join Performance: All Bucketing Scenarios

This is a critical interview topic — how does join behavior change based on bucketing state of each table?

```
Legend: B = Bucketed, x = 4 buckets, y = 8 buckets (different count)
        {col} = different bucket column used
```

| Scenario | Table 1 | Table 2 | Result | Why |
|---|---|---|---|---|
| **1** | B, 4 buckets | B, 4 buckets (same col) | ✅ **No shuffle** | Same hash function → rows co-located |
| **2** | B, 4 buckets | B, 8 buckets (same col) | ⚠️ **One side shuffled** | Different bucket count → hash mismatch |
| **3** | B, 4 buckets | B, 4 buckets (different col) | ❌ **Full shuffle** | Join key ≠ bucket key → co-location lost |
| **4** | B, 4 buckets | Not bucketed | ❌ **Full shuffle** | Non-bucketed side can't be co-located |
| **5** | Not bucketed | Not bucketed | ❌ **Full shuffle** | Default behavior |

### Scenario 2 Deep Dive — Bucket Count Mismatch

```
orders_bucketed:  4 buckets  →  bucket_num = hash(product_id) % 4
products_bucketed: 8 buckets →  bucket_num = hash(product_id) % 8

Bucket 0 in orders ≠ Bucket 0 in products
(hash % 4 = 0 covers different product_ids than hash % 8 = 0)

→ Spark cannot co-locate rows → must shuffle one side (smaller table typically)
```

**Key Rule:** For bucketing to eliminate a shuffle on a join, **both tables must be bucketed on the same column with the same number of buckets**.

### Scenario 3 Deep Dive — Wrong Column Bucketed

```python
# orders bucketed by product_id
# products bucketed by product_id
# But join is on customer_id (not the bucket column)

result = orders_bucketed.join(products_bucketed, on="customer_id")
# Spark sees bucket column = product_id, join key = customer_id → MISMATCH
# Full shuffle required — bucketing gives zero benefit
```

## 8. How to Determine the Ideal Number of Buckets

### Formula

```
No. of buckets = Total dataset size / Optimal bucket size

Optimal bucket size = 128 MB to 200 MB  (same as ideal partition size)

Example:
  Dataset = 1,000 MB (1 GB)
  Optimal bucket size = 200 MB
  → Buckets = 1000 / 200 = 5 buckets
```

### Estimating Dataset Size Without Running the Job

```
Formula: Size (MB) = (N × V × W) / 1,048,576

Where:
  N = Number of rows
  V = Number of columns
  W = Average width in bytes per column (approximate)

Common column widths:
  INT / INTEGER     → 4 bytes
  BIGINT / LONG     → 8 bytes
  FLOAT             → 4 bytes
  DOUBLE            → 8 bytes
  STRING (avg 20ch) → 20 bytes
  DATE              → 4 bytes
  TIMESTAMP         → 8 bytes
  BOOLEAN           → 1 byte

Example: orders table
  N = 10,000,000 rows
  V = 6 columns (order_id INT, product_id INT, customer_id INT,
                  quantity INT, order_date DATE, total_amount INT)
  W = (4+4+4+4+4+4) / 6 = 4 bytes avg

  Size = (10,000,000 × 6 × 4) / 1,048,576
       = 240,000,000 / 1,048,576
       ≈ 229 MB

  Buckets = 229 / 200 ≈ 2 buckets
  Round up to next power of 2 for hash efficiency → 2 buckets
```

### Power-of-2 Recommendation

Use powers of 2 for bucket count (2, 4, 8, 16, 32, ...). Hash functions distribute more uniformly across power-of-2 boundaries, reducing skew within buckets.

```
Dataset size  →  Recommended buckets
< 400 MB      →  2
400 MB–1 GB   →  4–8
1 GB–5 GB     →  8–16
5 GB–20 GB    →  16–32
> 20 GB       →  32–64+ (evaluate with profiling)
```

## 9. Bucket Pruning — Fine-Grained Filtering

When you filter on the bucket column with an **equality condition**, Spark computes which bucket the value falls in and reads **only that bucket file**.

```python
# product_id = 1
# hash(1) % 4 = 1  (or whichever bucket it lands in)
# Spark reads ONLY that one bucket file

df_orders_bucketed \
    .filter(F.col("product_id") == 1) \
    .groupBy("product_id") \
    .agg(F.sum("total_amount").alias("sales")) \
    .explain()

# Physical Plan shows:
# SelectedBucketsCount: 1 out of 4
# → 75% of data skipped at read time
```

### When Bucket Pruning Does NOT Happen

```python
# Range filter — Spark doesn't know which buckets contain values in range
df.filter(F.col("product_id") > 100)              # ❌ No pruning (reads all 4 buckets)

# IN list — Spark CAN compute which buckets (one per value)
df.filter(F.col("product_id").isin([1, 22, 45]))  # ✅ Reads only relevant buckets

# Function on column — Spark can't evaluate at planning time
df.filter(F.col("product_id") % 2 == 0)           # ❌ No pruning

# Non-bucket column filter — bucket layout irrelevant
df.filter(F.col("customer_id") == 500)             # ❌ Full scan of all buckets
```

## 10. Bucketing Limitations — What Your Notes Missed

These are critical for a 3.5-year level interview. Knowing the limitations shows production maturity.

### 1. Requires Hive Metastore (No Raw Path Support)

```python
# WORKS: saves to Hive table, metastore stores bucket metadata
df.write.bucketBy(4, "product_id").saveAsTable("orders_bucketed")

# FAILS: bucketing metadata NOT stored — Spark won't use it
df.write.bucketBy(4, "product_id").parquet("s3://bucket/path/")
# No error, but no shuffle elimination either
```

### 2. No Native Support in Delta Lake

Delta Lake (as of Spark 3.x) does **not** natively support `bucketBy`. Delta's solution is `OPTIMIZE + ZORDER` which achieves similar (though not identical) co-location benefits. On Databricks, you can get shuffle elimination via **Liquid Clustering** (Databricks Runtime 13.3+).

```python
# Delta Lake alternative to bucketing:
spark.sql("""
    OPTIMIZE orders
    ZORDER BY (product_id)
""")
# Data is co-located by product_id within files — not the same as bucketing
# but reduces shuffle in many practical cases via data skipping
```

### 3. Bucket Count Is Immutable After Write

Once a table is bucketed with N buckets, you **cannot change N** without rewriting the entire table. If your data grows 10× and you now need more buckets, you must rewrite.

### 4. AQE Can Conflict With Bucketing

Adaptive Query Execution (AQE) can sometimes override the shuffle-skip optimization. If you see unexpected shuffles on bucketed tables:

```python
# Disable AQE for this session if bucketing is not being honored
spark.conf.set("spark.sql.adaptive.enabled", "false")

# Or keep AQE but ensure bucketing is respected:
spark.conf.set("spark.sql.sources.bucketing.enabled", "true")  # default true
spark.conf.set("spark.sql.sources.bucketing.autoBucketedScan.enabled", "true")
```

### 5. Both Tables Must Be Bucketed the Same Way

As covered in Scenario 2 and 3 above — mismatched bucket counts or columns negate the shuffle elimination benefit.

## 11. Full Code Reference

```python
from pyspark.sql import SparkSession
import pyspark.sql.functions as F

spark = SparkSession.builder \
    .appName("BucketingDemo") \
    .config("spark.sql.warehouse.dir", "/tmp/spark-warehouse") \
    .enableHiveSupport() \
    .getOrCreate()

# ── Step 1: Create sample data ─────────────────────────────────────────────
df_orders = spark.createDataFrame([
    (1, 22, 101, 3, "2024-01-01", 300),
    (2, 45, 102, 1, "2024-01-02", 100),
    (3, 22, 103, 2, "2024-01-03", 200),
    (4, 10, 104, 5, "2024-01-04", 500),
], ["order_id", "product_id", "customer_id", "quantity", "order_date", "total_amount"])

df_products = spark.createDataFrame([
    (22, "Laptop",  "Electronics", "Dell",   999, 50),
    (45, "Phone",   "Electronics", "Samsung", 699, 100),
    (10, "Tablet",  "Electronics", "Apple",  499, 75),
], ["product_id", "product_name", "category", "brand", "price", "stock"])

# ── Step 2: Write bucketed tables ──────────────────────────────────────────
# Both tables: 4 buckets on product_id — MUST match for shuffle elimination
df_orders.write \
    .bucketBy(4, "product_id") \
    .sortBy("product_id") \
    .mode("overwrite") \
    .saveAsTable("orders_bucketed")

df_products.write \
    .bucketBy(4, "product_id") \
    .sortBy("product_id") \
    .mode("overwrite") \
    .saveAsTable("products_bucketed")

# ── Step 3: Read bucketed tables ───────────────────────────────────────────
df_orders_b   = spark.table("orders_bucketed")
df_products_b = spark.table("products_bucketed")

# ── Step 4: Bucketed Join (no shuffle) ────────────────────────────────────
print("=== BUCKETED JOIN PLAN ===")
result_join = df_orders_b.join(df_products_b, on="product_id", how="inner")
result_join.explain()
# Verify: NO 'Exchange hashpartitioning' in plan

# ── Step 5: Bucketed GroupBy (no shuffle) ─────────────────────────────────
print("=== BUCKETED GROUPBY PLAN ===")
result_agg = df_orders_b.groupBy("product_id").agg(F.sum("total_amount").alias("sales"))
result_agg.explain()
# Verify: NO 'Exchange hashpartitioning' in plan

# ── Step 6: Bucket Pruning ─────────────────────────────────────────────────
print("=== BUCKET PRUNING PLAN ===")
result_pruned = df_orders_b \
    .filter(F.col("product_id") == 22) \
    .groupBy("product_id") \
    .agg(F.sum("total_amount").alias("sales"))
result_pruned.explain()
# Verify: SelectedBucketsCount: 1 out of 4

# ── Step 7: Compare plans side-by-side ────────────────────────────────────
print("=== NON-BUCKETED JOIN PLAN (for comparison) ===")
df_orders_raw   = spark.read.csv("orders.csv", header=True, inferSchema=True)
df_products_raw = spark.read.csv("products.csv", header=True, inferSchema=True)
result_no_bucket = df_orders_raw.join(df_products_raw, on="product_id", how="inner")
result_no_bucket.explain()
# You'll see 'Exchange hashpartitioning' — the shuffle that bucketing avoids

# ── Step 8: Combine Partitioning + Bucketing ───────────────────────────────
# Partition by date (low cardinality) + bucket by product_id (high cardinality)
df_orders.write \
    .partitionBy("order_date") \         # directory per date
    .bucketBy(4, "product_id") \         # 4 bucket files within each date dir
    .sortBy("product_id") \
    .mode("overwrite") \
    .saveAsTable("orders_partitioned_bucketed")
```

## 12. Interview Q&A – Tiered

---

### 🟢 Basic Level

**Q1. What is bucketing in Spark and how is it different from partitioning?**

**A:** Both organize data at write time, but differently. Partitioning creates directories on disk for each unique value of a column — great for low-cardinality columns like date or region. Bucketing divides data into a fixed number of files (buckets) using `hash(column) % num_buckets` — designed for high-cardinality columns like `product_id` or `customer_id` where partitioning would create millions of tiny directories. Bucketing's primary purpose is to **eliminate shuffle** on future joins and GroupBy operations, which partitioning cannot do.

---

**Q2. What operations benefit from bucketing?**

**A:** Three operations:
- **Joins** — if both tables are bucketed on the join key with the same number of buckets, Spark skips the shuffle entirely (no Exchange step)
- **GroupBy / Aggregations** — rows with the same key are already in the same bucket, so Spark can aggregate without a shuffle redistribution
- **Filters on the bucket column** (bucket pruning) — Spark computes `hash(value) % N` at query time and reads only the matching bucket file

---

**Q3. Why can't you use `bucketBy` with `.parquet("path")` instead of `.saveAsTable()`?**

**A:** Bucketing metadata (number of buckets, bucket column) must be stored in the **Hive metastore** so Spark can retrieve it when planning future queries. When you use `.saveAsTable()`, Spark registers this metadata. When you write to a raw path with `.parquet()`, there is no metastore entry — Spark doesn't know the data is bucketed and will not apply shuffle elimination. The data gets written as if bucketing were applied (the hash assignment happens), but Spark ignores it at read time.

---

### 🟡 Medium Level

**Q4. Two tables are bucketed on `product_id` — one with 4 buckets, one with 8. Does bucketing help the join?**

**A:** No — only partially. A bucket count mismatch means the hash functions produce different bucket assignments. `hash(product_id) % 4 = 0` and `hash(product_id) % 8 = 0` cover different sets of product IDs. Spark cannot co-locate matching rows without a shuffle. In practice, Spark will shuffle the smaller table to match the larger table's bucketing. For full shuffle elimination, **both tables must use the same column AND the same bucket count**.

---

**Q5. How does bucketing interact with AQE (Adaptive Query Execution)?**

**A:** AQE can sometimes override bucketing optimization. AQE dynamically adjusts the query plan at runtime — and in some cases it may decide to re-shuffle data (e.g., to handle skew), which nullifies the shuffle elimination that bucketing provides. If you're seeing unexpected shuffles on bucketed tables, check if AQE's `autoBucketedScan` is enabled (`spark.sql.sources.bucketing.autoBucketedScan.enabled = true`) and consider whether disabling AQE for that specific job is warranted. Generally, bucketing and AQE can coexist but require care.

---

**Q6. How would you decide the number of buckets for a 5 GB dataset?**

**A:** Target 128–200 MB per bucket. `5 GB / 200 MB = 25 buckets`. Round to the nearest power of 2 → **32 buckets**. Powers of 2 distribute data more uniformly across hash boundaries. If I don't know the exact size, I estimate: `Size (MB) = (num_rows × num_cols × avg_col_width_bytes) / 1,048,576`. I then validate after writing by checking average bucket file size in the warehouse directory.

---

### 🔴 Advanced Level

**Q7. You have a Delta Lake table on Databricks and want bucketing-like behavior. How do you achieve it?**

**A:** Delta Lake doesn't natively support `bucketBy`. There are three approaches depending on the Databricks Runtime version:
1. **ZORDER BY** (all versions) — `OPTIMIZE table ZORDER BY (product_id)` co-locates data with similar product_ids within files. This enables data skipping at the row-group level, not full shuffle elimination, but significantly reduces I/O.
2. **Liquid Clustering** (DBR 13.3+) — `CLUSTER BY (product_id)` incrementally clusters data as it's written and compacted. More flexible than bucketing — you can recluster without rewriting everything.
3. **Hive-compatible bucketing on a non-Delta table** — write a Parquet-format Hive table alongside your Delta table for join optimization, then use it specifically for high-frequency joins.

---

**Q8. You've bucketed orders and products on `product_id` with 4 buckets. Now you need to join orders with a new `customers` table on `customer_id`. Does bucketing help this join?**

**A:** No. The bucket column is `product_id`, but the join key for the new operation is `customer_id`. Spark checks if the join key matches the bucket key — it doesn't, so the bucket co-location is irrelevant. Spark performs a full shuffle on `customer_id`. To benefit from bucketing on both joins, you'd need to bucket orders twice (which isn't possible for one table) or accept that bucketing helps only the `product_id` join. In practice, bucket on the join key you use most frequently, and use broadcast join or AQE for other joins.

---

**Q9. A team is debating whether to use bucketing or broadcast join for a frequently-run join between a 100 GB fact table and a 150 MB dimension table. What's your recommendation?**

**A:** Broadcast join is the better choice here. At 150 MB, the dimension table is well within practical broadcast range (raise `spark.sql.autoBroadcastJoinThreshold` to 200 MB). Broadcast join eliminates the shuffle without any write-time setup, works with Delta Lake, works with any file format, and doesn't require both tables to be rewritten and registered in the metastore. Bucketing is the better choice when: (1) the dimension table is too large to broadcast (>500 MB), (2) the join runs hundreds of times daily where the one-time write cost is justified, or (3) you also want to eliminate shuffle on GroupBy operations on the fact table. In short: **broadcast first, bucket when broadcast isn't possible or GroupBy optimization is also needed**.

## 13. Real-World Scenarios & Solutions

---

### 🏗️ Scenario 1 – MDM Golden Record Pipeline: Eliminating Repeated Join Shuffles

**Context:** In your MDM pipeline, you join a 50 GB `transactions` table with a 2 GB `master_entities` table on `entity_id` — 10 times per day as new transactions arrive. Each join takes 12 minutes (dominated by shuffle). `master_entities` is too large to broadcast reliably.

**Problem:**
```
10 joins/day × (50 GB + 2 GB shuffled) = 520 GB network traffic daily
12 min × 10 = 2 hours of compute time daily on join alone
```

**Solution:**

```python
# ONE-TIME SETUP: bucket both tables with the same key and count
# Master entities: relatively stable, rewrite when updated
spark.read.format("delta").load("/delta/master_entities") \
    .write \
    .bucketBy(16, "entity_id") \
    .sortBy("entity_id") \
    .mode("overwrite") \
    .saveAsTable("master_entities_bucketed")

# Transactions: bucket on load (or during initial ingestion)
spark.read.format("delta").load("/delta/transactions") \
    .write \
    .bucketBy(16, "entity_id") \
    .sortBy("entity_id") \
    .mode("overwrite") \
    .saveAsTable("transactions_bucketed")

# DAILY JOIN: no shuffle, every time
txn    = spark.table("transactions_bucketed")
master = spark.table("master_entities_bucketed")

golden_records = txn.join(master, on="entity_id", how="left") \
                    .groupBy("entity_id") \
                    .agg(
                        F.max("transaction_date").alias("last_activity"),
                        F.sum("amount").alias("total_spend"),
                        F.first("entity_name").alias("entity_name")
                    )

golden_records.write.format("delta").mode("overwrite").save("/delta/golden_records")
```

**Result:** Join time drops from 12 minutes to ~2 minutes (no shuffle). Over 30 days: **~100 hours of compute saved**.

### 🏗️ Scenario 2 – Iceberg Migration: Bucketing Strategy

**Context:** You're migrating a heavily-joined 200 GB `orders` table to Iceberg. The table is joined with `products` (1 GB) on `product_id` dozens of times daily by analysts. After migration to Iceberg, shuffle costs returned (Iceberg doesn't use Hive bucketing).

**Solution — Iceberg Sort Order as a Bucketing Alternative:**

```python
# Iceberg approach: define sort order at table creation
spark.sql("""
    CREATE TABLE iceberg_catalog.db.orders (
        order_id    INT,
        product_id  INT,
        customer_id INT,
        total_amount DOUBLE,
        order_date  DATE
    )
    USING iceberg
    PARTITIONED BY (months(order_date))
    TBLPROPERTIES (
        'write.distribution-mode' = 'hash',
        'write.sort-order' = 'product_id'
    )
""")

# When writing, Spark co-locates rows with the same product_id
# within each monthly partition — similar to bucketing within a partition

# Periodic compaction to maintain sort order
spark.sql("""
    CALL iceberg_catalog.system.rewrite_data_files(
        table => 'db.orders',
        strategy => 'sort',
        sort_order => 'product_id ASC'
    )
""")

# For the products table: keep as Hive-bucketed for join compatibility
# OR: use broadcast join since 1 GB is borderline broadcastable with high driver memory
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", str(1.2 * 1024 * 1024 * 1024))
spark.conf.set("spark.driver.memory", "8g")
```

### 🏗️ Scenario 3 – Combining Partitioning + Bucketing for a Sales Analytics Table

**Context:** A 500 GB `sales_events` table is queried two ways:
- Dashboard: `WHERE event_date = '2024-01-15'` (date filter, 70% of queries)
- Product analysis: `JOIN products ON product_id GROUP BY product_id` (30% of queries)

**Solution — Partition by Date + Bucket by Product:**

```python
# Best of both worlds:
# partitionBy("event_date") → pruning on date (low cardinality: 365 dirs/year)
# bucketBy(8, "product_id") → shuffle elimination on product joins and GroupBy

spark.read.format("delta").load("/delta/sales_events") \
    .write \
    .partitionBy("event_date") \
    .bucketBy(8, "product_id") \
    .sortBy("product_id") \
    .mode("overwrite") \
    .saveAsTable("sales_events_optimized")

# Dashboard query: partition pruning kicks in
spark.table("sales_events_optimized") \
    .filter("event_date = '2024-01-15'") \
    .agg(F.sum("revenue")) \
    .show()
# → Reads only event_date=2024-01-15/ directory (partition pruning)

# Product analysis: bucketing kicks in (if products also bucketed with 8 buckets)
spark.table("sales_events_optimized") \
    .join(spark.table("products_bucketed"), on="product_id") \
    .groupBy("product_id") \
    .agg(F.sum("revenue").alias("total_revenue")) \
    .show()
# → No shuffle on join, no shuffle on GroupBy
```

│   ├── part-00000.parquet  (bucket 0 for this date)
│   ├── part-00001.parquet  (bucket 1 for this date)
│   ├── ...
│   └── part-00007.parquet  (bucket 7 for this date)
├── event_date=2024-01-16/
│   ├── part-00000.parquet
│   └── ...

## Quick Reference — Bucketing Cheat Sheet

```
WRITE SYNTAX:
  df.write
    .bucketBy(N, "col")          # required
    .sortBy("col")               # recommended (enables sort-merge without sort stage)
    .mode("overwrite")
    .saveAsTable("table_name")   # MUST use saveAsTable — not .parquet(path)

SHUFFLE ELIMINATION RULES:
  ✅ Both tables: same column, same N     → No shuffle on join
  ⚠️  Same column, different N            → One side shuffled
  ❌  Different join col than bucket col  → Full shuffle
  ❌  One table not bucketed              → Full shuffle

VERIFY IN QUERY PLAN:
  .explain() → look for:
  ✅ NO 'Exchange hashpartitioning' → shuffle eliminated
  ✅ 'SelectedBucketsCount: 1 out of 4' → bucket pruning active
  ❌ 'Exchange hashpartitioning(col, 200)' → shuffle NOT eliminated

NUMBER OF BUCKETS:
  N = dataset_size_MB / 200
  Round to nearest power of 2: 2, 4, 8, 16, 32, 64

DELTA LAKE ALTERNATIVES:
  OPTIMIZE table ZORDER BY (col)   → coarse shuffle reduction
  CLUSTER BY (col) [DBR 13.3+]    → Liquid Clustering (best option)

BUCKETING vs BROADCAST vs PARTITIONING:
  Small dim table (<200 MB)   → Broadcast join (simplest, no rewrite needed)
  Large dim table, many joins → Bucketing (one-time write cost, permanent benefit)
  Low-cardinality filter col  → Partitioning (date, region, country)
  High-cardinality filter col → Bucketing (product_id, customer_id)
```
