* Bucketing for Efficient Filtering
-------------------------------------------
1. Filter Operations -->
df_order.filter(product_id = x)

how would you store the dataset

option 1 - Store dataset as it is.
-- it is bad beacouse it wil go all records. it scan all rows.

option 2 - partitioning
-- we are partiton the dataset on product_id. 
    we underrstand product_id is high cardinality col(more unique values).
    we end up with small partition. so it will lead to small file problems.
-- so it is not crt option.

option 3 - Bucking
-- we divied all row in surten buckets.
-- like 4 buckets

0   1   2   3

hash(product_id) --> it will very large

hash(product_id)%4

eg. product_id = 22
--> 22%4 == 2
it will go to bucket no. 2

Spark Does H(Prduct_id)%4 By itself when lookin for the product_id='X' If dataset is bucketed.
It will save time.



* Bucketing for Enhanced Joins
---------------------------------------
2. Join Operations -->
-->
suppose, u got large file form src.

for order and product.

option 1 - as it is -->
we u do join, it will go through 3 steps
shuffle / sort / merge

shuffle - its costly operation(basiclay redistribute ur data which is going to make sure that the same keys reside on the same partiton. read hole dataset.)
it is not good approch.

option 2 - Partitiong -->
product_id is high cadinality col. so before joining order and product tbl using product_id col.
there are so many small partitions. so small file problew will came.
If col has high cardinality then partition is may not be solve this problem.


Option 3 - Bucking -->
we create 4 bucket

0   1   2   3   --> order tbl data
0   1   2   3   --> product tbl data

logic is --> H(product_id) % No_of_buckets
             H(product_id) % 4

when we are join those table.
those 2 bucket are same executor.
the movment they fall in same executor. join operation perform.
so u can simply do join operation.
then u need to do sort/merge.

if u repeating the join multiple time. u will not do sguffle. it will save time and cost.

* How to Determine the Ideal Number of Buckets
-->
deside what should be the optimal no. bucket the u should split ur data into.

size of dataset = x
optimal bucket size = 128-200MB
No. of bucket  = size of ur dataset / optimal bucket size   
eg.              = 1000MB / 200MB
               = 5 buckets

how to demine size of dataset?
-->
no. of Megabytes = M = (N*V*M) 1024^2
where:
    N = No of records
    v = no of Variables( no. of col )
    W = avg width in bytes of a variable (approximate size of col)
