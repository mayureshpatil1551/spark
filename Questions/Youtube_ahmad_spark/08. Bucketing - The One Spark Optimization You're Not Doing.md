## 📘 Topic: Spark Memory Management

### 🔍 Detailed Explanation

Spark uses a **Unified Memory Model** where executor memory is divided into:

1. **Execution Memory**
   - Used for shuffle, joins, aggregations, sorting
   - Temporary computations

2. **Storage Memory**
   - Used for caching (persist), broadcast variables

### ⚡ Key Behavior

- If **Execution memory is full** → it can evict Storage memory (spill to disk)
- If **Storage memory is full** → it can borrow free Execution memory
- ❗ Storage **cannot evict Execution memory** (Execution has priority)

### 🧠 Why this matters
- Avoids job failures due to memory pressure
- Helps optimize caching and performance tuning

---

## ❓ Interview Questions + Answers

### Q1: What is Spark Unified Memory Management?
**Answer:**
It is a memory model where execution and storage share a common region dynamically, improving memory utilization and reducing OOM errors.

---

### Q2: What happens when execution memory is full?
**Answer:**
Spark evicts cached storage blocks and may spill data to disk to free up memory for execution tasks.

---

### Q3: Can storage memory evict execution memory?
**Answer:**
No. Execution memory always has priority over storage memory.

---

### Q4: What is spilling in Spark?
**Answer:**
Spilling occurs when data is moved from memory to disk due to insufficient memory during execution.

---

### Q5: Difference between cache() and persist()?
**Answer:**
- `cache()` → stores in memory only (default)
- `persist()` → allows storage levels (memory, disk, serialized, etc.)


3. Persist only required datasets:
   ```python
   df.persist()
   ```

4. Tune memory configs:
   ```
   spark.executor.memory
   spark.memory.fraction
   ```

## 📘 Topic: Bucketing – Advanced Spark Optimization

### 🔍 Detailed Explanation

**Bucketing** is a technique used to divide large datasets into a fixed number of buckets based on a **hash function** applied to a column.

👉 Formula used internally:
```
hash(column) % number_of_buckets
```

Unlike partitioning:
- Buckets are **fixed in number**
- Each bucket contains multiple values
- Avoids **small file problem**

### ⚡ Where Bucketing Helps
- Filtering
- Joins
- Aggregations (groupBy)

---

## 🧠 Bucketing vs Partitioning

| Feature | Partitioning | Bucketing |
|--------|-------------|----------|
| Based on | Column value | Hash of column |
| No. of splits | Dynamic | Fixed |
| Best for | Low cardinality | High cardinality |
| Small file issue | Yes | No |

---

## 🔎 Bucketing for Filtering

### Problem
Filtering on high cardinality column (e.g., product_id)

### Without Bucketing
- Full table scan

### With Partitioning
- Too many small partitions → performance issue

### With Bucketing ✅
- Spark calculates bucket using hash
- Reads only relevant bucket

---

## 🔗 Bucketing for Joins

### Without Bucketing
- Shuffle + Sort + Merge (expensive)

### With Bucketing
- Same bucket IDs colocated
- Shuffle avoided (in ideal case)

👉 Condition for best performance:
- Same column
- Same number of buckets

---

## 📊 Bucketing for Aggregations

### Without Bucketing
- Shuffle required

### With Bucketing
- Data already grouped by hash
- Shuffle reduced or avoided

---

## ⚙️ Join Performance Scenarios

| Dataset 1 | Dataset 2 | Result |
|----------|----------|--------|
| Bucketed (same col, same buckets) | Bucketed | ✅ No shuffle |
| Bucketed | Non-bucketed | ⚠ Partial shuffle |
| Bucketed (different join column) | Bucketed | ❌ Full shuffle |

---

## 📏 How to Decide Number of Buckets

### Rule of Thumb
- Ideal bucket size: **128MB – 200MB**

### Formula
```
No_of_buckets = Total_data_size / Ideal_bucket_size
```

### Example
- Data size = 1000MB
- Bucket size = 200MB
→ Buckets = 5

## ❓ Interview Questions + Answers

### Q1: What is bucketing in Spark?
**Answer:**
Bucketing divides data into fixed number of buckets using hash function on a column to optimize joins, filtering, and aggregations.

---

### Q2: Difference between partitioning and bucketing?
**Answer:**
Partitioning is based on column values and creates dynamic partitions, while bucketing uses hash function and creates fixed buckets.

---

### Q3: When should you use bucketing?
**Answer:**
- High cardinality columns
- Frequent joins on same column
- Large datasets

---

### Q4: Can bucketing avoid shuffle completely?
**Answer:**
Yes, if both datasets are bucketed on same column with same number of buckets.

---

### Q5: What is bucket pruning?
**Answer:**
Spark reads only relevant buckets during filtering instead of scanning entire dataset.

---

### Q6: Does bucketing work in Iceberg/Delta?
**Answer:**
Modern table formats like Iceberg prefer partitioning and sorting; traditional Spark bucketing is limited or not recommended.
